## 현황 및 문제

AI 채팅 패널 1단계 구현은 일부 핵심 기능이 완료되었으나, MVP 초기 기획에 포함되었던 의도 제안, 의도 구체화, diff-check, 명세서/청구항 수정 반영 방식 등은 아직 구현 방향이 완전히 확정되지 않은 상태다.

현재 완료 또는 정리된 항목은 다음과 같다.

- AI 응답 출력 스트리밍 방식 구현 완료
- Markdown 렌더링 구분 구현 완료
- 첫 세션 시작 시 행동 제안 구현 완료
- 정상적인 응답이 오지 않았을 때 에러를 표시하는 컴포넌트 구현 완료
- 피드백(good/bad) 및 대화 기록 수집은 우선순위가 낮아 백로그로 보류

현재 논의가 필요한 항목은 다음과 같다.

- 모호한 의도 제안 기능의 현재 진행 상태 확인
- 의도 구체화가 별도 tool 없이 일반 응답으로 처리되는 구조인지 확인
- Diff-check 기능의 진행 상태 확인
- `edit_spec`, `edit_claim` 사용 시 실제 명세서 본문 및 청구항이 수정되는지, 또는 수정 제안 텍스트만 출력되는지 확인
- 문서별 기존 채팅 내역 조회 기능의 배포 경로 결정
- 기존 채팅 내역 조회 시 컨텍스트 한도 및 한도 초과 정책 필요 여부 확인
- 응답 생성 중 사용자가 위로 스크롤했을 때 자동으로 아래로 앵커되는 현상 개선
- 기재불비 `일괄보정`, `개별보정` 버튼의 동작 방식 및 AI tool/API 연동 방식 확정

## 결정 사항

이번 회의에서 결정해야 할 사항은 다음과 같다.

- 신규 기능 추가보다는, 현재 1단계 구현 이후 드러난 기획 허점과 미구현 기능을 빠르게 정리하고 마무리한다.
- 문서별 채팅 조회 API(`PS-1093`)는 속도를 위해 `main`에 바로 반영할지, `dev-staging-main` 절차를 거칠지 결정한다.
- 3번 항목인 피드백(good/bad) 및 대화 기록 수집은 백로그로 유지한다.
- 5번, 6번 항목은 1단계 구현 완료 후 다시 논의하기로 했던 항목이므로, 이번 회의에서 구현 여부와 범위를 재검토한다.
- 기재불비 `일괄보정`, `개별보정`은 현재 UI에는 버튼이 있으나 미구현 상태이므로, 버튼 클릭 시 제공할 보정 경험과 실제 수정 반영 방식을 결정한다.
- AI Chat 패널과 동일한 BE 통신 방식을 재사용할지, 기재불비 보정 전용 API/흐름이 필요한지 결정한다.

## 기술 세부사항

### 1. AI 채팅 패널 후속 구현 상태 점검

- 완료
  - AI 응답 스트리밍
  - Markdown 렌더링 구분
  - 첫 세션 시작 시 행동 제안
  - 비정상 응답 시 에러 컴포넌트 표시
- 백로그
  - 피드백(good/bad) 수집
  - 대화 기록 수집
- 진행 상태 확인 필요
  - 모호한 의도 제안
  - Diff-check
  - 의도 구체화 응답 처리 방식

### 2. 의도 제안 및 의도 구체화

- 모호한 의도 제안 기능이 현재 컴포넌트화되어 있는지 확인한다.
- 성현님, 윤주님이 관련 작업을 진행 중인지 또는 홀딩 상태인지 확인한다.
- 의도 구체화 기능이 별도 tool 호출 없이 일반 AI 응답으로 처리되는 구조인지 확인한다.
- 관련 대화 식별자
  - `conversation_id`: `04418548-3140-43e9-a9c6-e85da30c9576`
  - `user_message_id`: `9a87c14e-c277-4e83-bd82-a354851349db`

### 3. 수정 제안 및 실제 문서 반영 방식

- `edit_spec`, `edit_claim` 사용 시 결과가 단순 텍스트 출력인지, 실제 명세서 본문/청구항 수정까지 반영되는지 확인한다.
- Diff-check 기능이 수정 전후 비교를 어디까지 지원해야 하는지 정리한다.
- 실제 수정이 발생한다면 다음 항목을 함께 정의한다.
  - 수정 대상 영역 식별 방식
  - 수정 전/후 diff 표시 방식
  - 사용자의 최종 승인 흐름
  - 실패 시 복구 또는 재시도 정책

### 4. 문서별 기존 채팅 내역 조회

- 기존 채팅 내역 조회는 이번 주 작업 예정으로 공유되었다.
- `PS-1093-문서별-채팅-조회-api`가 상용에 바로 반영된 상태이므로, 프론트 작업도 상용 기준으로 바로 진행할지 결정이 필요하다.
- 기존 채팅 내역 조회와 함께 응답 생성 중 스크롤 앵커 문제도 같이 처리할 수 있다.
- 컨텍스트 한도 정책이 필요하다.
  - 채팅 내역을 어디까지 불러올지
  - 컨텍스트 한도 초과 시 어떤 기준으로 제외할지
  - 사용자에게 한도 초과 또는 일부 생략 상태를 보여줄지

### 5. 기재불비 보정 버튼 구현

명세서 편집 화면 좌측 청구항 패널에는 다음 버튼이 이미 제공되고 있으나 현재 미구현 상태다. 이미지 기준으로 빨간 박스 (1)은 전체 청구항 대상 `일괄보정`, 빨간 박스 (2)는 선택 청구항 대상 `개별보정`이다.

- 전체 청구항 대상 `일괄보정`
- 개별 청구항 박스의 `보정`

논의가 필요한 구현 방향은 다음과 같다.

- `일괄보정` 클릭 시 전체 청구항을 대상으로 기재불비 검토 및 보정안을 생성할지
- `개별보정` 클릭 시 선택한 청구항만 대상으로 보정안을 생성할지
- `tool:review_claim`, `tool:write_claim`을 활용할 수 있는지
- 해당 tool 호출 방식의 API 명세가 필요한지
- BE 통신 방식을 AI Chat 패널과 동일하게 둘지, 별도 보정 전용 흐름으로 분리할지
- 보정 결과를 즉시 반영할지, 사용자가 확인 후 적용하도록 할지

#### 간략 기획안

기재불비 보정 버튼은 우측 AI 코파일럿에서 제공하는 청구항 검토/작성 기능을 좌측 청구항 패널의 명시적 액션으로 호출하는 기능으로 정의한다. 사용자는 좌측 청구항 패널에서 `일괄보정` 또는 개별 청구항의 `보정` 버튼을 클릭하고, 시스템은 대상 청구항을 기준으로 기재불비 검토 결과와 보정안을 생성한다.

- `일괄보정`
  - 대상: 현재 문서의 전체 청구항
  - 목적: 전체 청구항의 기재불비 항목을 한 번에 검토하고, 청구항별 보정안을 생성
  - 결과: 청구항별 문제 유형, 검토 사유, 보정 제안, 적용 가능 상태 표시
- `개별보정`
  - 대상: 사용자가 클릭한 단일 청구항
  - 목적: 특정 청구항의 기재불비를 빠르게 검토하고 보정안 생성
  - 결과: 해당 청구항의 문제 유형, 검토 사유, 보정 제안, 적용 가능 상태 표시

권장 UX는 “자동 생성 후 사용자 확인 적용”이다. 버튼 클릭만으로 원문 청구항을 즉시 수정하지 않고, 먼저 보정안을 생성한 뒤 사용자가 `적용`을 선택했을 때 청구항 본문에 반영한다. 이는 AI 결과의 법률/문서 리스크를 낮추고, 기존 코파일럿의 보조 역할과도 일관된다.

#### API 논의안

현재 우측 AI 코파일럿에서 사용하는 `tool:review_claim`, `tool:write_claim`을 버튼 액션에서 직접 호출하기 위한 API 명세가 없다. 따라서 다음 중 하나를 결정해야 한다.

- 기존 AI Chat 패널 API를 재사용하여 버튼 클릭을 코파일럿 메시지/툴 호출로 변환한다.
- 기재불비 보정 전용 API를 별도로 만든다.
- 기존 AI Chat 패널 API는 응답 생성에만 사용하고, 보정안 적용은 별도 문서 수정 API로 처리한다.

회의에서 확인할 최소 API 요구사항은 다음과 같다.

- 요청 대상
  - `document_id`
  - `claim_id` 또는 `claim_ids`
  - `mode`: `bulk` 또는 `single`
  - 현재 청구항 본문
  - 필요 시 명세서 본문/도면/종속관계 등 참조 컨텍스트
- 처리 단계
  - `tool:review_claim`로 기재불비 검토 수행
  - `tool:write_claim`로 보정안 생성
  - 보정 전/후 diff 생성
  - 사용자가 승인한 보정안만 실제 청구항에 반영
- 응답 형식
  - 청구항 ID
  - 문제 유형
  - 검토 사유
  - 원문
  - 보정안
  - diff
  - 적용 가능 여부
  - 실패 사유

#### 화면 동작 초안

1. 사용자가 `일괄보정` 또는 `보정` 버튼을 클릭한다.
2. 좌측 청구항 패널 또는 우측 코파일럿 영역에 생성 중 상태를 표시한다.
3. AI가 기재불비 검토 및 보정안을 생성한다.
4. 결과를 청구항별로 표시하고, 사용자는 `적용`, `건너뛰기`, `다시 생성` 중 선택한다.
5. `적용` 선택 시에만 청구항 본문을 수정한다.
6. 적용 후 수정 이력 또는 diff를 확인할 수 있게 한다.
