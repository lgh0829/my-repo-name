## 1. 명세서 데이터 단락별 저장 구조

### 현황 문제
- 팟솔이 키프리스에서 받아오는 로우 데이터는 문단 번호(`P Number`)가 존재하지만, 현재 ES(Elasticsearch)에는 텍스트 전체를 통으로(스트링) 저장 중
- 조회 시 단락 구분 없이 줄글로 쫙 나오는 형식 → 가독성·파싱 모두 불리

### 결정 사항
- **DB 저장 방식**: 명세서 편집 화면과 동일하게 문단 단위로 분리해서 저장
- **API 응답 구조**: 도큐먼트를 JSON으로 래핑, 각 문단을 배열 요소로 구성

```json
{
  "background": [
    { "id": "0001", "text": "..." },
    { "id": "0002", "text": "..." }
  ],
  "problem": [
    { "id": "0001", "text": "..." }
  ]
}
```

- ES(신규 구축)에도 동일한 문단 분리 구조로 적재 요청 필요 → **상훈님께 전달**
- 스트리밍 방식 → **대체(배열) 방식으로 전환** 요청

### 담당
- 승봉님이 초안 작성 후 상훈님 검토 요청 예정

## 2. 인라인 이미지 파싱

### 현황 문제
- XML 본문 내 `<com:image>` 태그로 인라인 이미지가 삽입되어 있음
- 현재 파싱 시 해당 태그를 무시, 도면 영역에 통합 노출 중 → 잘못된 위치에 표시
- 이미지 위치 정보(폭·높이 등)가 태그 내에 포함되어 있으나 활용 안 됨

### 결정 사항
- **파싱 기준**: `<com:image>` 태그를 기준으로 인라인 이미지 추출 및 위치 분리
- 태그 내 이미지 파일명이 S3 키 경로와 일치하는지 확인 후 활용 → **건우님 확인 완료** (형식 일치)
- ES 저장 시 태그 포함한 형태로 저장 요청 (구분자 역할) → **상훈님 전달 필요**
- S3 키 형식이 특허마다 상이한 경우가 있으므로 패턴 검증 필요

### 담당
- 미주님: `P No. 046` 기준으로 샘플 데이터 전달 예정
- 건우님: S3 키 형식 및 태그 내 경로 일치 여부 최종 확인

## 3. 재생성 중 UX 정책

### 배경
- 도면 설명·상세 설명 변경 시 백그라운드로 명세서 재생성 로직이 실행됨
- 재생성 중 명세서 편집을 동시에 시도하면 충돌 발생 가능

### 결정 사항
- 재생성 중에는 **명세서 편집 관련 행위만 비활성화**
- 검색 히스토리 이동, 다른 탭 조회 등 편집 외 행위는 허용
- 백그라운드 동작 취지(다른 작업 가능)를 살리되 데이터 정합성 보호

---

## 4. 편집 중 이탈 시 UX 정책

### 논의 배경
- 항목 편집 중 저장 완료 없이 다른 항목 편집 시도·도면 전환·페이지 이탈 케이스 처리 필요

### 결정 사항

| 케이스 | 처리 방식 |
|---|---|
| 같은 페이지 내 다른 항목 편집 전환 | 기존 편집 모드 닫힘 + "저장되지 않은 변경사항" 배지 표시 |
| 도면 등 다른 탭·화면으로 전환 | 팝업 모달: "변경사항이 있습니다. 저장하시겠습니까?" |
| 페이지 이탈 (다른 페이지로 완전 이동) | 팝업 모달로 저장 유도 |

- 임시 저장은 **현재 페이지가 열려 있는 동안만** 유지 (로컬 스토리지 또는 세션 기반 확인 필요)
- 자동 저장 후 토스트로만 알리는 방식은 **보류** (저장 버튼 클릭이 명시적 저장 트리거라는 원칙 유지)
