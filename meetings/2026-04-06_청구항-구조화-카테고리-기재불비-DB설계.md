# 청구항 구조화·카테고리·기재불비·DB 설계 정책 확정

- 날짜: 2026년 4월 6일 (월)
- 참석자: 이금희, 한상호, 이윤주, 민건호

> **한줄 요약:** 청구항 구조, 카테고리 분류, 기재불비 처리, 화면 기획 및 DB 설계 방식에 대한 정책 확정

---

## 1. 청구항 구조 및 카테고리 분류

### 청구항 구조화 방식

**결정 사항:**
- ✅ 독립항 기준으로 카테고리 분류 적용. 독립항에 발명 카테고리 필드 부여
- ✅ 종속항은 부모 독립항 카테고리 상속
- ✅ 독립항끼리는 서로 다른 카테고리 적용 가능 (카테고리 : 독립항 = 1 : N)
- ✅ 현재는 독립항 1개에 종속항 N개 구조 → 발명 카테고리 1개가 되는 구조로 마이그레이션
- ✅ `parent_id` → `child_id`로 변경. `child_id`에 2개 이상 값 저장 가능

### 발명 카테고리

**결정 사항:**
- ✅ 마이그레이션 시 AI 기반 자동 분류 적용 (LLM이 말미어 분석 → 초기값 설정)

### 트리 구조 시각화 방식

**결정 사항:**
- ✅ 청구항 리스트 화면에 트리 구조 시각화 기능 추가
- ✅ 편집(write) 화면에서는 트리 구조 조회 버튼 제공
- ✅ 복잡한 UI 방지를 위해 트리 구조는 별도 조회 방식으로 분리
- ✅ 편집(write) 화면에서는 Google Patents를 참고하여 발명 카테고리 / 독립항 / 종속항의 구분만 두고, 종속항의 종속관계는 앵커로 제공

---

## 2. 기재불비 처리 및 시스템화

### 기재불비 항목 구조화

**결정 사항:**
- 📌 청구항별 기재불비 모델링에 대한 결정 필요 (청구항 + 명세서항목)

### 기재불비 검증 방식

**결정 사항:**
- ✅ 기계적(규칙 기반) 검증: 항목 10개 내외로 한정. 구조적 오류(다중인용 재인용 등) 포함
- 📌 AI 프롬프트 기반 가변적 검증 병행 적용 (명확성·지지요건 등)
- ✅ **현재 스프린트는 시스템(기계적) 검증까지만 구현**. AI 검증은 이후 단계
- 📌 기재불비 검토 기능은 향후 유료화 검토

**주요 논의:**
- 시스템만으로는 모든 기재불비 검출 부족. AI 개입 체크 필요하나 단계적으로 구현
- 기재불비 검토 기능의 비즈니스 가치 인정 → 유료화 가능성 명시

---

## 3. DB 및 서비스 설계 정책

**결정 사항:**
- ✅ 서비스 레이어: 한국 기준 기재불비 검증 적용
- ✅ DB 레이어: 글로벌 확장 대비 모든 케이스 수용 (다중인용 재인용 구조 포함)

### 명세서 생성 시 참조 기록

**결정 사항:**
- ✅ 명세서 생성 시 기재불비·청구항·도면 중 어디를 참고했는지 이력 저장 필요
- 📌 구체적인 저장 구조 및 연동 방식은 추후 별도 논의

---

## 4. 협의 완료 사항

| 항목 | 결정 내용 |
|---|---|
| 청구항 카테고리 분류 단위 | 독립항 기준. 종속항은 부모 상속 |
| 카테고리 표시 | 청구항마다 개별 표시 (이금희 1안) |
| 실질 독립항 판별 기준 | 말미어 법적 보호 객체 기준 (한상호 확인) |
| 기재불비 저장 방식 | 청구항·명세서 항목별 개별 저장 |
| 기재불비 검증 방식 | 기계적 검증 (현재) + AI 검증 (이후) 병행 |
| DB 구조 | 글로벌 수용 + 서비스는 한국법 기준 |
| parent_id 구조 | child_id로 변경, 다중 값 허용 |
