## 2. 핵심 논의

### 1) 검색 화면 구조 개편

**배경/이슈**

페르소나·유저저니 대비 VoC 분석 결과, 두 가지 핵심 페인 포인트가 도출됨

**① LLM 검색 결과에 대한 불신 (블랙박스 문제)**
- 키워드 검색은 어떤 조건으로 어느 DB를 검색했는지 투명하게 보이고 직접 컨트롤 가능
- LLM 검색은 어떤 쿼리를 생성했는지, 어떤 문서를 근거로 결과를 정리했는지 알 수 없음
- 변리사가 직접 설계한 검색식은 전문적 판단의 산물인 반면, LLM이 임의로 생성한 검색 결과는 신뢰하기 어려움

**② 선행기술 누락에 대한 불안감**
- "얼마나 찾았는지", "무엇이 빠졌을 수 있는지"를 LLM이 알려주지 않음
- 검색이 완전한지 스스로 판단할 수 없는 구조가 신뢰를 무너뜨릴 수 있음

**③ 재검색 불가 구조로 인한 통제감 부재**
- 현재는 검색 세션당 최초 1회 검색만 가능하며 재검색이 불가능한 구조
- 사용자가 LLM의 검색 결과를 보정하거나 개입할 수단이 없음

→ 개선 방향: LLM 검색 결과가 의도한 검색 범위와 일치하는지 확인할 수 있는 구조 + 결과를 보고 재검색(사용자 개입)할 수 있는 구조 필요

**제안**
- 화면을 **정보 섹션**과 **작업 섹션**으로 분리해 챗봇 에이전트 형태로 제공
  - **정보 섹션**: 높이 고정, 내부 콘텐츠가 스크롤되는 방식. 검색 히스토리(검색 결과)를 탭으로 관리
  - **작업 섹션**: 버튼 클릭 등 요청 시 정보 섹션 레이아웃은 그대로 유지하고 내부 콘텐츠만 교체. 세션(채팅) 단위로 대화 표시
- 계층 구조 정의
  - **프로젝트**: 검색 → 청구항 → 명세서 전 단계를 아우르는 단위 (기존 '발명의 명칭'과 동일 개념)
  - **세션**: 프로젝트 하위. 윈텔립스 스텝검색처럼 한 프로젝트 내에서 여러 검색 결과를 세션 단위로 저장
  - **채팅**: 세션 하위. 청구항 생성 시 새로운 채팅이 열림 (백엔드에서 세션/채팅 분리 관리)
  - 청구항 생성 단계는 검색 세션과 별개로 진행

**작업 세션 초기화 논의**
- 대화가 계속되면 컨텍스트가 커져 토큰이 길어지는 문제에 대한 우려 제기
- 이금희: 일단은 세션이 유지되는 것을 상정. 프로젝트 단위로 작업 섹션을 관리
- 김순복: '새로운 작업 세션 시작하기' 버튼을 제공해 사용자가 직접 초기화 가능하게 제공
- 이금희: 이전에 검색했던 조합을 쌓아가면서 검색 결과 변화를 확인하는 과정이 있으므로, 세션이 바뀔 때마다 새로 제공해야 하는가에 대한 의문
- 이윤주: 변리사들은 GPT 사용 시 발명 주제에 따라 새 채팅을 열어 사용할 것으로 예상. 사용자가 토큰 소비에 대한 관심은 적을 것

**결정 사항**
- ✅ 정보 섹션 / 작업 섹션 분리 구조로 확정
- ✅ 계층 구조: 프로젝트 > 세션 > 채팅. 정보 섹션 = 검색 히스토리, 작업 섹션 = 채팅
- 📌 작업 세션 초기화 기준 미정. '새 세션 시작' 버튼 제공 방향으로 검토 중이나 확정되지 않음
- 🔎 용어 통일 필요: 프로젝트 / 세션 / 채팅 각 레벨 명칭 문서화

### 2) 토픽 필터 기능

**배경/이슈**
- 검색 결과를 주제(토픽)별로 필터링하는 기능 필요
- 람다 OS 매트릭스의 matrix maker(목적/수단 추출) 기능 활용 가능성 확인됨
  - 매트릭스 프로세스: ① 목적/수단 뽑기(토픽 추출) → ② 해당 영역 분류
- 엘라스틱서치(조건식 검색) 결과는 수만 건이 나올 수 있어 전체 LLM 처리 불가 → 자연어 검색 결과에만 제공

**제안**
- 토픽 추출 방식: 검색 결과(독립항) 건당 LLM으로 핵심 내용을 **구(phrase) 단위 토픽 5개** 추출 후 태그 부여 → 전체 결과 집계하여 상위 토픽을 칩 형태로 노출
- 자연어 검색 결과는 **100건 이내**로 조절 (엘라스틱 조건식 검색에는 미제공)
- UX: 검색 결과가 먼저 표시된 뒤 토픽 필터가 서서히 업데이트되는 방식
- 성능 기준: 현재 검색 시간보다 30초 이상 길어지면 허용 불가. 현재 개발 환경에서 검색 시간을 단축한 상태이므로 토픽 추가 시에도 기존 검색 시간 대비 큰 차이 없을 것으로 예상
- AND / OR 방식: 토픽이 유의미하게 겹치면 AND 방식 적용 검토
- 배성현 추가 아이디어: 선택한 토픽의 순서를 변경하면 검색 결과 정렬도 변경되는 기능

**결정 사항**
- ✅ 토픽 필터는 자연어(유사도) 검색 결과 100건에만 제공. 엘라스틱 조건식 검색에는 미제공 (사용자 안내 필요)
- ✅ 건당 LLM 토픽 추출 방식으로 구현. 구 단위 토픽 추출
- ✅ 기능 구현 담당: 배성현
- 📌 AND / OR 방식은 구현 후 실제 토픽 분포 확인 후 결정
- 📌 토픽 필터 로딩 중 UX 처리(서서히 업데이트) 세부 설계 필요
- 🔎 성능 측정 필요: 단축된 검색 시간 기준으로 토픽 추가 시 소요 시간 측정 후 공유

### 3) 필터 UI 개선

**배경/이슈**
- 현재 '결과 내 검색' UI가 필터 기능이라는 인상을 주지 못함
- 왼쪽 고정 레이아웃이 화면 공간을 차지하며 직관성이 낮음
- 실제 사용자들이 조건 필터를 잘 사용하지 않는다는 의견 (검색 > 청구항 생성이 주 목적)

**제안**
- 필터 항목을 칩 형태로 상단에 배치 + 클릭 시 모달/사이드 패널로 세부 설정
- 전체 UI 개편(새 페이지)은 별도 배포로 진행. 급한 기능(토픽 필터 등)은 현재 레이아웃에 우선 추가
- 검색 결과 카드가 너무 커 한 화면에 적은 결과만 보임 → 컴팩트한 가로형 리스트 뷰 검토
- 뷰 전환 옵션(카드 뷰 / 리스트 뷰) 제공 검토

**결정 사항**
- 📌 조건 필터 UI 개선 방향(모달 vs 고정 레이아웃) 추후 결정
- 📌 검색 결과 카드 레이아웃 개선 추후 검토
- 🔎 전체 개편은 신규 페이지로 분리하되, 급한 기능은 현재 레이아웃에 선반영
