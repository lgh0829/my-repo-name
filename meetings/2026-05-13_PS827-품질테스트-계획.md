1. **모델 선정** — Orchestrator·Review·Author 에이전트별 최적 모델 조합 측정

## 2. Intent Taxonomy (v0.1 초안)

대화형 AI 시스템이 사용자의 발화를 해석할 때, **어떤 목적으로 무엇을 요청하는지**를 구조적으로 분류한 체계. 단순 키워드 매칭이 아닌, 요청의 의도(what the user wants to achieve)를 계층적으로 범주화함으로써 시스템이 상황에 맞는 행동을 선택할 수 있도록 한다. 잘 정의된 Intent Taxonomy는 에이전트의 의도 분류 정확도를 높이고, 허용·모호·금지 경계를 명시적으로 관리할 수 있게 한다.

> 가설 기반 초안. §3·§4 테스트 결과로 검증·수정한다.
> 현재 상태: 5단계 중 Step 3(구조 설계) 진행 중.

### 2.1. 의도 계층

```
[1] 조회  ─ 읽기 전용, 무료
    1-1. 청구항 구조 파악     T-08
    1-2. 선행기술 조회        T-07

[2] 검토  ─ 읽기 전용, 유료 게이트
    2-1. 청구항별 기재요건 검토     T-03
    2-2. 명세서 전체 기재요건 검토  T-04
    2-3. 도면 기재요건 검토         T-05

[3] 생성  ─ 문서 변경, 과금 (P-10 안내 필수)
    3-1. 단일 섹션 생성    T-01
    3-2. 전체 일괄 생성    T-02

[4] 수정  ─ 문서 변경, 무료
    4-1. 특정 섹션·문장 수정    T-06
    4-2. 선호 표현 일괄 적용    T-09

[5] 복합  ─ 다중 도구 연결 (P-13)
    5-1. 검토 → 수정    T-04 → T-06
    5-2. 검토 → 재생성  T-04 → T-01/T-02

[6] 모호
    6-1. 대상 불명확 (범위 미지정 포함)  → C-04 발동
    6-2. 작업 유형 불명확               → 무료 도구 전체 실행 → 유료 유도

[7] 금지  ─ 바리케이트
    7-1. 법적 결론 단정  (Type B — 소프트 리다이렉트)
    7-2. 도메인 외 법률 조언  (Type A — 하드 차단)
    7-3. 완전 off-topic  (Type A — 하드 차단)
```

### 2.3. 설계 결정 사항 (확정)

**결정 1 — 의도 처리 전체 결정 트리**

```
User Input
    ↓
[Layer 1: 금지 의도 체크]  // 의도 기준에 따라 소프트 리다이렉트 또는 하드 차단
    ↓ PASSED
[Orchestrator: 대상 × 행위 명확성 판단]
    ├── 대상 명확 + 행위 명확   // 즉시 실행
    ├── 대상 명확 + 행위 불명확 // 무료 도구 전체 실행 → 유료 유도
    └── 대상 불명확            // C-04 대상 선택 카드
                                   ↓ 사용자 선택 후 재판단
                               ├── 행위 명확   // 해당 행위만 실행
                               └── 행위 불명확 // 무료 도구 전체 실행 → 유료 유도
    ↓
[에이전트 레이어: Policy Gate]
    ├── 생성 실행  // P-10: 횟수·과금 안내 + 동의
    └── 유료 기능  // P-11: 구독 확인
    ↓ PASSED
실행
```

예시:

```
예A: "3항 스타일만 살짝 고쳐줘"  ← 대상 명확(3항) + 행위 명확(스타일 수정)
  → Layer 1 PASSED → 즉시 실행: T-06(스타일)만, 3항에만

예B: "3항 수정해줘"  ← 대상 명확(3항) + 행위 불명확
  → Layer 1 PASSED → 무료 도구 전체 실행: T-06 + T-09, 3항에
  → 완료 후 유료 유도: "기재요건 검토도 실행할까요?" (구독 필요)

예C: "스타일 고쳐줘"  ← 대상 불명확 + 행위 명확(스타일 수정)
  → Layer 1 PASSED → C-04: "어느 항목을 대상으로 할까요?"
  → 사용자가 [3항] 선택 → 행위 명확이었으므로 T-06(스타일)만 실행

예D: "수정해줘"  ← 대상 불명확 + 행위 불명확
  → Layer 1 PASSED → C-04: "어느 항목을 대상으로 할까요?"
  → 사용자가 [배경기술] 선택 → 행위 불명확이었으므로 무료 도구 전체 실행 → 유료 유도
```

**결정 2 — C-04 선택지 카드 구성**

C-04 카드는 현재 컨텍스트 탭에 로드된 항목(섹션·청구항)을 선택지로 구성한다. 행위 정보는 카드에 포함하지 않는다.

```
예: 청구항 1·2·3항, 배경기술 섹션, 발명의설명 섹션이 로드된 상태에서 "이거 수정해줘" 입력 시

  C-04: "어느 항목을 대상으로 할까요?"
  [1항] [2항] [3항] [배경기술] [발명의설명]

선택 후 재판단 → 결정 트리의 대상 명확 분기 재진입
  - 원래 행위가 명확했다면: 해당 행위만 실행
  - 원래 행위가 불명확했다면: 무료 도구 전체 실행 → 유료 유도
```

## 3. 모델 선정 테스트

### 3.1. 테스트 환경

#### 에이전트 구성 (기준선)

| 에이전트             | 역할                          | 모델              | Reasoning |
| ---------------- | --------------------------- | --------------- | --------- |
| **Orchestrator** | 의도 분류 → 적절한 에이전트 호출         | Claude Opus 4.7 | low       |
| **Review**       | 기재요건 검토 (T-03~T-05)         | GPT-5.5         | low       |
| **Author**       | 섹션 생성·수정 (T-01, T-02, T-06) | GPT-4.1         | low       |

**컨텍스트 탭:** 청구항 / 발명의설명 / 도면 / 기재요건

#### 사용 도구

| 에이전트       | 도구                                         |
| ---------- | ------------------------------------------ |
| **Review** | `review_claim`, `review_spec`              |
| **Author** | `write_claim`, `edit_spec`, `suggest_expr` |

#### 평가 기준

| 레이블  | 기준                          |
| ---- | --------------------------- |
| 정확   | intent·confidence 모두 예상과 일치 |
| 부정확  | intent 틀림                   |
| 형식오류 | JSON 스키마 불일치, 필드 누락         |
| 근거부족 | confidence high인데 맥락과 불일치   |

## 4. 의도 처리

### 4.1. 금지된 의도

금지 의도는 두 가지 근거로 구분한다.

- **단정 표현 리스크**: AI가 등록 가능성·신규성 여부를 단정했다가 틀렸을 때, 사용자가 그 판단을 신뢰하고 의사결정하면 서비스 책임 문제가 발생한다.
- **서비스 범위 외**: 명세서 작성·검토 서비스로서 이 범위를 벗어난 요청에 대한 응답을 제공하지 않는다. 특허 절차 대응 문서(심사관 의견서)나 제3자 특허 무효화 전략, 특허 관련이지만 명세서 작성과 직접 관계없는 지식 질문, 특허와 무관한 일반 질문

## 7. Orchestrator 사전테스트 인사이트

> 작성일: 2026-05-20  
> 데이터: `pretest-runs-orchestrator-20260520T075431Z.csv` (38건)  
> 대상: Orchestrator 모델 선정 실험 1 (§3.3)

### 전제

`chat`은 특정 특허 도구로 라우팅되지 않는 일반 응답 경로다. Orchestrator가 특허 도메인 발화를 `chat`으로 분류하면 Review/Author 에이전트가 호출되지 않고 일반 LLM 응답으로 처리된다. 아래 인사이트는 이 전제를 기준으로 결과를 해석한다.

### 테스트 범위

| 발화 | 건수 | 정답 intent |
|---|---|---|
| "독립항이 몇 개야?" | 19 | `review_claim` |
| "청구항 구조를 검토해줘" | 16 | `review_claim` |
| "명세서 검토해줘" | 1 | `review_spec` |
| "1항 2항의 관계를 설명해줘" | 1 | `review_claim` |
| "1항검토하고 문제있으면 알려줘" | 1 | (평가 불일치) |

### 인사이트 1 — `chat` 분류 = 파이프라인 우회, 단순 오분류가 아님

"독립항이 몇 개야?"에서 gpt-4.1과 Anthropic 계열 모델이 모두 `chat`을 반환했다. 평가 당시 일부는 👍로 기록됐지만, `chat` 라우팅의 의미를 적용하면 **특허 파이프라인이 완전히 바이패스되는 기능 장애**다. 변리사가 청구항에 대해 질문했을 때 일반 대화로 처리되는 것은 서비스 정의를 벗어난 응답이다.

| 모델 | "독립항이 몇 개야?" 실질 판정 |
|---|---|
| gpt-5.5 | 정확 (review_claim) |
| gpt-5-mini | 의도 정확, candidates 과다 (근거부족) |
| gpt-4.1 | **파이프라인 우회 (chat)** |
| claude-sonnet-4-6 | **파이프라인 우회 (chat)** |
| claude-opus-4-7 | **파이프라인 우회 (chat)** |
| gpt-4o-mini | 파이프라인 우회 (chat) |
| claude-haiku-4-5 | 파이프라인 우회 + 형식오류 |

### 인사이트 2 — 모델 세대가 특허 도메인 인식의 분기점

GPT-5 계열(5.5, 5-mini, 5-nano)은 "독립항이 몇 개야?"를 특허 전문 쿼리로 일관되게 인식한 반면, GPT-4 계열과 Claude 계열은 일반 대화로 분류했다. reasoning effort를 올려도 해소되지 않았다.

"청구항 구조를 **검토**해줘"는 모든 모델이 `review_claim`을 선택했다. 두 발화를 비교하면 분기점이 드러난다.

> "청구항 구조를 **검토**해줘" → 전 모델 `review_claim`  
> "독립항이 **몇 개야**" → GPT-5 계열만 `review_claim`, 나머지 `chat`

핵심 동사가 없는 구어체 질문에서 구형 모델과 Anthropic 모델이 도메인 컨텍스트를 유지하지 못하는 경향이 확인됐다. 발화가 짧고 구어적일수록 이 차이가 두드러질 가능성이 높다.

### 인사이트 3 — candidates 과다 포함은 프롬프트 문제, 모델 선택 문제 아님

"청구항 구조를 검토해줘" 16건 전체, 모든 모델이 `edit_spec`·`review_spec` 등 불필요한 도구를 candidates에 포함했다. reasoning effort를 xhigh까지 높여도 변하지 않았고, gpt-5.5도 동일하게 근거부족 판정을 받았다.

현재 프롬프트가 candidates 선정 기준을 충분히 명시하지 않는 것이 원인으로 판단된다. 개선 방향:

- **제약 방식:** "의미 있는 대안이 없으면 candidates 배열을 비우거나 1개로 제한"
- **채점 방식:** candidates는 "이 발화가 해당 도구로도 합리적으로 처리될 수 있는가" 기준으로만 포함

### 인사이트 4 — reasoning effort는 의도분류 정확도에 유효하지 않음

gpt-5.5를 none~xhigh 전 구간 테스트한 결과, 분류 결과와 confidence가 사실상 동일했다. `high`에서만 오히려 review_spec이 candidates에 추가되어 유일하게 근거부족을 받았다.

| effort | confidence | candidates | 결과 | 비용 | 지연 |
|---|---|---|---|---|---|
| none | 0.91 | [review_claim, chat] | **정확** | $0.060 | 1,328ms |
| low | 0.95 | [review_claim, chat] | **정확** | $0.062 | 2,084ms |
| medium | 0.82 | [review_claim, chat] | **정확** | $0.063 | 2,720ms |
| high | 0.92 | [review_claim, chat, review_spec] | 근거부족 | $0.063 | 2,510ms |
| xhigh | 0.95 | [review_claim, chat] | **정확** | $0.066 | 4,013ms |

**reasoning effort는 none 고정이 적절하다.** reasoning 자원은 Review·Author 에이전트에 할당하는 것이 효과적이다.

### 인사이트 5 — 테스트 커버리지가 협소함

38건 중 35건이 두 발화에 집중됐다. Intent taxonomy 7개 카테고리 중 5개가 검증되지 않았다.

### 인사이트 3 — 모델 품질 계층

**Tier A — 법적 정밀도 높음 (antecedent basis 포착)**
- claude-sonnet-4-6, claude-opus-4-7: claim 1의 "플랫폼" 내 구동계가 독립 선행사로 정의되지 않아 claim 2의 "상기 구동계" 인용이 불명확하다는 선행사(antecedent basis) 이슈를 포착. 다른 모델이 전혀 잡지 못한 법적으로 중요한 문제.

**Tier B — 주요 이슈 정확히 포착**
- gpt-5.5 (none/low), gpt-5-mini (low), claude-haiku-4-5: claim 2 하이브리드 의미 중복, claim 1 기능적 표현 문제를 올바르게 지적.

**Tier C — 표면적 분석**
- gpt-5-nano low, gpt-4.1: 이슈는 맞지만 법적 근거가 얕거나 단일 이슈만 제시. gpt-4.1은 642 토큰으로 지나치게 간략.

**Tier X — 사용 불가**
- gpt-4o-mini: 종속항이 "자율주행 차량 시스템"을 반복한다고 지적 → 종속항의 정상적인 구조를 오류로 인식. 특허 도메인 기초 지식 결여.
