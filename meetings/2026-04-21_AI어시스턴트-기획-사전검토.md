- 목적: `frd-spec-ai-assistant.md` 내용 개발팀 사전 공유 및 구현 가능성·비용 현실성 검토

> **한줄 요약:** 기획안의 전체 방향은 유효하나, 입력 컨텍스트 항목 일부 조정·에이전트 POC 우선 진행·비용 구조 설계를 병행하기로 하고 다음 주 월요일 전체 팀 기획 검토 회의를 잡기로 결정.

## 1. 기존 Q&A 챗봇 구조 참고

- 현재 "특허 자세히 보기" 화면에 이미 구현된 Q&A 챗봇을 명세서 AI 어시스턴트 기획의 참조 기반으로 활용하기로 함
- **확인 필요 사항**: LLM 연동 방식 / 컨텍스트 주입 방식 / 토큰 사용량 수준
- 명세서 AI 어시스턴트가 이 챗봇과 공통되는 부분과 차이점을 파악하면 구현 출발점이 명확해질 것

---

## 2. 입력 컨텍스트 구성 재검토

### 결정된 조정 사항

| 항목 | 기존 | 조정 | 사유 |
|---|---|---|---|
| 현재 명세서 전체 텍스트 — 청구항 생성 시 | 필수 | **권장** | 청구항 생성 후 명세서를 작성하는 게 정상 흐름. 명세서 전체가 이미 있는 경우는 재작업 상황이라 필수 전제 과도 |
| 비교발명 분석 — 명세서 본문/도면 생성 시 | 권장 | **선택 재검토** | 선행특허 분석 내용을 넣으면 오히려 선행특허와 유사한 내용이 생성될 위험 있음 |

### 유지 확인 사항

- **발명 자료**: 명세서 본문·도면 생성 시 필수 — 재확인 완료
- **기재불비 기준**: 이미 청구항/본문/도면 카테고리별로 분류돼 있으므로 생성 유형에 맞는 카테고리만 로드하면 토큰 절약 가능
- **비교발명 분석**: 청구항 생성 시 필수 유지

### 추가 논의된 사항

- **수동 수정 보호**: AI가 사용자가 직접 편집한 섹션을 다시 덮어쓰지 않아야 함. 프론트에서 변경된 부분을 감지해서 넘겨주는 방식으로 처리하기로 함 (실행 취소/되돌리기 기능과 연동)
- **인용 텍스트 드래그 입력**: 드래그 → 채팅 메시지에 인용으로 삽입하는 기능은 전체 완성 시 포함, 현재 컨텍스트 항목 분류에 영향 없음

---

## 3. 에이전트 오케스트레이션 구조

- **단일 에이전트 vs 멀티 에이전트**: 단일 에이전트에 도구를 명확히 정의해두면 오케스트레이터 없이도 처리 가능하다는 의견 (성연). 멀티 에이전트 구성은 규모가 더 커질 때 고려
- **오케스트레이션 노하우**: 공개된 레퍼런스가 부족하며, 실제 구현하며 데이터를 수집해가며 개선하는 방식 불가피
- **에이전트 동작 로그 표시**: AI가 어떤 도구를 호출했는지, 어떤 섹션을 변경했는지 UI에 표시 필요 (변리사가 AI 동작 추적 가능하도록)

---

## 4. 의도 분류 처리 방식

세 가지 케이스로 처리:

1. **의도 명확 + 읽기 전용** → 즉시 실행
2. **의도 명확 + 문서 변경** → 실행 전 확인(미리보기 또는 횟수 안내)
3. **의도 모호** → 선택지 카드 제공

모호한 의도 판단 기준은 규칙 기반으로 시작하되 실사용 데이터 수집 후 점진 개선.

---

## 5. 구현 단계(Phase) 조율

- **Phase 1 기준 재확인**: 읽기 전용 도구(선행기술 조회, 기재불비 검토) + 컨텍스트 기반 텍스트 출력. 명세서 섹션 생성 초안 기능도 Phase 1에 포함 검토
- **Phase 1.5**: 기재불비 검토 연동으로 생성 품질 고도화
- **Phase 2**: 에이전트형 문서 변경(일괄 적용, 변경 목록 표시, 수정 전후 diff UI)
- **현실적 목표**: Phase 1로 30~40% 수준 오픈 후 실사용 반응 보며 단계 진행
- **변경 목록 vs 수정 전후 비교 UI**: 구분 필요
  - 변경 목록: 패널 형태, 일괄 적용 또는 선택 적용
  - 수정 전후 비교: 원문에서 변경 부분 하이라이트, 세밀한 확인용

---

## 6. 비용 및 요금제 방향

- **읽기 전용 채팅 답변**: 무료 또는 일반 구독 포함
- **명세서 초안 생성**: 1회 차감 (기존 명세서 생성 요금 체계 그대로)
- **기재불비 검토 등 고급 AI 기능**: 유료 고객 한정 제공 (품질 집중 투자 영역)
- **한도 표시 방식**: 수치보다 퍼센테이지 방식 고려 (Claude 참고)
- **세션 도중 한도 초과**: 실행 중 끊기지 않도록 처리 필요. 한 명세서 건당 한도 설정 또는 퍼센테이지 기반 소진 방식 검토
- **에이전트 기능 전용 요금제**: 별도 요금 체계 신설 필요 여부 추가 검토 필요 (읽기 전용 답변 무료 / 문서 변경 실행 유료 구분 가능성)

## 7. POC 방향

에이전트 POC를 두 방향으로 병행하는 것이 출발점으로 적합:

1. **에이전트 프레임워크 스터디**: 오케스트레이션 구조, 도구 분류, 호출 판단 방식
2. **컨텍스트 조합별 품질 비교 테스트**: 어떤 입력 조합에서 어떤 모델이 좋은 결과를 내는지 검증
