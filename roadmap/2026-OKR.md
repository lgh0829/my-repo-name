## 1분기 OKR (2026 Q1)

### Objective: 검색 품질 및 고도화를 통해 사용자가 원하는 특허를 더 정확하게 찾을 수 있게 한다

---

#### KR 1. 검색 성능 업그레이드

| # | Key Result | 평가 방법 |
|---|-----------|---------|
| 1-1 | 검색 결과에 "왜 이 특허가 나왔는지" 이유 표시 기능 제공 | 기능 배포 여부 |
| 1-2 | "원하는 결과가 안 나와요" 버튼 도입 및 재검색/쿼리 수정 이벤트 수집 | 이벤트 수집 정상 동작 여부 |
| 1-3 | 청구항 / 요약 / 문제 / 효과를 필드별 벡터로 분리 임베딩 적용 | 기존 단일 벡터 대비 검색 품질 지표 향상 |
| 1-4 | Top-K 후보를 재평가하는 리랭킹 모델(또는 LLM) 운영 | 리랭킹 적용 전후 nDCG@K 비교 |
| 1-5 | Recall@K / nDCG@K 기반 검색 품질 평가 체계 수립 | 정기 평가 파이프라인 운영 여부 |
| 1-6 | 검색 쿼리 구조화 체계 개선 완료 | 구조화 쿼리 적용 후 검색 품질 지표 개선 |

#### KR 2. 검색 방식 고도화 (1차 검색 결과 기반 신규 분야 추천)

| # | Key Result | 평가 방법 |
|---|-----------|---------|
| 2-1 | 클릭 / 저장 / 청구항 확장 / 체류시간 / 추천 클릭 이벤트 수집 구현 | 이벤트 수집 항목 완비 여부 |

## 2분기 OKR (2026 Q2)

### Objective 1. 변리사가 PatSol 유료구독할 가치가 있는 제품이 되도록, 핵심 워크플로우를 고도화한다

---

#### KR

> 퍼널 구조: 활성 무료 이용자(모수) → 핵심 가치 경험 → 유료 전환
> 활성 기준: 주 1회 이상 로그인 (WAU 재방문율 내포)

| # | Key Result | 평가 방법 |
|---|-----------|---------|
| 1 | 분기 내 활성 무료 이용 변리사 60명 | 주 1회 이상 로그인 계정 수 |
| 2 | 활성 이용자 중 명세서 생성 완료 또는 다운로드 비율 (TBD%) | 세션 로그 및 다운로드 이벤트 기준 |
| 3 | 유료 전환 변리사 10명 | 분기 내 유료 결제 완료 계정 수 |
| 4 | 쿼리 응답 속도 p95 기준 35% 단축 | 개선 전후 p95 응답시간 비교 |

---

#### Initiative 1. AI 명세서 품질 개선
> 소폭 수정 후 출원 가능한 수준(최종 명세서 근접)까지 품질 끌어올리기 · 도면 생성 고도화 포함

**세부 작업**
- 변리사 실사용 피드백 수집 → 불만 유형 분류 (문장 어색함 / 누락 항목 / 포맷 불일치 / 기재불비 미반영)
- 프롬프트 개선 사이클 운영 (수집 → 수정 → 재검증 반복)
- 출력 포맷 표준화 (청구항·요약·상세설명 구조 고정)
- 도면 생성 품질 개선 (구성요소·관계·배치 정확도 향상)

| Sub-KR | 목표 |
|--------|------|
| 변리사 "소폭 수정만으로 출원이 가능하다" 응답률 | 60% 이상 |
| 명세서 완료 후 다운로드 또는 인앱 편집 시작율 | 60% 이상 |

---

#### Initiative 2. 고도화된 검색 엔진 적용
> 검색 성능 향상 및 데이터베이스 재정비 — 검색 품질이 전체 워크플로우 진입 품질을 결정

**세부 작업**
- 청구항 / 요약 / 문제 / 효과 필드별 벡터 분리 임베딩 적용
- 리랭킹 모델(또는 LLM reranker) 운영
- IP5 특허 데이터베이스 재정비 (한국, 미국 우선)
- 검색 쿼리 구조화 체계 개선

| Sub-KR | 목표 |
|--------|------|
| 검색 결과 정확도 | 80% 이상 |
| IP5 커버리지 (한국, 미국) | 80% 이상 |

---

#### Initiative 3. 핵심 워크플로우 UX 개선
> 검색 → 청구항 생성 → 명세서 생성 플로우 재구현 · 방법/장치 유형별 명세서 생성 지원

**세부 작업**
- 핵심 플로우 사용성 테스트 (파일럿 유저 5명 이상, 가이드 없이 완료 여부 확인)
- 이탈 구간 화면 재설계 (명세서 진입 전 단계 집중)
- 방법 발명 / 장치 발명 유형에 맞는 명세서 생성 분기 구현
- 온보딩 플로우 설계 (첫 로그인 → 명세서 완료 첫 경험 유도)

| Sub-KR | 목표 |
|--------|------|
| 가이드 없이 핵심 플로우 완료한 유저 비율 (활성유저 대비) | 60% 이상 |
| 검색 → 청구항 생성 → 명세서 완료 전환율 | 기존 대비 +20%p |

---

#### Initiative 4. AI 어시스턴스 LLM 설계 및 도입
> 워크플로우 단계별로 막히는 순간 AI가 맥락 있게 개입 — 완주율 향상

**세부 작업**
- 단계별 AI 어시스턴스 시나리오 정의 (검색식 생성 / 청구항 보완 / 명세서 항목 설명)
- 사용자 입력·선택 맥락을 LLM에 전달하는 컨텍스트 설계
- 챗 UI vs 인라인 제안 UI 방식 결정 및 IA 반영
- 프롬프트 관리 체계 수립 (단계별 시스템 프롬프트 버전 관리)

| Sub-KR | 목표 |
|--------|------|
| 어시스턴스 응답 품질 내부 리뷰 통과 | 시나리오별 3회 이상 검증 |
| 도입 이후 유저의 다음 단계 진행률 | 미사용 대비 +15%p |

#### Initiative 1. 스터디 운영

> 실제 명세서 1건 작성 완료를 목표로, 특허 출원 전 과정을 직접 경험하며 도메인 이해도 확보

**커리큘럼 1 — 특허 출원 기본정보**
- 특허 출원 절차 전반 (출원 → 심사 → 등록)
- 명세서 구성 요소 이해: 청구항 / 요약 / 상세설명 / 도면
- 기재불비 유형 및 거절이유 사례 학습

**커리큘럼 2 — 검색**
- 출원 주제 선정 기준
- 출원 주제 관련 선행특허 검색 실습
- 검색 결과 선별 기준: 무엇이 비교발명이 되는가
- 선별한 검색 건에서 명세서 작성에 필요한 정보 추출 방법

**커리큘럼 3 — 명세서 작성**
- 비교발명 내용을 바탕으로 청구항 구성하기
- 기재불비 체크리스트 기준으로 자가 검토
- 선행특허 검색 건과 작성 명세서를 1:1 대조 — 심사관 시각으로 미흡 지점 분석
- 개선 접근법 스터디 및 토론

**공통 — 페인포인트 기록**
- 커리큘럼 단계마다 막혔던 지점 문서화
- 세션 종료 후 팀 공유 → 제품 개선 인사이트로 연결

| Sub-KR | 목표 |
|--------|------|
| 커리큘럼 1~3 전 세션 운영 완료 | 일정 내 완료 여부 |
| 인당 명세서 1건 작성 완료 | 전원 달성 여부 |
| 페인포인트 수집 건수 | 인당 5건 이상 문서화 |
| 수집된 페인포인트 중 제품 개선 후보로 전환된 항목 | 1건 이상 |

#### Initiative 2. 내부용 특허 Q&A 봇 구축

> 스터디 중 반복 질문 해소 + 이후 신규 팀원 온보딩 자산으로 활용

**세부 작업**
- 자료 수집: 특허청 심사기준 / 기재불비 유형 / 명세서 예시 / 사내 작성 사례
- 자료 정제 및 청킹 (Q&A 형식으로 구조화)
- RAG 기반 Q&A 봇 구축 (사내 슬랙 or 웹 인터페이스)
- 스터디 진행 중 실사용하며 커버리지 보완

| Sub-KR | 목표 |
|--------|------|
| 수집·정제 문서 건수 | 30건 이상 |
| 봇 구축 및 팀 내 배포 완료 | 커리큘럼 2 시작 전 |
| 스터디 중 봇 활용 질문 건수 | 20건 이상 |
| 봇 미응답(모르겠음) 비율 | 30% 이하 |

## 채점 기준

| 점수 | 의미 |
|------|------|
| 0.7 ~ 1.0 | 목표 달성 |
| 0.4 ~ 0.6 | 부분 달성 — 원인 분석 필요 |
| 0.0 ~ 0.3 | 미달성 — 다음 분기 재검토 |
