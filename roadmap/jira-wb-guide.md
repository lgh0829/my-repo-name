# Jira + Workbase(WB) 사용 가이드

> Jira와 WB 목표관리를 함께 활용하는 방법을 안내합니다. 이슈 유형별 생성 규칙, OKR 연동 구조, 역할별 체크리스트를 담고 있습니다.

## 목적

**일한 것이 자연스럽게 목표와 연결되는 문화**를 만들기 위해 Jira와 WB 목표관리를 함께 운영합니다.

팀원은 "내가 왜 이 일을 하는가"를 스스로 확인할 수 있고, 리드와 동료는 별도 보고 없이도 서로의 업무 흐름을 파악할 수 있습니다. 분기가 끝났을 때는 무엇을 했고 어떤 성과를 냈는지 명확하게 돌아볼 수 있습니다. 이를 통해 임팩트 있는 일에 집중하는 팀 문화를 만들어 갑니다.

다음을 지양합니다.

- 업무 흐름을 Jira에 기록하지 않아 진행상황 파악이 어려워지는 것
- 업무를 진행하기 위한 사전 작업(탐색·리서치·검토 등)을 기록할 수 없어 본인의 업무 흐름을 관리하기 어려워지는 것
- 기록 부담이 커져 Jira 생성 자체를 꺼리게 되는 것
- 팀원마다 다른 방식으로 Jira를 사용해 분기 회고 시 현황 파악이 어려워지는 것

## Jira와 Workbase의 사용 방법

### Workbase 목표관리

팀의 분기 목표(OKR)를 설정하고 달성률을 추적하는 공간입니다. KR은 Jira의 Epic과 연동되어, Epic 하위의 작업 현황이 Workbase 목표관리에 자동으로 반영됩니다.

### Jira

Jira는 목표 달성을 위한 팀원 각자의 실행을 기록하는 공간입니다.

- **Epic**: KR 하나에 해당하는 업무 묶음. 리드가 생성하고, 팀원은 여기서 업무 방향을 확인합니다.
- **Task / Story / Bug**: 1~2주 안에 완료할 수 있는 실행 단위. 팀원이 자율적으로 만들고, WB에 자동 반영됩니다.
- **Subtask**: Task를 진행하는 과정에서 생기는 작은 작업 단위. WB에 반영되지 않으며 본인 업무 흐름 관리용입니다.

#### 어떻게 사용하나요?

**1단계 — Epic을 확인합니다 (리드 → 팀원)**

리드가 분기 시작 전 KR 기준으로 Epic을 생성하고, 설명에 실행 방향을 작성합니다. 팀원은 Epic 설명을 보고 "이번 분기에 무엇을 왜 해야 하는가"를 파악합니다.

Epic 설명은 다음 내용을 포함합니다.

| 항목 | 내용 |
|------|------|
| Objective | 상위 목표 |
| 배경 (Why) | 이 KR이 필요한 이유 |
| 실행과제 (Initiative) | KR 달성을 위한 실행 방향 목록 |

실행과제는 팀원과의 검토를 거쳐 분기 중에도 유연하게 수정할 수 있습니다.

**2단계 — Task를 만듭니다 (팀원)**

Epic 설명을 읽고 본인이 담당할 업무를 Task로 만듭니다. 1~2주 안에 완료할 수 있는 크기로 쪼개고, End date를 반드시 입력합니다. Task는 WB에 자동 반영되므로 별도 연결 작업이 필요 없습니다.

Task 설명에는 작업 배경(Why), 작업 내용(What), 완료 조건(Acceptance Criteria), OKR 연결 근거를 포함합니다.

**3단계 — Subtask로 진행 흐름을 관리합니다 (팀원)**

Task를 진행하는 과정에서 생기는 사전 작업(탐색·리서치·검토 등)은 Subtask로 기록합니다. Subtask는 WB에 동기화되지 않으며, 본인의 업무 흐름을 관리하기 위한 용도입니다.

#### 이슈 유형별 규칙

> 이슈 생성 시 `wand-protocol/rules/JIRA.md`를 참고합니다. 제목/설명/담당자/스프린트/기한이 반드시 포함되어야 합니다.

**Epic**
- 리드가 생성하며, 1 Epic = 1 KR입니다.
- Sprint와 독립적으로 동작합니다. Epic 기간이 Sprint보다 길어도 Sprint는 정상 종료됩니다.

**Task / Story**
- 팀원이 자율적으로 생성하며, 원칙적으로 Epic 하위에 생성합니다. (상황에 따라 예외 허용)
- 타 팀원의 작업과 연관이 생기면 "연결된 업무 항목"을 활용합니다.

**Bug**
- Epic 하위에 생성하여 OKR 체계 안에 포함합니다.
- 버그 수정·지원사업 등은 별도 Objective를 수립하여 Epic과 연결합니다.

**Subtask**
- Task 진행 과정에서 생기는 사전 작업을 기록합니다.
- **WB에 동기화되지 않습니다.** OKR 평가에 포함되지 않습니다.

#### 예시

**Epic (= KR)**
> 명세서에 연동 가능한 특허 데이터 구조 분해 30% 완료

```
Objective: 정확한 특허 검색을 위한 데이터베이스 체계 확립
배경 (Why): 명세서와 특허 데이터를 연동하려면 데이터 구조를 먼저 파악해야 함
실행과제 (Initiative):
  1. [분석] KIPRIS Plus 특허 데이터 구조 분석
     - KIPRIS Plus 데이터 필드 조사
     - 패밀리 특허 구조 분석
     - 특허 공보 / 법적 상태 변화 분석
  2. [설계] 특허 데이터 스키마 초안 설계
```

**Task** (팀원이 Epic을 보고 생성)
- `[분석] KIPRIS Plus 데이터 필드 조사` — 담당자: 한상호 / End date: 3/28
- `[분석] 패밀리 특허 구조 분석` — 담당자: 한상호 / End date: 4/4

**Subtask** (Task 하위에서 진행 흐름 관리)
- KIPRIS Plus API 문서 검토
- 샘플 데이터 수집 및 필드 목록 정리

#### 운영 규칙

**방향이 바뀌었을 때**
기존 티켓을 수정하지 않고, Drop 처리 후 새 티켓을 생성합니다.

**Task 크기 기준**
Sprint 단위(1~2주)를 기준으로 완료할 수 있는 크기로 만듭니다. 너무 크면 진행 상황이 보이지 않습니다.

**리서치·분석·테스트 Task**
OKR 매핑 시 평균점수 산출 방식에서 불리하게 작용할 수 있습니다. 보완된 평가 방식은 별도 논의 예정입니다.

### 구조 개요

```
OKR (WB 목표관리)
└── KR (WB 목표관리) = Epic (Jira)
        │   └─ 설명에 실행과제(Initiative) 방향 기술
        ├── Bug (Jira)
        └── Task / Story (Jira)
                └── Subtask (Jira)  ← WB 동기화 제외
```

## OKR 설정 가이드 (2분기~)

| 개념 | 역할 | 변경 가능 여부 |
|------|------|:---:|
| Objective | 분기 목표 | X |
| Key Result = Epic | 목표 달성 판단 기준 | 보수적으로 추가 |
| Initiative = Epic Description | 목표 달성 수단 (실행과제) | 유연하게 수정 가능 |

**2분기 방향**
- Culture·Operation OKR 비중을 줄이고, **Product·비즈니스 OKR을 늘립니다.**
- Objective는 목표지향적으로 작성합니다. (예: "검색/명세작성/도면생성 기능 구현", "UX 개선")
- 버그 수정·지원사업도 별도 Objective로 수립합니다.
- KR은 측정 가능하고 MECE하게 정의합니다.

## 체크리스트

**분기 시작 (리드)**
- [ ] WB에서 OKR(Objective + KR) 확정
- [ ] KR 기준으로 Jira Epic 생성
- [ ] Epic Description에 Objective, Why, Initiative 작성

**분기 중 (팀원)**
- [ ] Epic Description 기반으로 본인 Task 생성 (1~2주 단위)
- [ ] Task에 End date 입력
- [ ] 개인 메모·진행상황은 Subtask 활용
- [ ] 방향 재조정 시 기존 티켓 Drop → 신규 생성

**분기 회고**
- [ ] Epic 기준으로 OKR 달성률 검토
- [ ] 누락된 KR·Task 확인 및 다음 분기 반영
