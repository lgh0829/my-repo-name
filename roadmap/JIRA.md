# Wand JIRA Protocol

Wand 팀의 JIRA 이슈 작성 프로토콜입니다. 모든 업무는 JIRA로 트래킹되며, **AI 기반 성과 평가의 핵심 입력 데이터**가 됩니다. 이슈 품질이 곧 평가 정확도와 직결되므로 이 프로토콜을 반드시 준수하세요.

---

## 1. 왜 이슈 품질이 중요한가

성과 평가는 다음 흐름으로 진행됩니다:

```
JIRA 이슈 작성 → OKR 연결 → 작업 수행 → 동료 투표 → AI 최종 평가
```

AI는 이슈의 **제목, 설명, 유형, 우선순위, 스토리 포인트, 라벨, 연결된 OKR** 정보를 기반으로 4개 카테고리를 평가합니다:

| 카테고리 | 가중치 | 설명 |
|----------|--------|------|
| **OKR 기여도** | **최상** | 연결된 KR 달성에 얼마나 직접적으로 기여하는가 |
| 기술적 복잡도 | 높음 | 기술적 난이도와 변경 범위 |
| 비즈니스 영향력 | 높음 | 사용자/매출/운영에 미치는 영향 |
| 업무 필수도 | 보통 | 이 작업 없이 다른 작업 진행이 가능한가 |

> **핵심**: OKR 기여도가 0~2점이면 최종 점수는 최대 3.0을 넘지 못합니다. OKR과 무관한 작업은 아무리 기술적으로 뛰어나도 높은 평가를 받을 수 없습니다.

## 2. 이슈 계층 구조

```
Epic  (= KR 단위, 분기마다 생성)
  ├─ Story  (PM/Designer 작성 — 기능 명세/정책 정의)
  │    └─ Task  (Story 연관 구현 작업 — Linked Issues로 연결)
  │         └─ Sub-task  (작성자 본인의 Task 세부 분해)
  └─ Task  (Story 없이 Epic에 직접 연결되는 독립 작업)
       └─ Sub-task  (작성자 본인의 Task 세부 분해)
```

Story를 거치지 않고 Epic에 직접 연결되는 Task도 존재합니다. 인프라, 데이터, 운영, 문서 등 기능 명세가 선행될 필요 없는 작업이 이에 해당합니다.

이슈 간 연결은 **Linked Issues**를 활용합니다 (→ 6. 연결 규칙 참고).

## 3. Epic 가이드라인

### Epic = KR (Key Result)

Epic은 분기 OKR의 KR 단위로 생성합니다. Epic 하나가 KR 하나에 대응됩니다.

### Epic 제목 형식

```
[YYYY QN] KR 제목
```

예시:
- `[2026 Q1] 검색 성능 업그레이드`
- `[2026 Q2] 명세서 수정 기능`
- `[2026 Q2] 도면 생성 기능 — 도면 설명 텍스트 생성`

### Epic 설명 템플릿

```markdown
## Objective
이 Epic이 속한 분기 Objective를 한 줄로 기술합니다.

## KR 목표
- KR 항목과 평가 방법을 원문 그대로 옮깁니다.
- 예: Top-K 후보를 재평가하는 리랭킹 모델 운영 / nDCG@K 비교

## 기간
- 시작: YYYY-MM-DD
- 종료: YYYY-MM-DD (분기 말)

## 현재 진행률
- 0% (Epic 생성 시 기입, 이후 주기적으로 업데이트)

## 연관 Story / Task
- 이 Epic에 연결된 주요 이슈를 간략히 나열 (자동 집계 보완용)
```

### Epic 운영 원칙

- **작성 주체**: PO 또는 팀 리드
- **생성 시점**: 분기 OKR 확정 직후
- **Epic 수정**: KR 목표가 변경되면 Epic 설명도 함께 업데이트
- **완료 처리**: 분기 종료 후 KR 달성 결과를 Description 하단에 기록 후 Done 처리

## 4. 이슈 유형 (Issue Type) 선택

| 유형 | 작성 주체 | 사용 시점 |
|------|-----------|-----------|
| **Epic** | PM / 팀 리드 | KR 단위 목표 트래킹 |
| **Story** | PM / Designer | 사용자 기능 명세, 정책 정의, UX 기준 수립 |
| **Task** | 누구나 | 구현, 기획, 데이터, 운영, 문서 등 모든 실무 작업 |
| **Bug** | 누구나 | 기존 기능의 오류 수정 |
| **Sub-task** | 누구나 | 본인 소유 Story/Task의 세부 분해 |

### 유형 선택 기준

```
기능을 정의/설계하는 작업인가?
├── Yes → Story (PM/Designer만)
└── No → 기존 기능의 오류인가?
          ├── Yes → Bug
          └── No → Task
```

## 5. 이슈 제목 (Summary) 작성법

### 형식

```
[작업유형] 대상 - 구체적 행동/목적
```

### Prefix 목록

| Prefix | 용도 | 작성 주체 | 예시 |
|--------|------|-----------|------|
| `[feature]` | 기능 명세 및 정책 정의 | PM / Designer | `[feature] 검색 결과 — 유사도 근거 표시 기능 명세` |
| `[feat]` | 새로운 기능 구현 | 개발자 | `[feat] 유사도 분석 - 청구항 기반 비교 알고리즘 추가` |
| `[fix]` | 버그 수정 | 누구나 | `[fix] 로그인 - 소셜 인증 토큰 만료 처리` |
| `[refactor]` | 구조 개선 | 개발자 | `[refactor] 검색 모듈 - Milvus 쿼리 최적화` |
| `[perf]` | 성능 개선 | 개발자 | `[perf] 대시보드 - 초기 로딩 3초→1초 이내로 개선` |
| `[infra]` | 인프라/배포 | 개발자 | `[infra] CI/CD - GitHub Actions 배포 파이프라인 구축` |
| `[design]` | UI/UX 설계 및 변경 | Designer | `[design] 검색 결과 - 카드 컴포넌트 리디자인` |
| `[docs]` | 문서 작업 | 누구나 | `[docs] API 명세 - Swagger 문서 작성` |
| `[data]` | 데이터 파이프라인/분석 | 데이터 | `[data] KIPRIS 수집 - 재시도 로직 및 누락 알림 추가` |
| `[chore]` | 기타 잡무 | 누구나 | `[chore] 의존성 - 보안 패치 업데이트` |

### 제목 작성 원칙

| 규칙 | Good | Bad |
|------|------|-----|
| 구체적 대상 명시 | `[feat] 유사도 분석 - 청구항 기반 비교 알고리즘 추가` | `기능 추가` |
| 행동과 목적 포함 | `[fix] 명세서 생성 - PDF 내보내기 시 한글 깨짐 수정` | `버그 수정` |
| 측정 가능한 표현 | `[perf] 대시보드 - 초기 로딩 3초→1초 이내로 개선` | `속도 개선` |

## 6. 연결 규칙 (Linked Issues)

Story와 구현 Task는 **Sub-task 관계가 아닌 Linked Issues로 연결**합니다. Sub-task는 본인 이슈의 세부 분해에만 사용합니다.

### 관계 타입

| 관계 | 사용 시점 |
|------|-----------|
| `implements / is implemented by` | Task가 Story의 내용을 구현할 때 |
| `is blocked by / blocks` | 선행 작업이 완료되어야 시작 가능할 때 |
| `relates to` | 직접 의존은 없지만 참고 관계일 때 |

### 연결 예시

```
[feature] 검색 결과 — 유사도 근거 표시 기능 명세  (Story, PM 소유)
  ← implements ─  [feat] 검색 결과 - 근거 텍스트 BE API 구현  (Task, BE 소유)
  ← implements ─  [feat] 검색 결과 - 근거 표시 UI 컴포넌트  (Task, FE 소유)
  ← implements ─  [design] 검색 결과 - 근거 표시 화면 설계  (Task, Designer 소유)
```

### OKR 연결과의 관계

- **Story**: 해당 Epic(KR)에 직접 연결
- **Story 연관 Task**: Story에 `implements` 링크로 연결하면 OKR 별도 연결 불필요
- **독립 Task** (Story 없이 단독으로 수행하는 작업): 직접 Epic에 연결

## 7. 이슈 설명 (Description) 작성법

설명은 AI 평가에서 **3,000자까지** 읽힙니다.

### Story 전용 템플릿 (PM/Designer)

```markdown
## 기능 개요 (What & Why)
- 어떤 기능/컴포넌트인가?
- 왜 필요한가? (사용자 문제 또는 비즈니스 목적)

## 정책 및 규칙
- 동작 조건, 예외 처리, 제약 사항
- 엣지 케이스

## 디자인
- Figma 링크:
- 주요 화면/인터랙션 설명

## 미결 사항 (Open Questions)
- [ ] 결정이 필요한 항목

## 완료 조건 (Acceptance Criteria)
- [ ] 조건 1
- [ ] 조건 2
```

---

### Story 연관 Task 템플릿

Story에 `implements` 링크가 걸린 Task는 상단에 연관 정보를 명시합니다.

```markdown
## 연관 Story
- Story 티켓: [PS-XXX] 기능명
- 담당 범위: (이 Task가 Story 중 어느 부분을 구현하는지 한 줄 요약)

## 배경 (Why)
이 작업이 필요한 이유와 현재 상황을 설명합니다.

## 작업 내용 (What)
- 변경 대상 (모듈, API, 화면 등)
- 주요 변경 사항 목록

## 기술적 접근 (How) — 개발 이슈인 경우
- 사용 기술/라이브러리
- 아키텍처 변경 여부
- 영향 받는 모듈/서비스

## 완료 조건 (Acceptance Criteria)
- [ ] 조건 1
- [ ] 조건 2
```

---

### 독립 Task 템플릿 (Story 연관 없는 작업)

```markdown
## 배경 (Why)
이 작업이 필요한 이유와 현재 상황을 설명합니다.

## 작업 내용 (What)
- 변경 대상 (모듈, API, 화면 등)
- 주요 변경 사항 목록

## 기술적 접근 (How) — 개발 이슈인 경우
- 사용 기술/라이브러리
- 아키텍처 변경 여부
- 영향 받는 모듈/서비스

## 완료 조건 (Acceptance Criteria)
- [ ] 조건 1
- [ ] 조건 2

## OKR 연결 근거
- 연결된 KR: (KR 제목)
- 기여 방식: (직접적/간접적 기여 설명)
- 예상 KR 진행률 변화: (예: 현재 30% → 작업 완료 후 45%)
```

## 8. 우선순위 (Priority) 설정

| 우선순위 | 기준 | 예시 |
|----------|------|------|
| **Highest** | 서비스 장애, 데이터 손실 위험 | 프로덕션 DB 연결 불가 |
| **High** | 핵심 기능 장애, 마감 임박 | 결제 실패, 스프린트 마감 전 필수 작업 |
| **Medium** | 일반적인 개발/기획 업무 | 신규 기능 구현, 명세 작성 |
| **Low** | 개선 사항, 기술 부채 | 코드 정리, 문서 업데이트 |
| **Lowest** | 나이스 투 해브 | UI 미세 조정, 주석 추가 |

## 9. 스토리 포인트 (Story Points) 산정

스토리 포인트는 **기술적 복잡도 평가**의 핵심 참고 지표입니다.

| 포인트 | 난이도 | 기준 |
|--------|--------|------|
| **1** | 매우 쉬움 | 설정 변경, 텍스트 수정 (1~2개 파일) |
| **2** | 쉬움 | 단순 버그 수정, 기존 패턴 내 소규모 변경 |
| **3** | 보통 | 기존 패턴 내 기능 추가, 일반적인 CRUD |
| **5** | 복잡함 | 여러 모듈 변경, 새로운 비즈니스 로직 설계 |
| **8** | 매우 복잡함 | 아키텍처 변경, 외부 시스템 통합 |
| **13** | 극히 복잡함 | 신기술 도입, 대규모 리팩토링 → **이슈를 분할하세요** |

> 13 이상은 이슈를 나누는 것을 권장합니다. 세분화된 이슈가 평가에서도 더 정확한 점수를 받습니다.

## 11. OKR 연결 가이드

**OKR 연결은 평가에서 가장 중요한 요소입니다.**

### 연결 원칙

1. **작업 시작 전에 연결**: 이슈 생성 시점에 OKR을 연결합니다
2. **가장 직접적인 KR 선택**: 여러 KR에 연관될 경우, 가장 직접적으로 기여하는 KR을 선택합니다
3. **무관한 KR에 억지 연결 금지**: 적합한 KR이 없으면 팀 리드와 논의하여 새로운 KR을 만들거나 Objective를 조정하세요

### 이슈 유형별 연결 방법

| 이슈 유형 | OKR 연결 방법 |
|-----------|---------------|
| Epic | OKR의 KR과 1:1 매핑 (Epic 자체가 KR) |
| Story | 해당 Epic에 직접 연결 |
| Story 연관 Task | Story에 `implements` 링크 → OKR 별도 연결 불필요 |
| 독립 Task | 해당 Epic에 직접 연결 + Description에 연결 근거 명시 |

### KR 진행률과의 관계

AI는 KR의 현재 진행률도 고려합니다:

- **KR 진행률이 낮을 때** (0~30%) 핵심 기여 작업 → 높은 OKR 기여도
- **KR 진행률이 높을 때** (90%+) 소규모 작업 → 낮은 OKR 기여도

독립 Task의 Description에 "현재 KR 진행률 30%이며, 이 작업으로 45%까지 끌어올릴 수 있음"처럼 구체적으로 명시하면 더 정확한 평가를 받습니다.

## 12. 평가 점수를 높이기 위한 체크리스트

이슈 작성 완료 후, 제출 전에 확인하세요.

### 필수 체크

- [ ] **제목**에 작업 유형 Prefix, 대상, 구체적 행동이 포함되어 있는가?
- [ ] **설명**에 배경(Why)과 작업 내용(What)이 명확히 구분되어 있는가?
- [ ] **완료 조건**이 체크리스트로 작성되어 있는가?
- [ ] **이슈 유형**이 작업 성격에 맞게 선택되어 있는가?
- [ ] **우선순위**가 적절하게 설정되어 있는가?
- [ ] **스토리 포인트**가 산정되어 있는가?
- [ ] **OKR 또는 Story**에 연결되어 있는가?

### 권장 체크

- [ ] 설명에 **기술적 접근(How)**이 포함되어 있는가? (개발 이슈)
- [ ] **라벨**이 1개 이상 설정되어 있는가?
- [ ] 스토리 포인트 13 이상인 경우, **이슈가 분할**되어 있는가?
- [ ] 설명에 **측정 가능한 수치** (before/after)가 포함되어 있는가?

## 13. 이슈 완료 시 처리

이슈를 Done으로 전환하기 전에:

1. **완료 조건 체크**: Description의 체크리스트를 모두 완료 상태로 업데이트
2. **결과 기록**: 실제 작업 결과를 코멘트로 남기기
   ```
   ## 작업 결과
   - 응답 시간: 8초 → 1.5초 (81% 개선)
   - 변경 파일: 12개 (search 모듈 전면 리팩토링)
   - 관련 PR: PS-123
   ```
3. **예상과 다른 점 기록**: 처음 계획과 달라진 부분이 있으면 명시

> 완료 후 기록이 충실할수록 동료 투표 시 정확한 평가가 가능하며, AI 최종 평가도 더 공정해집니다.

## 14. 안티 패턴

| 안티 패턴 | 문제점 | 올바른 방법 |
|-----------|--------|-------------|
| 제목만 쓰고 설명 비워두기 | AI가 "설명 없음"으로 판단, 낮은 평가 | 최소 배경 + 작업 내용 작성 |
| "기타 작업", "잡무" 같은 모호한 제목 | 기여도 판단 불가 | 구체적 대상과 행동 명시 |
| 하나의 이슈에 여러 작업 몰아넣기 | 복잡도/기여도 산정 왜곡 | 논리적 단위로 분할 |
| OKR과 무관한 KR에 억지 연결 | AI가 불일치 감지, 오히려 감점 | 적합한 KR 없으면 팀 리드와 논의 |
| 스토리 포인트 미설정 | 기술적 복잡도 판단 근거 부족 | 반드시 산정 |
| Story를 개발자가 작성 | 명세와 구현의 경계 혼재 | Story는 PM/Designer만 |
| Sub-task로 타인 이슈에 달기 | 소유권 혼재, 평가 왜곡 | Linked Issues 활용 |
| 완료 후 아무 코멘트 없이 Done 전환 | 동료 투표/AI 평가 시 근거 부족 | 작업 결과 코멘트 남기기 |
