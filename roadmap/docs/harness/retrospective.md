# 분기 회고 (Quarterly Retrospective)

**트리거**: "N분기 회고", "retro", 분기 Jira CSV를 주며 회고 요청 시.

먼저 `csv_parsing.md`(파싱)와 `okr_mapping.md`(Epic↔OKR)를 함께 참조한다.

---

## 절차

1. **기준선 로드** — `2026-OKR.md`에서 해당 분기 Objective·KR·Initiative와 채점 기준(0.7=달성)을 읽는다.
2. **실적 집계** — CSV에서 다음을 계산:
   - 전체: 이슈 수 / 완료율 / 드롭 수 / 미완료 목록
   - Epic별: 완료율, 미완료·드롭 이슈
   - 처리량: 월별 resolved 추이, cycle time(만듦→해결됨) 중앙값·p90
   - 인원: 담당자별 완료 건수 (SP는 입력 누락 많으므로 보조 지표로만)
3. **OKR 채점** — 각 KR/Sub-KR에 0.0~1.0 점수. **측정 데이터가 없으면 점수 대신 `[측정불가]`로 표기**하고 그 자체를 회고 항목으로 올린다. (지표 트래킹 Initiative의 실패는 다른 모든 KR을 채점불가로 만든다 — 이 사실을 1순위 발견으로 명시)
4. **정성 분석** — Keep / Problem / Try 구조. 데이터에서 관찰된 사실 → 원인 가설 → 다음 액션 순.
5. **다음 분기 인풋** — 미완료 이월 후보, 드롭 사유 점검, 우선순위 제안.

---

## 채점 원칙 (`2026-OKR.md` 따름)

| 점수 | 의미 |
|---|---|
| 0.7~1.0 | 달성 (stretch goal 기준 0.7이 "잘함") |
| 0.4~0.6 | 부분 달성 — 원인 분석 |
| 0.0~0.3 | 미달성 — 재검토 |

- **출력 완료율 ≠ OKR 달성**. 이슈를 다 쳐냈어도 KR 지표(활성 60명·유료 10명 등)가 측정·달성되지 않으면 달성이 아니다. 이 구분을 회고에서 명확히 한다.
- 추정/사실/가설을 섞지 않는다. 데이터로 말할 수 없는 부분은 "측정 안 됨"이라고 쓴다.
