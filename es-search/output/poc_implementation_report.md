# Spec Embodiment 자동생성 PoC — 구현 인계 리포트

> **작성 목적**: PoC에서 검증된 분석 결과 및 프롬프트 아키텍처를 구현 담당자에게 전달
> **기준 일자**: 2026-06-10
> **PoC 범위**: 공통 문체 블록 + 발명 본질 유형 블록 A(도면-공간) 검증 완료. 발명 본질 유형 B·C 미검증.
> **검증 대상 프롬프트**: `output/generated_prompts/_template.md`

---

## 1. 배경 및 문제 정의

### 1-1. 현행 파이프라인 문제

운영 중인 명세서 자동생성 파이프라인(P8 `get_specification_embodiment()`)이 생성하는 실시예는
아래 두 가지 KIPO 심사 기준에서 반복적으로 부족함이 확인됨:

| 문제 | 원인 | 관련 법조항 |
|---|---|---|
| 종속항 구성요소 누락 | P8에 독립항 1개만 입력 | 특허법 42조 4항 (뒷받침 요건) |
| 부호 불일치·임의 부호 생성 | 부호 매핑 정보 없이 생성 | 특허법 42조 3항 (실시가능 요건) |

### 1-2. PoC 목표

1. 3개 주요 특허 사무소(김앤장, 리앤목특허법인, 장수길 변리사) 명세서 3,192건 전수 분석으로 **정량적 스타일 패턴** 도출
2. 분석 결과를 P8 프롬프트에 반영하여 KIPO 기재요건 통과율 향상
3. 재현 가능한 프롬프트 아키텍처(공통 문체 + 발명 본질 유형 조합) 설계·검증

## 2. 핵심 분석 발견 (구현에 직접 연결되는 항목만)

### 2-1. 두 독립 축 발견 ★★★

명세서 작성 스타일은 서로 독립적인 두 축으로 구성됨이 실증됨:

```
축 1 — 문체 (Style Axis)
  결정 요인: 대리인(사무소/변리사)
  특성: 청구항 구조, 선행인용 방식, 과제 어법, 면책 문구 위치
  변동성: 사무소별 고정 (동일 사무소는 CPC 무관하게 동일 문체)

축 2 — 발명 본질 유형 (Invention Essence Type)
  결정 요인: CPC 분류코드
  특성: 실시예 본문의 서술 전략 (도면 중심 / 동작 순서 / 실험 수치)
  변동성: 동일 사무소도 CPC에 따라 발명 본질 유형 다름
```

**→ 구현 함의**: 프롬프트를 `[문체 블록] + [발명 본질 유형 블록]` 두 레이어로 분리 조합해야 함.
단일 프롬프트로 모든 발명을 처리하면 전개 패턴이 왜곡됨.

### 2-2. 발명 본질 유형 3종 분포 (국내 특허 3,192건 기준)

| 발명 본질 유형 | 비율 | CPC | 서술 전략 |
|---|---|---|---|
| **도면-공간** | ~68% | H·B·E·F (기계·전자) + G01·G02·G03 일부 | 도면 순서로 구조 전개, 공간 배치 중심 |
| **실험-수치** | ~20% | C·H(공정) (화학·반도체) + G03F(리소그래피) | 공정 단계 + 수치 범위 한정 + 비교예 |
| **동작-시간** | ~10% | G06·G11·G16 (SW·알고리즘) | 장치 블록도 → 방법 흐름도 2-layer, S단계 시간순 전개 |

> ⚠️ **G계열 내부 분화 주의**: G계열 전체가 동작-시간형이 아님 (실측 394건 분석).
> - G06·G16·G11: 흐름도(S110) 비율 25~80% → 동작-시간 우세
> - G01 (측정/분석): 흐름도 14% → 도면-공간형에 가까움
> - G02·G03 (광학/리소): 흐름도 5~10% → 도면-공간 또는 실험-수치로 분류

**→ 구현 함의**: CPC 첫 두 글자만으로 발명 본질 유형을 자동 결정하면 G계열 오분류 발생.
G로 시작하는 경우 세 번째 자리(G06/G01/G02 등)까지 확인 후 분기해야 정확도 확보 가능.
`get_cpc_chatgpt()` 결과를 발명 본질 유형 분기에 사용 시 G계열 세부 분기 로직 추가 필요.

### 2-3. 문체 클러스터 3종 (body style clustering v2)

JP(일본 출원인) 제외, 국내 출원 기준:

| 클러스터 | 비율 | 특성 | 주요 사무소 |
|---|---|---|---|
| A — 선행인용형 | 20% | 배경기술에 특허문헌 명시 인용 | 김앤장 일부 |
| B — 직접과제형 | 25% | 과제를 "~문제가 있었다" 직접 기술 | 장수길 |
| C — 간접·표준형 | 47% | 과제를 "~가 요구된다" 간접 기술 | 리앤목, 김앤장 다수 |

**PoC 검증 대상**: C형(간접·표준형) 단일 문체. 현재 _template.md는 C형 고정.

### 2-4. 사무소별 주요 정량 지표

| 지표 | 김앤장 | 리앤목 | 장수길 |
|---|---|---|---|
| 선행기술 특허문헌 인용률 | 84% | 10% | 20% |
| 복수 실시예("제1실시예…") 비율 | 23% | 3% | 5% |
| 과제 섹션 채움률 | 94% | 100% | 97% |
| 효과 섹션 채움률 | 99% | 65% | 70% |
| 독립항 전략 (장치단독/순수형 비율) | 45% | 60% | 52% |

> ⚠️ `SolutionProblem` 필드는 사무소 간 채움률 편차 극심(김앤장 50% / 리앤목·장수길 0%)으로 횡단 비교 부적합.

## 3. 프롬프트 아키텍처 (검증된 범위)

### 3-1. 조합 구조

```
최종 P8 시스템 프롬프트
= [역할 선언]            ← 구현 시 추가 필요 (현재 누락)
+ [공통 문체 블록]        ← PoC 검증 완료 ✅
+ [내용 충실도 규칙 F1~F4] ← PoC 검증 완료 ✅
+ [발명 본질 유형 블록 (A/B/C 중 1개)] ← 발명 본질 유형 A만 PoC 검증 ✅ / B·C 미검증

Human Turn
= [청구항 전체]          ← 독립항 + 종속항 모두 (기존 독립항만 → 수정 필요)
+ [P_REF 출력 또는 §3 목록]  ← 신규 단계 (기존 없음 → 추가 필요)
+ [도면의 간단한 설명]
```

### 3-2. 공통 문체 블록 — 검증된 규칙

아래 규칙들은 5건 생성 테스트에서 준수 여부 검증됨:

| 규칙 | 내용 | 위반 시 증상 |
|---|---|---|
| **번호 목차 금지** | "1. 구성", "2.1. …" 식 보고서형 구조 금지 | LLM 기본 출력에서 자주 위반 |
| **청구항 용어 유지** | 청구항 용어를 임의로 재명명하지 말 것 | "마이크로프로세서"→"제어부" 등 |
| **실시예 단일 흐름** | "제1실시예/제2실시예" 분할 금지 | 국내 표준형 C에서 부적합 |
| **효과 별도 섹션** | 효과 서술을 실시예 본문에 넣지 말 것 | 실시예/효과 섹션 중복 |
| **면책 문구 1회** | AND 조건이 아닌 OR 조건으로 1문단만 | 초기 버전에서 2문단 중복 삽입 |
| **영문 약어 인라인 병기** | "HF(high frequency)" 형식 | 약어 무설명 또는 용어집 별도 분리 |

### 3-3. 발명 본질 유형 블록 A (도면-공간) — 검증된 규칙 5가지

| 규칙 | 내용 |
|---|---|
| (1) 면책 문구 위치 | `{example}` 스타일 따름. 없으면 말미 간결형. **1문단만** |
| (2) 도면 도입 설명 | 도 N 첫 언급 시 "도 N은 본 발명의 일 실시예에 따른 …이다." 1줄 선행 |
| (3) 구조도 전개 패턴 | 구성 나열 문단 → 구성별 설명 문단 순서 고정. '상기' 재참조 |
| (4) 연속 참조 생략 | 동일 도면 연속 참조 시 "도 N을 참조하면" 반복 금지 |
| (5) 복합 도면 처리 | 대표 구조도 상술 후, 나머지 도면은 한 문단으로 묶음 |

**분량 기준**: 8,000자 ±20% (6,400~9,600자). 청구항 수 무관 고정.

## 4. 신규 단계: P_REF (구성요소·부호 매핑)

### 4-1. 도입 배경

기존 P8는 부호 정보 없이 실시예를 생성하여 **임의 부호 생성** 문제가 반복됨.
P_REF는 P9(도면 간단설명 생성) 직후 / P8 직전에 실행하여 부호 일관성을 보장함.

### 4-2. 실행 분기 로직

```python
if §3_구성요소목록_있음:          # 기존 특허 (도면부호 목록 제공)
    → §3 목록을 그대로 P8 {drawings}에 주입  # P_REF 불필요
else:                              # 신규 발명
    → P_REF 실행 (claim_text + drawings 입력)
    → 출력: <구성요소_부호_목록> + <도면별_구성요소>
    → P8 {drawings} 앞에 주입
```

### 4-3. P_REF 출력 포맷

```xml
<구성요소_부호_목록>
(100) 복합식 가습기 — 등장 도면: 도1, 도2, 도3
(110) 수조 — 등장 도면: 도2, 도3
(120) 초음파발진기 — 등장 도면: 도2, 도3
...
</구성요소_부호_목록>

<도면별_구성요소>
도 1 (사시도): (100) 복합식 가습기, (130) 송풍기, ...
도 2 (단면도): (110) 수조, (120) 초음파발진기, (140) 증발기, ...
</도면별_구성요소>
```

### 4-4. PoC 검증 결과

- **CMP 발명 (17청구항)**: P_REF 적용 전 → 스크린필터 0회 언급. 적용 후 → 15회 언급.
- **디스플레이 장치 (20청구항)**: 부호 일관성 대폭 향상. 도면 누락 없음.

## 5. 변수 입력 스펙 변경 사항

### 5-1. `{claim_text}` — 수정 필요 ⚠️

| 현행 | 변경 후 |
|---|---|
| 독립항 1개만 전달 | **독립항 + 종속항 전체** `¶` 구분 전달 |
| 예: 청구항 1만 | 예: 청구항 1¶청구항 2¶…¶청구항 N |

**이유**: 종속항에만 등장하는 구성요소가 실시예에 누락되어 뒷받침 요건 위반.

### 5-2. `{output_length}` — 수정 필요 ⚠️

| 현행 | 변경 후 |
|---|---|
| 코드 하드코딩 `700` | `8000` 또는 `{output_length}` 플레이스홀더 |

**주의**: 파이프라인 코드(`prompts.yaml` id:8 호출부)에서 700 → 8000으로 수정 필요.

### 5-3. `{drawings}` — 확장

| 현행 | 변경 후 |
|---|---|
| 도면의 간단한 설명 원문만 | P_REF 출력 `<구성요소_부호_목록>` + `<도면별_구성요소>` + 원문 |

## 6. 내용 충실도 규칙 F1~F4 (KIPO 기재요건 대응)

구현 시 시스템 프롬프트에 아래 4개 규칙을 반드시 포함해야 함:

| 규칙 | 내용 | 대응 기재요건 |
|---|---|---|
| **F1** 전수 반영 | 전체 청구항(독립+종속) 모든 구성요소를 실시예에 포함 | 42조 4항 뒷받침 |
| **F2** 감지→동작 연결 | 센서/측정 구성이 있으면 그 다음 제어·판단·대응 동작까지 서술 | 42조 3항 실시가능 |
| **F3** 도면 누락 금지 | 도면의 간단한 설명에 언급된 도면을 본문에서 모두 참조 | 42조 3항 실시가능 |
| **F4** 부호 일치 | P_REF 또는 §3 목록의 부호를 그대로 사용. 임의 생성 금지 | 도면 부합 요건 |

> ⚠️ F4 추가 제약: 청구항·도면·제공 자료에 없는 정량 수치는 임의 생성하지 말 것(환각 방지).

## 8. 구현 우선순위

### 즉시 수정 (High Impact)

1. **P8 `{claim_text}` 입력**: 독립항만 → 전체 청구항 (`¶` 구분)
2. **P8 `{output_length}` 값**: 700 → 8000 (코드 또는 prompts.yaml 수정)
3. **P_REF 단계 추가**: P9와 P8 사이에 신규 프롬프트 단계 삽입

### 권장 수정 (Medium Impact)

4. **발명 본질 유형 자동 결정 로직**: `get_cpc_chatgpt()` 결과 → A/B/C 분기 코드 추가
5. **역할 선언 추가**: 시스템 프롬프트 최상단에 "당신은 한국 특허 명세서 작성 전문가입니다" 추가
6. **프롬프트 XML 구조화**: `[ ]` 표기법 → `<style_rules>`, `<content_rules>`, `<invention_essence>` 태그로 전환

### 향후 과제 (미검증)

7. **발명 본질 유형 블록 B (동작-시간)**: G06·G11·G16 계열 SW 발명 대상 별도 검증 필요 (실측 평균 22,200자 / 2-layer: 장치 블록도 → 방법 흐름도)
8. **발명 본질 유형 블록 C (실험-수치)**: C계열 화학·공정 발명 대상 별도 검증 필요 (수치 환각 위험 높음)
9. **문체 다변화**: 현재 C형(간접·표준형) 단일 고정. A형(선행인용) · B형(직접과제) 추가 시 별도 문체 블록 필요

## 9. 검증 체크리스트 (QA 기준)

생성된 실시예의 품질 확인 시 아래 항목을 점검:

**형식 (공통)**
