<실시예>
본 실시예들은 호스트(host)가 어플리케이션(application)의 오프로드(offload) 처리를 요청함에 따라 컨트롤러(controller)가 메모리 장치(memory device)에 접근하여 상기 어플리케이션을 처리하고, 호스트가 어플리케이션 처리에 사용되는 데이터를 메모리 장치의 타겟 영역(target region)에 저장함에 따라 상기 타겟 영역의 접근 권한(access permission)을 컨트롤러가 갖도록 설정하며, 어플리케이션의 처리 상태(processing state)에 따라 상기 타겟 영역의 접근 권한을 컨트롤러 또는 호스트가 갖도록 동적으로 설정하는 공유 메모리 관리부(shared memory manager)를 포함하는 데이터 처리 시스템(data processing system)에 관한 것이다. 이하, 도 1 내지 도 14를 참조하여 다양한 실시예를 상세히 설명한다. 이하에서 동일 또는 유사한 구성요소에는 동일 또는 유사한 참조부호를 부여할 수 있으며, 중복되는 설명은 간략히 하거나 생략할 수 있다.

(1) 전자 장치 및 데이터 처리 시스템의 개요(도 1, 도 2)
도 1은 일 실시예에 의한 데이터 처리 시스템을 포함하는 전자 장치의 구성도이다. 도 1을 참조하면, 전자 장치(100)는 호스트(110) 및 데이터 처리 시스템(120)을 포함할 수 있다. 호스트(110)는 중앙처리장치(CPU), 그래픽처리장치(GPU), 신경망처리장치(NPU) 중 적어도 하나를 포함하는 프로세서(111), 메인 메모리(112), 및 호스트 인터페이스(113)를 포함할 수 있다. 데이터 처리 시스템(120)은 메모리 장치(130) 및 컨트롤러(140)를 포함할 수 있다.

호스트(110)는 어플리케이션 실행 중 일부 연산(예: 검색/필터링/집계/압축/복호화/벡터 연산/머신러닝 전처리 등)을 데이터 이동 비용을 최소화하기 위해 저장장치 측으로 오프로드하도록 구성될 수 있다. 즉, 호스트(110)는 오프로드 요청(command)을 컨트롤러(140)에 전송하고, 컨트롤러(140)는 메모리 장치(130)에 직접 접근하여 해당 어플리케이션 처리를 수행함으로써 호스트(110)로의 대량 데이터 이동을 감소시킬 수 있다.

도 2는 일 실시예에 의한 데이터 처리 시스템을 포함하는 전자 장치의 구성도이다. 도 2를 참조하면, 전자 장치(200)는 네트워크 인터페이스(201), 사용자 인터페이스(202), 저장장치 인터페이스(203) 등을 더 포함할 수 있다. 저장장치 인터페이스(203)는 PCIe, NVMe, CXL, UFS, SATA, SAS 등의 표준 또는 커스텀 인터페이스일 수 있다. 특히 본 실시예에서는 호스트(110)와 컨트롤러(140) 사이의 명령 및 데이터 교환이 PCIe/NVMe 기반으로 수행되는 예를 중심으로 설명하나, 이에 한정되지 않는다.

데이터 처리 시스템(120)은 오프로드 처리를 위하여 메모리 장치(130) 내부 또는 인접한 위치에서 수행되는 연산(near-data processing, in-storage computing)을 지원할 수 있다. 이때 호스트(110)는 오프로드 대상 데이터(입력 데이터, 파라미터, 중간 결과 또는 출력 버퍼 등)를 메모리 장치(130)의 타겟 영역(예: 특정 논리 주소 범위, 특정 네임스페이스, 특정 LUN/파티션, 특정 메모리 윈도우)에 저장할 수 있고, 공유 메모리 관리부(150)는 해당 타겟 영역의 접근 권한을 처리 단계에 따라 컨트롤러(140) 또는 호스트(110)에 부여하도록 설정할 수 있다.

(2) 메모리 모듈 및 메모리 장치 구성(도 3)
도 3은 일 실시예에 의한 메모리 모듈의 구성도이다. 도 3을 참조하면, 메모리 모듈(300)은 복수의 메모리 다이(memory die)(310_1 내지 310_n)와, 모듈 인터포저 또는 패키지 기판(320) 및 배선/채널(330)을 포함할 수 있다. 각 메모리 다이(310)는 비휘발성 메모리(예: NAND 플래시) 또는 휘발성 메모리(예: DRAM, SRAM), 혹은 이들의 조합으로 구성될 수 있다. 본 실시예에서 메모리 장치(130)는 NAND 기반 SSD 형태일 수 있으며, 컨트롤러(140)는 플래시 변환 계층(FTL)을 포함하여 논리 주소와 물리 주소를 매핑할 수 있다.

메모리 장치(130)는 논리적 관점에서 사용자 데이터 영역(user data area)과 메타데이터 영역(metadata area), 오프로드 실행을 위한 타겟 영역(또는 공유 영역)(target/shared area), 및 펌웨어/로그 영역 등을 포함할 수 있다. 타겟 영역은 호스트(110)가 오프로드 처리에 사용될 데이터(입력/출력/중간 데이터/모델 파라미터)를 저장하는 영역으로 정의될 수 있다. 타겟 영역은 예컨대 (i) NVMe Namespace 단위로 분리되거나, (ii) 특정 LBA 범위로 설정되거나, (iii) 컨트롤러(140) 내부의 메모리 윈도우 테이블에 의해 지정될 수 있다.

(3) 컨트롤러 구성 및 오프로드 처리부(도 4, 도 5)
도 4는 일 실시예에 의한 컨트롤러의 구성도이다. 도 4를 참조하면, 컨트롤러(140)는 호스트 인터페이스(141), 메모리 인터페이스(142), 제어부(143), 오프로드 처리부(144), 버퍼 메모리(145) 및 공유 메모리 관리부(150)를 포함할 수 있다. 제어부(143)는 펌웨어 또는 마이크로코드에 의해 동작하며, 호스트로부터 수신한 오프로드 요청을 해석하고 필요한 내부 작업을 스케줄링할 수 있다. 오프로드 처리부(144)는 하드웨어 가속기(예: 암복호 엔진, 압축 엔진, 정규식 매칭 엔진, 벡터 연산 엔진, 합/최대/최소 집계 엔진, ML 전처리 엔진 등) 또는 프로그래머블 프로세서(예: RISC 코어, DSP, NPU)를 포함하여 어플리케이션 처리를 수행할 수 있다.

도 5는 다른 관점에서 본 컨트롤러의 구성도이다. 도 5를 참조하면, 컨트롤러(140)는 복수의 실행 컨텍스트(execution context) 또는 큐(queue)를 운용할 수 있고, 오프로드 요청별로 작업 식별자(job ID) 또는 트랜잭션 식별자(TID)를 부여할 수 있다. 각 작업(job)은 입력 데이터 위치(타겟 영역 내 주소), 출력 데이터 위치(타겟 영역 내 주소 또는 별도 영역), 처리 함수 ID, 파라미터, 권한 정책 등을 포함할 수 있다. 컨트롤러(140)는 작업 상태를 “대기(queued)”, “입력 준비(input ready)”, “실행 중(running)”, “완료(done)”, “오류(error)”, “호스트 인계(hand-off)” 등으로 구분하여 관리할 수 있다.

본 실시예에서 오프로드 처리부(144)는 타겟 영역에 저장된 입력 데이터를 직접 읽어 처리한 뒤 결과를 타겟 영역 또는 별도 지정 영역에 기록할 수 있다. 이때 공유 메모리 관리부(150)는 처리 상태에 따라 타겟 영역의 접근 권한을 컨트롤러(140) 또는 호스트(110)에 부여하도록 설정하여, (i) 호스트와 컨트롤러의 동시 접근에 의해 발생할 수 있는 데이터 레이스(race), (ii) 미완료 결과를 호스트가 조기에 읽는 문제, (iii) 처리 중 입력 데이터가 변경되는 문제 등을 방지할 수 있다.

(4) 공유 메모리 관리부의 구성(도 6)
도 6은 일 실시예에 의한 공유 메모리 관리부의 구성도이다. 도 6을 참조하면, 공유 메모리 관리부(150)는 (i) 타겟 영역 관리 테이블(151), (ii) 권한 설정부(152), (iii) 상태 추적부(153), (iv) 트랩/차단부(154), 및 (v) 통지/동기화부(155)를 포함할 수 있다.

타겟 영역 관리 테이블(151)은 타겟 영역을 식별하기 위한 정보(예: 시작 LBA, 길이, 네임스페이스 ID, 영역 ID, 메모리 윈도우 핸들)와, 해당 영역에 대한 접근 권한 소유자(owner) 또는 접근 모드(mode)(예: Host-RO, Host-RW, Ctrl-RO, Ctrl-RW, Exclusive-Ctrl, Shared) 등을 저장할 수 있다. 권한 설정부(152)는 호스트(110)가 타겟 영역에 데이터를 저장한 후 오프로드 처리를 요청할 때, 또는 컨트롤러(140)가 오프로드 실행을 시작/완료할 때, 소유자 및 접근 모드를 갱신할 수 있다.

상태 추적부(153)는 작업 ID별 또는 영역 ID별 처리 상태를 추적할 수 있다. 예를 들어, 상태 추적부(153)는 “호스트 준비 단계(Host preparing)”, “컨트롤러 실행 단계(Ctrl running)”, “결과 기록 단계(Result writing)”, “완료/호스트 접근 허용 단계(Host allowed)” 등으로 상태를 구분할 수 있다. 트랩/차단부(154)는 현재 권한 정책에 반하는 접근이 감지될 경우 해당 접근을 차단하거나 지연시킬 수 있다. 예컨대 호스트가 컨트롤러 독점(exclusive) 권한이 부여된 타겟 영역에 읽기/쓰기를 시도하면, 트랩/차단부(154)는 NVMe 오류 상태(예: Access Denied, Command Aborted)로 응답하거나, 완료 큐에 상태를 반영하여 호스트에 통지할 수 있다. 통지/동기화부(155)는 권한 전환 이벤트, 작업 완료 이벤트 등을 호스트에 인터럽트 또는 폴링 레지스터 업데이트 방식으로 통지할 수 있다.

(5) 타겟 영역의 설정 방식 및 예시
타겟 영역은 오프로드 처리 단위(작업 단위)로 동적으로 할당될 수도 있고, 시스템 설정 시 정적으로 예약될 수도 있다.

(가) 정적 예약 예시
호스트(110)는 SSD 초기화 과정에서 특정 네임스페이스(NS_offload)를 오프로드 전용으로 생성할 수 있다. 이 네임스페이스의 전체 주소 범위를 타겟 영역으로 간주하고, 공유 메모리 관리부(150)는 이 네임스페이스에 대해 기본 접근 모드를 “호스트 RW, 컨트롤러 RO”로 설정할 수 있다. 이후 호스트가 오프로드 요청을 제출하면, 공유 메모리 관리부(150)는 해당 작업의 입력/출력 범위에 대해 “컨트롤러 RW(또는 Exclusive-Ctrl)”로 전환할 수 있다.

(나) 동적 할당 예시
호스트(110)는 오프로드 요청과 함께 “타겟 영역 핸들 요청”을 수행하고, 컨트롤러(140)는 가용 영역(Free list)에서 일정 크기의 타겟 영역을 할당하여 핸들(handle)을 반환한다. 호스트는 반환된 핸들에 대응하는 LBA 범위에 데이터를 기록한 후, “데이터 준비 완료” 플래그를 포함하는 오프로드 실행 명령을 전송할 수 있다. 이때 공유 메모리 관리부(150)는 “호스트 데이터 저장 단계” 동안은 Host-RW를 허용하고, “실행 단계” 진입 시 Ctrl-Exclusive로 전환할 수 있다.

(6) 데이터 처리 시스템의 동작 방법(도 7, 도 8)
도 7 및 도 8은 실시예들에 따른 데이터 처리 시스템의 동작 방법을 설명하기 위한 흐름도이다. 도 7을 참조하면, 데이터 처리 시스템(120)의 동작 방법은 다음 단계들을 포함할 수 있다.

S710: 타겟 영역 설정 단계  
호스트(110)는 오프로드 처리에 사용할 타겟 영역을 설정한다. 이는 정적 예약 방식(예: 특정 네임스페이스 지정) 또는 동적 할당 방식(핸들 기반)일 수 있다. 공유 메모리 관리부(150)는 타겟 영역 관리 테이블(151)에 영역 정보 및 초기 권한을 기록한다.

S720: 호스트 데이터 저장 단계  
호스트(110)는 어플리케이션 처리에 사용될 데이터(입력/파라미터/출력 버퍼 초기화 등)를 타겟 영역에 저장한다. 이 단계에서 타겟 영역의 접근 권한은 호스트에 부여될 수 있으며(Host-RW), 컨트롤러는 필요 시 읽기만 허용(Ctrl-RO)되거나 완전히 차단될 수도 있다.

S730: 오프로드 요청 단계  
호스트(110)는 컨트롤러(140)에 오프로드 처리를 요청한다. 오프로드 요청은 (i) 처리 함수 ID, (ii) 입력 데이터 위치(타겟 영역 오프셋/길이), (iii) 출력 위치, (iv) 검증 정보(예: 체크섬), (v) 권한 전환 정책 등을 포함할 수 있다.

S740: 권한 전환(컨트롤러 소유) 단계  
공유 메모리 관리부(150)는 호스트가 데이터를 타겟 영역에 저장함에 따라, 그리고 오프로드 요청이 수신됨에 따라 타겟 영역의 접근 권한을 컨트롤러(140)가 갖도록 설정한다. 예컨대 해당 범위에 대해 Host 접근을 Read-only로 낮추거나(Host-RO), 완전히 차단(Host-Deny)하고 컨트롤러에 Ctrl-RW 또는 Ctrl-Exclusive를 부여한다.

S750: 컨트롤러 오프로드 실행 단계  
컨트롤러(140)는 메모리 장치(130)에 접근하여 타겟 영역의 데이터를 읽고 어플리케이션 처리를 수행한다. 수행 중간에 결과가 타겟 영역에 기록되는 경우 공유 메모리 관리부(150)는 호스트가 미완료 결과를 읽지 못하도록 권한을 유지한다.

S760: 처리 상태에 따른 권한 전환(호스트 또는 컨트롤러) 단계  
상태 추적부(153)는 작업 상태 변화(예: 실행 완료, 결과 기록 완료, 오류 발생, 부분 완료)를 감지하고, 권한 설정부(152)는 그 상태에 따라 타겟 영역의 접근 권한을 컨트롤러 또는 호스트가 갖도록 설정한다. 예를 들어 정상 완료 시 Host-RW로 복귀시키고, 오류 시에는 Host-RO로 제한하면서 오류 로그 영역만 Host-RW로 허용하는 식의 정책이 가능하다.

S770: 완료 통지 및 후속 처리 단계  
통지/동기화부(155)는 작업 완료를 호스트(110)에 통지한다. 호스트는 결과를 타겟 영역에서 읽거나, 후속 오프로드 작업의 입력으로 재사용할 수 있다.

도 8을 참조하면, 다른 실시예로서 “부분 권한 전환”을 적용하는 흐름이 가능하다. 즉, 타겟 영역 전체에 대해 일괄적으로 권한을 전환하는 대신, 입력 영역과 출력 영역을 분리하여 각각 다른 권한 정책을 적용할 수 있다. 예컨대 실행 중에는 입력 영역을 Ctrl-RO(입력 안정성 확보를 위해 Host-Deny), 출력 영역을 Ctrl-RW(Host-Deny)로 두되, 실행이 상당 부분 진행된 후 출력의 일부 블록이 확정되면 확정 구간에 대해 Host-RO를 허용하여 파이프라이닝된 소비(consume)가 가능하도록 할 수 있다.

(7) 권한 모델 및 정책의 구체 예
본 실시예에서 “접근 권한”은 읽기/쓰기/삭제/트림(TRIM)/맵핑/캐시 플러시 등의 권한을 포함할 수 있다. 또한 권한은 “누가(owner)”뿐 아니라 “어떤 방식(mode)”으로 접근 가능한지를 포함할 수 있다.

(가) 배타적(Exclusive) 모델  
컨트롤러 실행 단계 동안 타겟 영역(또는 해당 작업 범위)을 컨트롤러가 배타적으로 소유한다. 호스트의 해당 범위 접근은 모두 차단된다. 실행 완료 후 Host-RW로 복귀된다. 이 모델은 구현이 단순하며 데이터 일관성 보장이 강하다.

(나) 읽기 공유(Read-share) 모델  
컨트롤러 실행 중에도 호스트의 읽기만 허용하고 쓰기는 차단할 수 있다(Host-RO). 다만 이 경우 호스트가 미완료 결과를 읽는 것을 방지하기 위해 결과 위치에 대해서는 별도 영역 분리 또는 “유효 비트(valid bit)” 관리가 필요하다.

(다) 영역 분할(Segmented) 모델  
타겟 영역을 입력/출력/중간 영역으로 세분화하고 각각에 대해 상이한 권한을 적용한다. 예컨대 입력은 Host-Deny/Ctrl-RO, 출력은 Host-Deny/Ctrl-RW, 로그는 Host-RO/Ctrl-RW 등으로 설정할 수 있다.

(8) 개념도에 따른 권한 전환 예(도 9, 도 10)
도 9 및 도 10은 실시예들에 따른 데이터 처리 시스템의 동작 방법을 설명하기 위한 개념도이다. 도 9를 참조하면, 시간축(t)에 따라 타겟 영역의 소유권(owner)이 Host→Controller→Host로 전환되는 모습을 도시할 수 있다. 구체적으로, 구간 T1에서는 호스트(110)가 입력 데이터를 타겟 영역에 기록하므로 Host-RW이고, 구간 T2에서는 오프로드 실행을 위해 컨트롤러(140)가 해당 영역을 독점하므로 Ctrl-Exclusive이며, 구간 T3에서는 작업 완료 후 호스트가 결과를 읽고 후속 처리를 수행하므로 Host-RW로 복귀한다.

도 10을 참조하면, 입력 구간과 출력 구간이 분리된 타겟 영역에서 권한이 다르게 전환되는 예가 도시될 수 있다. 예컨대 입력 영역 A는 실행 동안 Ctrl-RO로 유지되고 Host는 접근 불가이며, 출력 영역 B는 실행 동안 Ctrl-RW로 유지되되, “확정된 출력 블록”에 한해 Host-RO가 순차적으로 허용될 수 있다. 이를 위해 공유 메모리 관리부(150)는 출력 블록 단위의 유효성 맵(validity map) 또는 완료 비트맵(done bitmap)을 유지할 수 있다.

(9) 오프로드 어플리케이션 처리 예시들
본 실시예의 어플리케이션은 특정 도메인에 한정되지 않는다. 다만 청구항의 요지를 명확히 하기 위해, 컨트롤러(140)가 메모리 장치(130)에 접근하여 처리하고, 공유 메모리 관리부(150)가 타겟 영역 권한을 상태에 따라 전환하는 대표 예시를 몇 가지 제시한다.

(가) 데이터 필터링/검색 오프로드 예  
호스트(110)는 로그 데이터 또는 레코드 집합을 타겟 영역에 저장한다. 오프로드 요청에는 “키워드 매칭”, “범위 조건 필터”, “정규식 검색” 등의 조건이 포함된다. 컨트롤러(140)는 타겟 영역을 스캔하여 매칭되는 레코드의 인덱스 목록 또는 결과 레코드만을 출력 영역에 기록한다. 처리 중에는 Host-Deny로 입력이 변경되지 않도록 하고, 완료 후 Host-RW로 복귀하여 결과를 읽을 수 있게 한다.

(나) 압축/암호화 오프로드 예  
호스트(110)는 원본 데이터를 타겟 영역 입력 구간에 저장하고, 출력 구간을 예약한다. 컨트롤러(140)는 압축 또는 암호화 연산을 수행하여 출력 구간에 저장한다. 실행 중에는 출력 구간을 Ctrl-RW로 하여 컨트롤러만 쓰게 하고, 완료 후 출력 구간을 Host-RO 또는 Host-RW로 전환하여 호스트가 전송/저장/후처리에 사용하게 한다.

(다) 집계/통계 오프로드 예  
호스트는 대규모 수치 데이터(예: 센서 데이터)를 타겟 영역에 저장하고, “평균/분산/최대/상위 K” 등을 오프로드로 요청한다. 컨트롤러는 데이터 이동 없이 메모리 장치 내부/근처에서 집계를 수행하여 작은 결과만 호스트가 읽도록 한다. 권한 전환은 실행 중 Ctrl-Exclusive로 설정하여 입력이 변형되는 것을 방지한다.

(10) 접근 권한 제어의 구현 예
공유 메모리 관리부(150)의 권한 제어는 다양한 레벨에서 구현될 수 있다.

(가) 논리 블록 주소(LBA) 필터 방식  
컨트롤러(140)는 호스트로부터 들어오는 읽기/쓰기 명령의 LBA 범위를 검사한다. 해당 범위가 타겟 영역이며 컨트롤러 소유로 설정된 경우, 호스트 명령을 거부하거나 지연한다. 이 방식은 FTL 상단 또는 NVMe 명령 처리 레이어에서 구현 가능하다.

(나) 네임스페이스/큐 단위 ACL 방식  
오프로드 전용 네임스페이스를 구성하고, 오프로드 실행 중에는 해당 네임스페이스에 대한 호스트 I/O 큐를 일시적으로 제한한다. 예를 들어 특정 Queue ID에 대해서만 접근 허용을 두거나, 특정 명령(opcode)만 허용한다.

(다) 메모리 윈도우/키 기반 방식  
호스트가 접근할 때 키(key) 또는 토큰(token)을 제시하도록 하고, 공유 메모리 관리부(150)가 현재 유효한 키만 허용한다. 오프로드 실행 전환 시 키를 갱신하여 이전 키로는 접근이 불가하게 할 수 있다. 이는 보안 요구가 있는 환경에서 유리하다.

(11) 처리 상태에 따른 권한 전환의 세부 예
처리 상태(processing state)는 최소한 “실행 전/실행 중/실행 완료”를 포함할 수 있고, 더 세분화될 수 있다. 예를 들어:
- State 0: Idle(작업 없음) → Host-RW
- State 1: HostPreparing(호스트 입력 기록 중) → Host-RW, Ctrl-Deny(또는 Ctrl-RO)
- State 2: OffloadRequested(요청 수신) → 전환 준비(Host-RO 또는 Host-Deny로 변경)
- State 3: CtrlRunning(컨트롤러 실행) → Ctrl-Exclusive
- State 4: ResultCommitting(결과 기록/메타 업데이트) → Ctrl-Exclusive 유지
- State 5: Done(완료) → Host-RW(또는 Host-RO, 정책에 따라)
- State 6: Error(오류) → Host-RO + 오류 로그 Host-RW 등

공유 메모리 관리부(150)는 이러한 상태 전이를 상태 추적부(153)에서 감지하고, 권한 설정부(152)를 통해 타겟 영역 관리 테이블(151)을 갱신한다. 트랩/차단부(154)는 갱신된 정책에 따라 즉시 접근을 차단한다.

(12) 적층형 반도체 장치로의 적용(도 11 내지 도 13)
도 11 내지 도 13은 실시예들에 의한 적층형 반도체 장치의 구성도이다. 본 실시예의 데이터 처리 시스템(120)은 2.5D/3D 적층 패키징을 통해 구현될 수 있다. 예컨대, 도 11을 참조하면, 컨트롤러 다이(410)와 메모리 다이(420)가 인터포저(430) 상에 적층 또는 병렬 배치될 수 있다. 컨트롤러 다이(410) 내에 오프로드 처리부(144) 및 공유 메모리 관리부(150)가 구현되고, 메모리 다이(420)는 NAND/DRAM 등을 포함할 수 있다.

도 12를 참조하면, 컨트롤러와 메모리 사이가 TSV(Through Silicon Via)로 연결되는 3D 적층 구조가 가능하다. 이 경우 메모리 장치 접근 지연이 감소하여 오프로드 처리 효율이 증가한다. 도 13을 참조하면, 복수의 메모리 스택과 하나 이상의 컨트롤러 스택이 패키지 내에서 연결될 수 있고, 공유 메모리 관리부(150)는 스택 간 타겟 영역의 권한을 일관되게 관리하기 위해 전역 테이블(global table)과 로컬 캐시(local cache)를 함께 운용할 수 있다.

(13) 네트워크 시스템으로의 확장(도 14)
도 14는 일 실시예에 의한 네트워크 시스템의 구성도이다. 도 14를 참조하면, 네트워크 시스템(900)은 서버(910), 스위치/패브릭(920), 및 저장 노드(930)를 포함할 수 있다. 저장 노드(930)는 데이터 처리 시스템(120)을 포함하는 저장 장치 또는 저장 서버일 수 있다. 서버(910)는 호스트(110)의 역할을 수행하며, 오프로드 요청을 원격으로 전송할 수 있다. 이때 타겟 영역의 접근 권한 제어는 원격 DMA(RDMA) 또는 NVMe-oF 환경에서도 필요하며, 공유 메모리 관리부(150)는 원격 호스트의 읽기/쓰기 요청을 동일한 정책으로 차단/허용할 수 있다.

예를 들어, 서버(910)가 오프로드 입력 데이터를 저장 노드(930)의 타겟 영역에 기록한 뒤 오프로드 요청을 전송하면, 저장 노드(930)의 공유 메모리 관리부(150)는 오프로드 실행 동안 해당 영역에 대한 원격 읽기/쓰기를 차단한다. 작업 완료 후에만 결과 읽기를 허용함으로써, 네트워크 기반 공유 환경에서도 데이터 일관성과 보안을 확보할 수 있다.

(14) 구체적인 시나리오 기반 동작 예(권한 전환 포함)
이하에서는 “호스트가 타겟 영역에 데이터를 저장 → 컨트롤러 오프로드 처리 → 처리 상태에 따라 권한 전환”이라는 청구항 1의 구성을 충실히 만족하는 구체 시나리오를 제시한다.

[시나리오 1: 암호화 오프로드]
(a) 타겟 영역 할당  
호스트(110)는 컨트롤러(140)에 타겟 영역 할당을 요청하고, 컨트롤러는 타겟 영역 핸들 H1(예: NS=3, LBA=0x100000~0x10FFFF)을 반환한다. 공유 메모리 관리부(150)는 테이블(151)에 영역 ID=R1을 생성하고 초기 상태를 Host-RW로 설정한다.

(b) 데이터 저장  
호스트는 평문 데이터(예: 64MB)를 R1의 입력 구간에 기록하고, 출력 구간(동일 R1 내 후반부 또는 별도 R2)을 예약한다.

(c) 오프로드 요청  
호스트는 “AES-GCM 암호화” 함수 ID, 입력 주소, 출력 주소, IV/키 핸들, 태그 저장 위치 등을 포함한 오프로드 요청을 전송한다.

(d) 권한 전환  
공유 메모리 관리부(150)는 상태를 CtrlRunning으로 전환하면서 입력/출력 범위 전체를 Ctrl-Exclusive로 설정한다. 호스트가 동일 범위를 읽거나 쓰려 하면 트랩/차단부(154)가 이를 차단한다.

(e) 실행 및 결과 기록  
컨트롤러(140)는 입력을 읽어 암호화하고 출력에 기록한다. 결과 기록이 완료되면 상태는 ResultCommitting을 거쳐 Done으로 변경된다.

(f) 권한 복귀 및 통지  
Done 상태에서 공유 메모리 관리부(150)는 출력 범위를 Host-RW로 전환하고(정책에 따라 Host-RO도 가능), 입력 범위는 Host-RW로 복귀하거나 자동 삭제(Trim 허용)할 수 있다. 통지/동기화부(155)는 완료 이벤트를 호스트에 통지한다.

[시나리오 2: 데이터베이스 필터/집계 오프로드]
(a) 호스트는 타겟 영역에 테이블의 특정 컬럼 블록들을 저장한다(또는 이미 저장된 블록의 위치를 지정한다).  
(b) 오프로드 요청으로 “조건: colA between x and y, 집계: sum(colB)”를 전달한다.  
(c) 공유 메모리 관리부(150)는 실행 중 입력 블록은 Host-Deny/Ctrl-RO로, 결과 블록은 Host-Deny/Ctrl-RW로 설정한다.  
(d) 컨트롤러는 조건 필터 후 sum 결과를 결과 블록에 기록한다.  
(e) 완료 후 결과 블록을 Host-RW로 전환하여 호스트가 작은 결과만 읽는다.

[시나리오 3: 파이프라인(부분 결과 선공개) 오프로드]
(a) 호스트는 입력을 타겟 영역에 기록하고, 오프로드 요청 시 “부분 완료 시 구간별 Host-RO 허용” 옵션을 지정한다.  
(b) 컨트롤러는 출력 블록을 순차 생성하고, 블록 단위 done 비트를 업데이트한다.  
(c) 공유 메모리 관리부(150)는 done 비트가 1인 블록 구간에 한하여 Host-RO를 허용하고, 나머지는 Host-Deny를 유지한다.  
(d) 호스트는 완료된 일부 출력부터 읽어 후속 처리를 진행할 수 있다.

(15) 효과 및 변형 가능성
상기 실시예들에 따르면, 오프로드 처리에서 핵심이 되는 타겟 영역을 공유 메모리 관리부(150)가 처리 상태에 따라 동적으로 권한 전환함으로써, 호스트와 컨트롤러의 경쟁 접근을 방지하고 데이터 무결성을 확보할 수 있다. 또한 오프로드 처리 단계에서 불필요한 데이터 복사나 락(lock) 오버헤드를 줄이고, 메모리 장치 근처에서 연산을 수행하여 시스템 성능 및 전력 효율을 개선할 수 있다.

또한 본 실시예는 (i) 단일 호스트-단일 컨트롤러 구성뿐 아니라, (ii) 다중 호스트가 하나의 저장 노드를 공유하는 구성, (iii) 하나의 호스트가 다수의 컨트롤러를 병렬로 사용하는 구성, (iv) 적층형 패키지 또는 칩렛 구조 등으로 변형 가능하다. 타겟 영역의 정의도 LBA 범위, 네임스페이스, 파티션, 키 기반 윈도우 등으로 다양하게 구현될 수 있으며, 권한 전환 정책 또한 작업 종류, 보안 등급, QoS 조건 등에 따라 다르게 적용될 수 있다.

결국, 본 실시예들은 “메모리 장치”, “호스트의 오프로드 요청에 따라 메모리 장치에 접근하여 어플리케이션을 처리하는 컨트롤러”, 및 “호스트가 타겟 영역에 데이터를 저장함에 따라 타겟 영역의 접근 권한을 컨트롤러가 갖도록 설정하고, 어플리케이션의 처리 상태에 따라 타겟 영역의 접근 권한을 컨트롤러 또는 호스트가 갖도록 설정하는 공유 메모리 관리부”를 포함하는 데이터 처리 시스템을 구현함으로써, 청구항 1의 구성을 충실히 뒷받침한다.
</실시예>
