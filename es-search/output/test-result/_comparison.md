## 3. 핵심 발견

### ① NEW의 문체 고정 지시가 실제로 작동
- new 5건 전부 **'실시형태' 0회** (문체 블록 "'실시예'만 사용" 준수)
- yaml은 지시가 없어 **3건에서 '실시형태' 혼입** — 국내 표준형 문체 위반

### ② NEW가 면책 말미를 일관 생성
- new 5/5 면책 문장 포함(문체 블록 지시). yaml은 CMP·기판에서 **누락**

### ③ NEW가 앵커별 전개를 정확히 재현 (가장 중요)
- **동작-시간(데이터)**: new가 S단계 58회로 "먼저→이후→이어서" 시간순 전개 강하게 구현. yaml은 9회로 약함
- **실험-수치(기판·박막)**: new가 비교예 구조(8·13회) 생성 — 원본의 "비교예 대비 입증" 재현. yaml은 비교예 거의 없음(0·5)
- → **앵커 지시(블록 B/C)가 발명 본질에 맞는 전개를 유도**함이 입증됨

### ④ YAML의 "수치 풍부"는 환각 위험 (NEW의 의도적 자제)
- 기판·박막에서 yaml이 수치(28·51) 많이 생성. 그러나 new 프롬프트의 **"수치·실험데이터는 발명자 제공값만, LLM 임의생성 금지"** 지시 때문에 new는 수치를 자제(7·0)
- 즉 **yaml의 풍부한 수치는 LLM이 임의 생성한 환각값일 가능성** — 특허 실무에서 위험
- **운영 시 해결**: new 프롬프트에 `{example}`로 발명자 실제 수치를 주입하면, 환각 없이 풍부한 수치 확보 가능

## 4. 결론

| 항목 | 판정 |
|---|---|
| 문체 재현(실시용어·면책) | **NEW 우세** — 지시가 명시적으로 작동 |
| 앵커별 전개(동작·실험) | **NEW 우세** — S단계·비교예 구조 재현 |
| 수치 정확성 | **NEW가 안전** — yaml은 환각 위험, new는 자제 후 example 주입 권장 |
| 종합 | **NEW(spec-prompt-architect) 프롬프트가 스타일 재현·안전성에서 우수** |

> **한 줄 요약**: 새 프롬프트는 ① 문체를 일관 고정하고 ② 발명 본질(앵커)에 맞는 전개를 유도하며 ③ 수치 환각을 억제한다. yaml 원본은 스타일 지시가 없어 실시형태 혼입·면책 누락·앵커 미반영·수치 환각이 발생했다.

> **개선 메모**: 실험-수치 케이스는 `{example}`에 발명자 제공 실험데이터를 주입해야 "안전 + 풍부"를 동시 달성. id:8의 `{output_length}`는 700자→앵커별 실측치(6천~17천자)로 상향 권장.
