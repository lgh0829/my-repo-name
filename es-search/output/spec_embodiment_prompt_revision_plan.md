# Spec Embodiment 프롬프트 수정안

## 1. 수정 목적

`spec_embodiment_style_output_evaluation.md`에서 확인된 문제를 해결하기 위해 Spec Embodiment 프롬프트를 다음 원칙으로 개편한다.

1. 스타일 프리셋은 **기술 내용이 아니라 표현 방식만 변경**한다.
2. 입력에 없는 구성, 참조부호, 수치, 공식, 재질, 시험조건 및 시험결과를 생성하지 않는다.
3. 청구항 필수 구성은 직접적·단정적으로 뒷받침하고, 선택 구성과 변형예만 가능형으로 표현한다.
4. `Description`과 `Embodiment`의 역할을 분리하여 중복을 줄인다.
5. JP 번역 출원 스타일은 국내 작성 스타일과 분리한다.
6. 스타일 점수와 별도로 기술 내용 보존 및 기재요건 위험을 검사한다.

---

## 2. 우선순위별 수정사항

| 우선순위 | 수정 대상 | 핵심 내용 |
|---|---|---|
| P0 | 기술 내용 잠금 | 스타일 적용 전 구성·관계·부호·도면·실시예를 고정 |
| P0 | 수치·시험 가드 | 출처 없는 수치·공식·시험결과 생성 금지 |
| P0 | 장수길 프리셋 | 국내형과 JP 번역형 분리 |
| P1 | 섹션 역할 | Description은 공통 설명, Embodiment는 실시예 차이 중심 |
| P1 | 스타일 규칙 | 필수 구성/선택 구성에 따른 어미 규칙 구분 |
| P1 | 스타일 검사 | 금지 표현과 어미 비율을 섹션별 검사 |
| P2 | 평가 지표 | 기술 불변성·출처 없는 숫자·중복률 추가 |

## 3. 권장 생성 구조

### 3-1. 1단계: 기술 사실 계층 생성

스타일과 무관한 하나의 `technical_facts` 객체를 먼저 확정한다.

```json
{
  "claim_required_components": [],
  "optional_components": [],
  "external_objects": [],
  "component_relationships": [],
  "reference_numerals": {},
  "embodiments": [],
  "drawing_plan": [],
  "verified_numeric_facts": [],
  "verified_test_facts": []
}
```

이 객체는 스타일별 생성 과정에서 읽기 전용으로 취급한다.

### 3-2. 2단계: 스타일 렌더링

스타일 프롬프트가 변경할 수 있는 항목을 다음으로 제한한다.

- 문장 어미
- 문단 길이
- 도입 정형구
- 문단 전개 순서
- 구성 설명의 분절 정도
- 변형 가능성 표현 위치
- 허용된 연결어

스타일 프롬프트가 변경할 수 없는 항목은 다음과 같다.

- 구성요소와 구성 명칭
- 필수/선택 구성 구분
- 구성요소 관계
- 참조부호
- 실시예 수와 실시예별 차이
- 도면 번호와 도면 유형
- 수치, 공식, 재질 및 시험 데이터
- 발명의 효과에 관한 사실관계

## 4. `shared_rules` 수정안

### 4-1. 삭제 또는 교체할 기존 규칙

기존:

```text
대표 수치와 조건은 비한정 대표 예시로 쓰고, 바로 뒤에 일반화 문장을 병기한다.
```

교체:

```text
수치와 조건은 청구항, 구조화 입력 또는 검증된 분석 결과에 존재하는 경우에만 사용한다.
입력에 수치가 없으면 임의의 대표 수치, 범위, 공차 또는 판정값을 만들지 않는다.
검증된 수치를 사용할 때에는 필요한 경우 비한정 예시임을 밝히고 적용 조건을 함께 기재한다.
```

### 4-2. 추가할 공통 규칙

```text
[기술 내용 불변성]
style_preset은 표현 형식에만 적용한다. style_preset에 따라 구성요소, 구성 명칭,
참조부호, 필수/선택 구분, 구성관계, 실시예 수, 도면 계획 또는 기술적 효과를
추가·삭제·치환하지 않는다.

[출처 없는 사실 생성 금지]
청구항, 구조화 입력 또는 검증된 분석 결과에 없는 수치, 공차, 재질, 회로값,
판정식, 임계값, 시험 횟수, 측정 결과, 성공률 또는 비교 결과를 새로 만들지 않는다.

[시험 사실 표현]
verified_test_facts가 비어 있으면 시험을 수행한 것처럼 `설정하였다`, `측정하였다`,
`그 결과`, `확인되었다`, `감소하였다`라고 쓰지 않는다. 시험 방법의 예시가 필요한
경우에는 `검증할 수 있다` 수준으로만 기술하고 결과를 예측하지 않는다.

[청구항 필수 구성]
claim_required_components에 포함된 구성은 직접적·단정적으로 설명한다.
`포함할 수 있다`, `구성될 수 있다`와 같이 존재 여부를 선택적으로 보이게 하는
표현을 사용하지 않는다.

[선택 구성]
optional_components 또는 변형예에만 가능형 표현을 사용할 수 있다.

[부호 고정]
reference_numerals에 없는 참조부호를 새로 만들지 않는다. 동일 구성에는 모든
스타일에서 동일한 참조부호를 사용한다.

[도면 고정]
drawing_plan에 없는 도면 번호, 도면 유형 또는 도면상 배치관계를 새로 만들지 않는다.
```

## 5. 섹션별 프롬프트 수정안

### 5-1. Effects

#### 삭제할 규칙

```text
필요하면 일 비교예와 일 시험예를 넣어 효과의 측정 조건과 비교 기준을 남긴다.
시험예에는 반복 횟수, 측정 대상, 측정 위치, 편차 정의, 비교 조건 중 최소 3개를 포함한다.
```

위 규칙은 입력에 시험 데이터가 없을 때 모델이 반복 횟수와 결과를 임의 생성하게 만든다.

#### 교체 규칙

```text
효과는 청구항의 차별 구성과 구조화 입력에 기재된 효과를 1:1 인과관계로 연결한다.

verified_test_facts가 있는 경우에만 시험예 또는 비교예를 작성한다. 이 경우 반복 횟수,
측정 대상, 측정 위치, 편차 정의 및 비교 조건은 verified_test_facts의 값을 그대로 사용한다.

verified_test_facts가 없으면 시험 횟수, 측정값, 성능 개선율 또는 비교 결과를 만들지 않는다.
필요하면 `해당 효과는 동일 조건에서 측정값의 편차를 비교하여 검증할 수 있다`와 같이
검증 방법만 설명하고 결과는 단정하지 않는다.

입력에 없는 효과를 새로 만들지 않는다. 통신 지연, 정확도, 전력, 수명, 성공률 등은
검증된 근거가 있을 때만 기재한다.
```

---

### 5-2. Description

#### 기존 규칙 수정

기존:

```text
도면부호별 공통 정의를 먼저 쓰고, 그 다음 연결관계, 신호/유체/힘/열/데이터 경로,
대표 수치/조건, 판독/측정/시험 기준, 변형 가능 범위 순서로 쓴다.
```

수정:

```text
도면부호별 공통 정의를 먼저 쓰고, 그 다음 구성요소의 연결관계와 청구항에 직접
기재된 신호·유체·힘·열·데이터 경로를 설명한다.

Description에서는 실시예별 조작 순서, 제조 조건, 시험 조건 및 비교 결과를 작성하지 않는다.
수치와 판정 기준은 verified_numeric_facts에 존재하는 경우에만 간단히 정의하고,
구체 적용은 Embodiment에서 설명한다.

공통 정의는 실제 발명 이해에 필요한 용어만 포함한다. 입력에 없는 축, 기준면, 노드,
좌표계 또는 수식을 설명 분량을 늘리기 위해 만들지 않는다.

Description의 마지막에는 변형 가능 범위를 1개 문단으로 작성하되, 필수 구성을
선택 구성처럼 약화시키지 않는다.
```

#### 중복 방지 규칙 추가

```text
Description에서 설명한 공통 구성 목록과 기본 데이터 흐름을 Embodiment에 그대로
복사하지 않는다. Embodiment에서는 해당 실시예에서 새로 특정되는 관계와 동작만 설명한다.
```

---

### 5-3. Embodiment

#### 삭제할 규칙

```text
제1 실시예에는 실제 제작자가 도면과 부품표를 만들 수 있도록 치수기호, 상대 치수식,
공차, 재질/공정, 회로값 또는 판정값 중 해당 분야의 최소 3종을 넣는다.
```

#### 교체 규칙

```text
제1 실시예는 claim_required_components 전체를 하나의 기준형으로 직접 뒷받침한다.

치수, 상대 치수식, 공차, 재질, 공정, 회로값 또는 판정값은 verified_numeric_facts에
존재하는 항목만 사용한다. 입력에 해당 정보가 없으면 최소 개수를 맞추기 위해 만들지 않는다.

각 실시예는 다음 순서로 작성한다.
1. 해당 실시예의 목적과 다른 실시예와의 차이
2. 추가되거나 구체화되는 구성관계
3. 실제 입력으로 확인되는 동작 또는 제조·처리 순서
4. 입력으로 확인되는 조건
5. 입력으로 확인되는 변형예

시험예 또는 비교예는 verified_test_facts가 있는 경우에만 작성한다.

embodiments에 정의되지 않은 제2·제3 실시예를 분량 확보 목적으로 만들지 않는다.
```

#### 도면 부합 규칙 수정

```text
drawing_plan에 포함된 도면만 인용한다. 도면 파일 또는 확정된 drawing_plan이 없으면
도면에 특정 구성이나 내부 배치가 도시된다고 단정하지 않는다.
```

## 6. 스타일 프리셋 수정안

### 6-1. 공통 — 표준 국문 출원형

```json
{
  "label": "표준 국문 출원형",
  "ending_form": "필수 구성은 단정형, 선택 구성과 변형예는 가능형",
  "tail_pattern": "전체 Embodiment 말미에 변형 가능성 문장 1회",
  "notes": "특정 사무소 관용구를 넣지 않는다. 입력 없는 수치·시험예를 생성하지 않는다."
}
```

추가 규칙:

```text
표준형은 가장 보수적인 기본값이다. 스타일 식별을 위한 특이 표현을 만들지 않는다.
Description과 Embodiment 합산 분량을 늘리기 위해 동일 구성을 반복하지 않는다.
```

---

### 6-2. 김성남 — 관용선언 삭제형

```json
{
  "label": "관용선언 삭제형",
  "ending_form": "필수 구성과 실제 동작은 확정형, 선택 구성만 가능형",
  "tail_pattern": "별도 관용적 말미 없음",
  "notes": "이를 위해, 이를 통해, 본 발명은, 이에 따라 등의 선언형 관용구를 사용하지 않는다. 수치는 검증된 입력에 있는 경우에만 구성·조건과 직접 연결한다."
}
```

추가 규칙:

```text
`그리고` 또는 `구체적으로`를 매 문단의 반복 연결어로 사용하지 않는다.
필수 구성 설명 문장의 80% 이상을 `한다`, `된다`, `이다`, `포함한다` 계열로 끝낸다.
`할 수 있다`는 optional_components와 변형예에서만 허용한다.
```

---

### 6-3. 김현진 — 하향식 분절 서술형

```json
{
  "label": "하향식 분절 서술형",
  "ending_form": "필수 구성은 단정형, 세부 구현 선택지는 가능형",
  "tail_pattern": "상위 역할 선언 후 해당 역할을 수행하는 세부 구성 열거",
  "notes": "이를 위해는 상위 역할이 바뀌는 문단에서만 1회 사용한다. 이를 통해는 Effects 외 사용 금지. 바람직하게는 전 섹션에서 금지."
}
```

추가 규칙:

```text
`이하에서는 ... 설명한다`는 대항목별 1회로 제한한다.
같은 구성에 대해 구조, 연결관계, 데이터 경로를 별도 문단으로 반복하지 않는다.
역할 선언 뒤에는 입력에 존재하는 세부 구성만 열거한다.
```

---

### 6-4. 리앤목 — 균등 보호형

```json
{
  "label": "균등 보호형",
  "ending_form": "필수 구성은 단정형, 구현 방식·선택 구성·변형예는 가능형",
  "tail_pattern": "전체 Embodiment의 마지막에만 실시 범위 일반화 1문장과 청구범위 내 변형 가능성 1문장",
  "notes": "가능형은 선택 가능성을 남길 필요가 있는 부분에만 사용한다. 균등 범위 문구를 각 실시예마다 반복하지 않는다."
}
```

추가 규칙:

```text
제1 실시예의 claim_required_components는 `포함한다`, `연결된다`, `수행한다`와 같이
단정형으로 직접 뒷받침한다.

`다양한 변경, 균등물 내지 대체물`과 같은 해석 보일러플레이트는 문서 도입부와
말미 중 한 곳에서만 사용한다.

실제 청구범위 입력이 없으면 `첨부된 특허청구범위`라고 쓰지 않는다.
```

---

### 6-5. 장수길 — 국내 출원형

현재 `장수길` 프리셋은 JP 번역체와 국내 작성 스타일을 혼합한다. 기본 프리셋을 국내형으로 바꾼다.

```json
{
  "label": "간결 확정형",
  "ending_form": "필수 구성과 동작은 확정형",
  "tail_pattern": "각 실시예 말미에 해당 실시예의 직접 효과 1문장",
  "notes": "실시예를 사용한다. JP 번역체 정형구, 실시 형태, 당해, 에 수반하여, 구비하고 있다를 사용하지 않는다. 문단을 간결하게 유지한다."
}
```

권장 도입문:

```text
이하, 첨부 도면을 참조하여 본 발명의 실시예를 상세히 설명한다.
```

추가 규칙:

```text
`할 수 있다`는 optional_components와 변형예에서만 허용한다.
입력에 없는 수치나 시험 결과를 확정형으로 단정하지 않는다.
`실시 형태`, `구비하고 있다`, `이하 ...에 대해 설명한다`를 사용하지 않는다.
```

#### 별도 선택 프리셋: 장수길 JP 번역형

JP 원문 번역 출원에서만 명시적으로 선택할 수 있도록 별도 분리한다.

```json
{
  "label": "장수길 JP 번역형",
  "applicability": "JP 원문 번역 출원인 경우에만 명시적으로 선택",
  "ending_form": "확정형",
  "terminology": "실시 형태 허용",
  "notes": "순수 국내 출원 기본값으로 사용하지 않는다."
}
```

## 7. 스타일 검사기 수정안

### 7-1. 공통 검사

| 검사 항목 | 실패 조건 |
|---|---|
| 구성 불변성 | 스타일 간 구성요소 집합이 다름 |
| 부호 불변성 | 동일 구성의 참조부호가 스타일별로 다름 |
| 관계 불변성 | 필수 연결관계가 추가·삭제·치환됨 |
| 실시예 불변성 | 실시예 수 또는 실시예별 차이가 달라짐 |
| 수치 출처 | 출력 숫자가 `verified_numeric_facts`에 없음 |
| 시험 출처 | 시험 결과 문구가 있으나 `verified_test_facts`가 비어 있음 |
| 도면 출처 | `drawing_plan`에 없는 도면 번호 또는 관계를 인용함 |
| 필수 구성 약화 | 필수 구성에 `포함할 수 있다` 사용 |

### 7-2. 스타일별 검사

| 스타일 | 필수 검사 |
|---|---|
| 공통 | 특정 사무소 전용 정형구 없음 |
| 김성남 | 필수 구성의 확정형 비율 80% 이상, 선언형 관용구 0회 |
| 김현진 | `이를 통해` Description/Embodiment 0회, `바람직하게는` 전체 0회 |
| 리앤목 | 필수 구성은 단정형, 균등 범위 문구 전체 1세트 이하 |
| 장수길 국내형 | `실시 형태` 0회, JP 번역체 표현 0회, 필수 구성 확정형 |
| 장수길 JP형 | JP 번역 출원 선택 플래그 확인 |

### 7-3. 점수 분리

하나의 `스타일 준수도`로 합산하지 않고 다음 점수를 별도로 표시한다.

```json
{
  "style_fidelity_score": 0,
  "technical_invariance_score": 0,
  "source_grounding_score": 0,
  "section_role_score": 0,
  "kipo_risk": "low | medium | high"
}
```

통과 기준:

- `style_fidelity_score >= 85`
- `technical_invariance_score = 100`
- `source_grounding_score = 100`
- `section_role_score >= 80`
- `kipo_risk != high`

기술 불변성과 출처 근거는 문체 점수로 상쇄할 수 없는 필수 조건으로 둔다.

## 8. 회귀 테스트 계획

### 8-1. 동일 입력 스타일 A/B 테스트

하나의 `technical_facts`를 5개 스타일에 공통 입력한다.

검증 항목:

1. 모든 출력의 구성요소와 참조부호가 동일한지 확인
2. 모든 출력의 실시예 수와 도면 계획이 동일한지 확인
3. 출력에 새 숫자와 공식이 추가되지 않았는지 확인
4. 스타일별 어미와 정형구만 달라졌는지 확인
5. Description과 Embodiment의 역할이 중복되지 않는지 확인

### 8-2. 수치 입력 없음 테스트

입력에 수치와 시험 데이터가 없을 때 다음이 0건이어야 한다.

- 단위가 붙은 새 숫자
- 판정 임계값과 공식
- 시험 반복 횟수
- 시험 결과 및 개선율
- 성공률과 지연시간

### 8-3. 수치 입력 있음 테스트

입력에 검증된 수치를 제공했을 때 다음을 확인한다.

- 입력 수치가 변경되지 않음
- 단위가 변경되지 않음
- 입력에 없는 결과를 추론하지 않음
- 스타일별로 동일한 수치가 유지됨

### 8-4. 장수길 분기 테스트

- 국내형: `실시예` 사용, JP 번역체 0회
- JP 번역형: 명시적 선택 시에만 `실시 형태` 허용

## 9. 적용 순서

1. `shared_rules`에 기술 불변성 및 출처 가드를 추가한다.
2. Effects의 시험예 강제 규칙을 제거한다.
3. Embodiment의 최소 3종 수치·공차 강제 규칙을 제거한다.
4. 기술 사실 계층과 스타일 렌더링 계층을 분리한다.
5. 장수길 국내형/JP 번역형을 분리한다.
6. 김성남·김현진·리앤목 스타일 규칙을 수정한다.
7. 불변성 및 출처 검사를 통과 게이트로 추가한다.
8. 동일 입력으로 5종 회귀 테스트를 다시 수행한다.

## 10. 최종 권장 프롬프트 블록

다음 블록은 모든 Spec Embodiment 스타일 프롬프트의 마지막에 공통으로 삽입한다.

```text
[최종 기술 내용 보존 규칙]

1. 스타일은 문장 어미, 문단 순서, 도입·말미 표현 및 설명의 분절 정도에만 적용한다.
2. 제공된 구성요소, 구성 명칭, 필수/선택 구분, 구성관계, 참조부호, 실시예 수 및 도면 계획을 변경하지 않는다.
3. 입력 또는 검증된 분석 결과에 없는 구성, 부호, 수치, 범위, 공차, 재질, 회로값, 공식, 임계값, 시험조건 및 시험결과를 만들지 않는다.
4. 실제 시험 데이터가 없으면 시험을 수행하거나 결과를 확인한 것처럼 쓰지 않는다.
5. 청구항 필수 구성은 단정형으로 직접 뒷받침하고, 선택 구성과 변형예만 가능형으로 표현한다.
6. Description의 공통 설명을 Embodiment에 반복하지 않는다.
7. drawing_plan에 없는 도면 번호와 도면상 관계를 만들지 않는다.
8. 위 규칙과 스타일 규칙이 충돌하면 기술 내용 보존 규칙을 우선한다.
```
