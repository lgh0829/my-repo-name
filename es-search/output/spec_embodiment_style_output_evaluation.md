# Spec Embodiment 스타일 프롬프트 산출물 평가

## 평가 범위

- 평가 대상: 동일 발명인 `차량용 배터리 진단 장치`를 5개 스타일 프리셋으로 생성한 명세서
- 중점 섹션: `발명을 실시하기 위한 구체적인 내용`, `실시예`
- 스타일 기준: 저장소의 `output/commercial_prompt_2_updated.json`에 정의된 5개 프로파일 및 공통 작성 규칙
- 기재요건 기준: 특허법 제42조상 명확성, 실시가능성, 뒷받침 및 도면 부합 관점
- 제한: 산출물에 실제 청구범위와 도면 원본이 없으므로 뒷받침 및 도면 부합은 최종 합격 판정이 불가능하다.

> Notion 원문은 연결 인증 문제로 직접 열람하지 못했다. 대신 같은 5개 스타일명과 규칙을 포함하는 저장소의 프롬프트 v3를 기준으로 평가했다.

법적 판단 기준은 특허청의 「특허·실용신안 심사기준」 중 발명의 설명 기재요건과 청구범위 기재요건을 참고했다.

- [특허·실용신안 심사기준 — 제3장 발명의 설명](https://www.kipo.go.kr/club/file.do?attachmentId=12417)
- [특허·실용신안 심사기준](https://www.kipo.go.kr/club/file.do?attachmentId=11473)

## 핵심 결론

1. **스타일 차이는 눈에 보인다.** 리앤목의 가능형·균등 범위 문구, 장수길의 확정형·`실시 형태`, 김현진의 `이를 위해` 하향식 분절은 뚜렷하다.
2. **그러나 스타일만 바뀐 것이 아니다.** 스타일별로 하위 구성, 참조부호, 판정식, 임계값, 배선 치수, 시험 조건이 달라졌다. 동일한 기술 내용을 유지한 문체 A/B 테스트로 보기 어렵다.
3. **가장 큰 위험은 임의 수치와 시험 결과 생성이다.** 실제 입력 근거가 확인되지 않은 상태에서 반복 횟수, 허용오차, 정상·열화 판정값 및 비교 결과를 만들었다.
4. **자동 스타일 준수도는 신뢰하기 어렵다.** 예를 들어 김성남은 100점이지만 의도한 `확정형 + 수치 연결`과 달리 본문에 가능형이 매우 많다. 김현진은 금지된 `이를 통해`와 `바람직하게는`을 반복한다.
5. **현재 상태에서 제품 적용은 비권고다.** 스타일 프롬프트가 기술 사실을 생성하지 못하도록 역할을 축소한 뒤 다시 평가해야 한다.

## 개별 평가

### 1. 공통 — 표준 국문 출원형

**스타일 효과**

- 가능형 중심이고 특정 사무소 표현에 치우치지 않는다.
- 구성, 연결관계, 변형예를 고르게 설명하여 기준선 역할은 한다.

**문제점**

- 5개 중 Description과 Embodiment 합산 분량이 두 번째로 길고, 같은 구성과 동작을 반복한다.
- `100 kΩ`, `0.5 mΩ 내지 5 mΩ`, `0 V 내지 16 V`, `20회 반복 측정` 등 입력 근거가 확인되지 않은 수치가 많다.
- “불일치율이 감소하는 경향을 확인할 수 있다”는 표현은 실제 시험을 수행한 것처럼 읽힐 수 있다.
- 표준형인데도 효과·시험·비교예를 과도하게 채워 가장 보수적인 기본값이 되지 못했다.

**판정**: 기준 문체로는 사용할 수 있으나, 임의 수치와 시험예를 제거하고 반복을 줄여야 한다.

### 2. 김성남 — 관용선언 삭제형

**스타일 효과**

- Description과 Embodiment에서 `이를 위해`, `이를 통해`, `바람직하게는`을 제거한 점은 의도와 부합한다.
- 수치·조건을 구성 설명에 바로 연결하는 경향이 나타난다.

**문제점**

- 프리셋의 어미 규칙은 `확정형 + 수치 연결`인데 `수 있다`가 117회이고 `한다`는 2회뿐이다. 100점이라는 자동 준수도와 실제 결과가 충돌한다.
- 관용구 대신 `그리고`, `구체적으로`가 반복되어 문체가 기계적이다.
- `0.3 V`, `15%`, `300회`, `95회 이상 정상 수신` 등 기술적 근거가 없는 값과 결과가 다수 생성됐다.
- 관용선언 삭제가 간결성으로 이어지지 않고, 오히려 수치형 문장을 양산했다.

**판정**: 현재 프리셋은 “관용구 삭제”만 작동하고 “확정형”은 작동하지 않는다. 스타일 프리셋과 검사기를 함께 고쳐야 한다.

### 3. 김현진 — 하향식 분절 서술형

**스타일 효과**

- `이하에서는 ... 설명한다`와 `이를 위해`를 이용해 상위 역할에서 세부 구성으로 내려가는 구조가 가장 뚜렷하다.
- 구조, 기준 좌표, 연결관계, 데이터 경로, 대표 조건, 변형예가 분절되어 탐색성은 좋다.

**문제점**

- 프롬프트가 Effects 외 사용을 금지한 `이를 통해`가 Description·Embodiment에 12회 등장한다.
- 공통 금지 표현인 `바람직하게는`이 6회 등장한다.
- `L:W:H=80:50:20`, `d1≤d2`, 센서 오차율, 저항값 등 입력에 없던 설계조건을 만들어냈다.
- 같은 역할 선언이 반복되어 5개 중 가장 길다. 하향식 구조가 명확성보다 템플릿 노출로 이어졌다.
- “설정하였다”, “그 결과”처럼 실제 시험 수행을 암시하는 문구가 있다.

**판정**: 스타일은 식별되지만 명시적 금지 규칙을 위반한다. 단순 후처리가 아니라 생성 프롬프트의 규칙 우선순위를 정리해야 한다.

### 4. 리앤목 — 균등 보호형

**스타일 효과**

- `수 있다` 가능형이 일관되고, 구성의 선택 가능성과 변형 범위를 자연스럽게 남긴다.
- 도입부의 해석 규칙과 말미의 변형·균등 범위 문구가 프리셋 의도와 가장 잘 맞는다.
- 다른 스타일보다 Description과 Embodiment의 문언 중복이 상대적으로 적다.

**문제점**

- “다양한 변경, 균등물 내지 대체물” 및 최종 보호범위 문장은 반복 사용 시 보일러플레이트가 된다.
- `η` 판정식, 100/150 임계값, 센서 범위, 패킷 길이 등 기술적 내용이 새로 생성됐다.
- 가능형이 지나치게 많아 필수 구성까지 선택 구성처럼 약화될 수 있다. 청구항 필수 구성은 단정형으로 직접 뒷받침해야 한다.
- 실제 청구범위가 없으므로 “첨부된 특허청구범위의 기술적 사상”이라는 말미의 적합성을 확인할 수 없다.

**판정**: 다섯 개 중 문체 재현은 가장 안정적이다. 다만 균등 범위 문구는 문서 말미 1회로 제한하고, 기술 수치 생성은 금지해야 한다.

### 5. 장수길 — 간결 확정형

**스타일 효과**

- `수 있다`가 0회이고 `한다/된다/이다`가 중심이어서 확정형 어미는 매우 잘 재현됐다.
- `실시 형태`가 일관되고 문단 전개가 다른 스타일보다 직접적이다.
- 도입부와 실시 형태별 구조가 선명해 스타일 식별성은 높다.

**문제점**

- `실시 형태`, `구비하고 있다`, `이하 ... 대해 설명한다`는 JP 번역 출원 문체다. 현재 분석 하네스는 JP 번역 출원을 국내 작성 스타일과 분리하도록 규정하며, 기존 장수길 리포트도 순수 국내 출원 프롬프트에서는 `실시예`로 바꾸도록 정리돼 있다. 이 프리셋은 구 기준을 사용한다.
- 확정형 때문에 근거 없는 수치가 사실처럼 단정된다. 예: 배선 길이, 위치 공차, `12.6 V`, `30 A`, 편차값 및 전송시간.
- “경향이 확인된다”, “정상으로 판정한다”는 실제 실험 데이터가 없는 경우 위험하다.
- 기준 프로파일(400), 기준 구간(410)처럼 다른 스타일에는 없는 별도 참조부호와 기술 구조가 추가됐다.
- 간결형이라고 보기에는 Description+Embodiment가 11,276자로 여전히 길고, 시험예·비교예를 6개 포함한다.

**판정**: 현재 프리셋 자체에는 잘 맞지만, 목표로 삼아야 할 국내 작성형 장수길 스타일에는 맞지 않는다. JP 번역형과 국내형을 분리해야 한다.

## 공통 기술·법적 위험

### 1. 스타일이 기술 내용을 변경함

동일 발명인데 제어부 하위 구성이 서로 다르다.

| 스타일 | 제어부 하위 구성 예 |
|---|---|
| 공통 | 측정값 수신부(131), 비교 처리부(132), 열화 판정부(133), 경고 신호 생성부(134), 기준 프로파일 저장부(135) |
| 김성남 | 입력 인터페이스(131), 기준 프로파일 저장부(132), 비교부(133), 판정부(134), 경고 신호 생성부(135) |
| 김현진·리앤목 | 기준 프로파일 저장부(131), 판정부(132), 경고 신호 생성부(133) |
| 장수길 | 측정값 수신부(131), 기준 프로파일 저장부(132), 비교부(133), 판정부(134), 경고 신호 생성부(135), 기준 프로파일(400), 기준 구간(410) |

스타일 프롬프트는 문장 전개와 어미만 바꿔야 한다. 구성요소와 부호표는 스타일 적용 전에 고정해야 한다.

### 2. 근거 없는 시험예와 수치

공통 규칙이 제1 실시예에 치수·공차·회로값·판정값 중 최소 3종을 강제하고, Effects에는 반복 횟수·측정 위치·편차 정의 등을 요구한다. 입력에 실제 값이 없으면 모델은 이를 채우기 위해 수치를 발명한다.

다음은 스타일이 아니라 기술 사실이므로 입력 근거가 없으면 생성하면 안 된다.

- 센서 입력 범위, 분해능, 허용오차
- 배선 길이, 위치 공차, 저항값
- 열화 판정식과 정상·열화 임계값
- 반복 시험 횟수, 측정 결과, 수신 성공률
- 비교예보다 성능이 우수하다는 결과

### 3. Description과 Embodiment 역할 중복

모든 문서가 `발명을 실시하기 위한 구체적인 내용`에서 이미 구성·동작·변형예를 상세히 설명한 뒤, 별도 `실시예`에서 같은 내용을 다시 전개한다. 이중 생성 구조는 분량을 늘리고 부호·수치 불일치 가능성을 높인다.

### 4. 청구항·도면 부재

- 청구범위가 없으므로 필수 구성 뒷받침 여부를 최종 판정할 수 없다.
- 실제 도면이 없으므로 `도 1`, `도 2`, `도 3`의 표시 부호와 배치가 본문에 부합하는지 확인할 수 없다.
- 따라서 자동 점수의 `통과 O`는 KIPO 기재요건 합격을 의미하지 않는다.

## 프롬프트 수정 우선순위

### P0 — 기술 사실 고정

1. 스타일 적용 전에 `구성요소`, `참조부호`, `관계`, `실시예 수`, `도면 계획`을 하나의 불변 구조로 확정한다.
2. 스타일 프롬프트는 어미, 문단 순서, 도입·말미 정형구만 바꾸도록 제한한다.
3. 스타일 프롬프트에서 새 구성·부호·수치·공식·시험 결과를 생성하지 못하게 한다.

### P0 — 수치와 시험예 가드

```text
입력 또는 검증된 분석 결과에 없는 수치, 공차, 재질, 회로값, 판정식,
시험 횟수, 측정 결과 및 비교 결과를 새로 만들지 않는다.
실제 시험 데이터가 없으면 시험을 수행한 것처럼 쓰지 않는다.
필요 시 수치 없는 구현 원리 또는 검증 항목만 설명한다.
```

### P1 — 섹션 역할 정리

- `발명을 실시하기 위한 구체적인 내용`을 공통 정의·전체 구성·용어 설명으로 제한한다.
- `실시예`는 각 실시예의 차이, 구체 동작 및 입력으로 확인된 조건만 기술한다.
- 또는 두 섹션을 하나로 통합해 중복 생성을 원천 차단한다.

### P1 — 스타일별 수정

- **공통**: 가장 보수적인 기본값으로 축소하고 시험예 자동 생성을 제거한다.
- **김성남**: `확정형`을 실제 검사 항목으로 추가하고 가능형 비율 상한을 둔다.
- **김현진**: `이를 통해`, `바람직하게는` 금지를 생성 후 검사에도 반영한다.
- **리앤목**: 필수 구성은 단정형, 선택·변형만 가능형으로 분리한다. 균등 범위 문구는 전체 말미 1회만 사용한다.
- **장수길**: `JP 번역형`과 `국내 출원형`을 분리한다. 국내형은 `실시 형태`를 `실시예`로 바꾸고 번역체 정형구를 제거한다.

### P2 — 평가 지표 교체

현재 스타일 준수도는 표현 존재 여부 중심으로 보인다. 다음을 함께 검사해야 한다.

- 필수 어미 비율과 금지 어미 비율
- 금지 문구의 섹션별 위치
- 스타일 간 구성요소·참조부호·수치 불변성
- 입력에 없는 숫자 및 시험 결과 생성 여부
- Description/Embodiment 의미 중복률
- 청구항 필수 구성의 직접 뒷받침 여부

### 실제 제품 적용 적합성

현재 상태에서는 **5개 모두 적용 보류**가 적절하다. 리앤목이 가장 안정적이지만, 모든 프리셋이 기술적 수치와 시험 결과를 임의 생성하는 공통 위험을 공유한다.
