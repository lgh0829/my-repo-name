## 1. 전체 아키텍처

```
┌─────────────────────────────────────────────────────────┐
│        Embodiment 생성 에이전트 (Parent)                 │
│                                                          │
│  ① 청구항·도면 컨텍스트 분석                              │
│  ② [서브에이전트 호출] → 관용문단 생성 에이전트             │
│  ③ 본문(실시예 설명) 생성                                 │
│  ④ 관용문단을 서두/용어정의/말미 위치에 단락으로 삽입        │
│  ⑤ 중간부 관용문단을 본문 단락 뒤 또는 독립 단락으로 삽입   │
└─────────────────────────────────────────────────────────┘
                        │ InventionContext (JSON)
                        ▼
┌─────────────────────────────────────────────────────────┐
│        관용문단 생성 서브에이전트                           │
│                                                          │
│  입력: InventionContext                                   │
│  처리: 조건 판단 → 법적 근거별 관용문단 선택 → 조립          │
│  출력: BoilerplateOutput (위치별 단락 세트)                │
└─────────────────────────────────────────────────────────┘
```

## 2. 법적 요건 ↔ 관용문단 매핑

심사기준에서 도출한 기재불비 방어 목적과 실제 관용문단의 대응.

| 기재불비 요건 | 관용문단 그룹 | 법적 조항 |
|---|---|---|
| 실시예가 청구범위를 한정하지 않음 선언 | 서두: 실시예 한정 부인 (S5/S6) | 42조 4항 1호 (뒷받침) |
| 도면과 연계한 실시 가능 기재 명시 | 서두: 도면 참조 (S1+T1 복합) | 42조 3항 1호 (실시 가능) |
| 서수 용어가 순위·한정 의미 아님 | 용어정의: 제1·제2 한정 아님 (T4) | 42조 4항 2호 (명확성) |
| 단수 표현의 복수 포함 해석 허용 | 용어정의: 단복수·포함하다 복합 (T28/T6+T7) | 42조 4항 2호 (명확성) |
| 개방형 청구항 지원, 추가 구성요소 허용 | 용어정의: 포함하다 정의 (T28/T7) | 42조 4항 1호 (뒷받침) |
| 도면이 실제 치수와 다름 면책 | 용어정의: 도면 과장·축소 (T28/T3) | 42조 3항 1호 (실시 가능) |
| 중복 설명 생략 정당화 | 용어정의: 도면부호 동일 (T1) | 42조 3항 1호 (실시 가능) |
| 방법 발명 단계 순서 고정 방지 | 용어정의: 단계 순서 비한정 (T8) | 42조 4항 1호 (뒷받침) |
| 3D 구조에서 상/하 방향 기준 명확화 | 용어정의: 방향 기준 정의 (T10) | 42조 4항 2호 (명확성) |
| 상위개념 청구항 범위 지지 | ~~중간: B1~~ → **기술 단락 작성 권장사항** §11 참조 | 42조 4항 1호 (뒷받침) |
| 예시 열거의 한정력 차단 | ~~중간: B2~~ → **기술 단락 작성 권장사항** §11 참조 | 42조 4항 1호 (뒷받침) |
| 실시예 전체가 발명 범위 아님 | 말미: 실시예는 예시적 (E3/E5) | 42조 4항 1호 (뒷받침) |
| 권리범위 기준은 설명이 아닌 청구항 | 말미: 청구범위 기준 (E1/E5) | 42조 4항 1호 (뒷받침) |
| 통상 기술자의 응용 범위 확보 | 말미: 변형 가능성 (E2/E5) | 42조 3항 1호 (실시 가능) |

## 3. 입력 스키마 (InventionContext)

서브에이전트가 수신하는 JSON 입력.

```json
{
  "invention_type": "device | method | manufacturing_method | composition",
  "has_drawings": true,
  "has_numbered_elements": true,
  "has_ordinal_terms": true,
  "has_step_sequence": false,
  "has_3d_stacking": true,
  "claim_scope_type": "open | closed",
  "has_multiple_embodiments": true,
  "technical_field": "반도체 | 전기전자 | 기계 | 화학 | 바이오 | 소프트웨어 | 기타",
  "style_preset": "cluster_A | cluster_B | cluster_D | minimal",
  "positions_requested": ["head", "term_def", "tail"],
  "custom_figure_count": 5,
  "custom_element_names": ["제1 구성요소", "제2 구성요소"]
}
```

### 파라미터 설명

| 파라미터 | 설명 | 관용문단 선택 영향 |
|---|---|---|
| `invention_type` | 발명 카테고리 | `method` → T8(단계 순서 비한정 단락) 추가 |
| `has_drawings` | 도면 첨부 여부 | `false` → 도면 관련 관용문단 전체 제외 |
| `has_numbered_elements` | 도면부호(숫자) 사용 | `true` → T1(도면부호 동일 단락) 추가 |
| `has_ordinal_terms` | 제1, 제2 등 사용 | `true` → T4(서수 용어 정의 단락) 추가 |
| `has_step_sequence` | 단계 순서 기재 | `true` → T8(단계 순서 비한정 단락) 추가 |
| `has_3d_stacking` | 3D 적층 구조 설명 | `true` → T10(상/하 방향 기준 단락) 추가 |
| `claim_scope_type` | 청구항 개방/폐쇄 | `open` → T28(포함하다 정의 포함 단락) 필수. B1·B2는 관용문단이 아닌 기술 단락 작성 규칙(§11)으로 처리 |
| `style_preset` | 문체 스타일 | 클러스터 A(풀블록) / B(간결) / D(삼성형) / minimal |

## 7. 에이전트 프롬프트 구조

### 시스템 프롬프트 핵심 섹션

```
역할: 특허 명세서 '발명을 실시하기 위한 구체적인 내용' 섹션의 기재불비 방어
관용문단을 생성하는 전문 에이전트.

출력 단위: 관용문단(단락 전체). 단일 문장이 아닌 bracket 번호 1개에
해당하는 단락 텍스트 전체를 반환한다.

## 법적 근거
- 특허법 제42조 제3항 제1호 (실시 가능 요건): 통상 기술자가 쉽게 실시할 수 있도록
  명확하고 상세하게 기재 → 도면 관련 면책, 변형 가능성 선언
- 특허법 제42조 제4항 제1호 (뒷받침 요건): 청구항이 발명의 설명에 의해 뒷받침
  → 실시예 한정 부인, 개방형 기재 지원, 청구범위 기준 선언
- 특허법 제42조 제4항 제2호 (명확성 요건): 발명이 명확하고 간결하게 기재
  → 용어 정의, 서수 비한정, 단복수 포함, 방향 기준

## 관용문단 선택 원칙
1. InventionContext를 분석하여 필요한 관용문단만 선택 (과잉 삽입 금지)
2. 발명 유형에 맞는 관용문단만 포함 (방법 발명에만 T8 단락)
3. 도면이 없으면 도면 관련 관용문단 전체 제외
4. style_preset에 따라 단락 배치 순서와 개수를 결정
5. 말미 E1(또는 E5_leeandmok)은 모든 경우에 포함 (가장 보편적, 20.5%)
6. 복합문단(T28, S1_T1, E5_leeandmok)은 구성 요소를 분리하지 않고 단락 전체 반환
7. B1·B2 패턴("더 포함할 수 있다", "이에 한정되지 않는다")은 관용문단이 아닌 기술 단락 작성 규칙 — 부모 에이전트의 기술 단락 생성 시 §11 권장사항 적용

## 금지사항
- 발명 내용에 관한 실질적 판단 금지 (예: 청구항 해석, 선행기술 평가)
- 라이브러리 외 임의 관용문단 창작 금지
- 단락을 개별 문장으로 분리하여 반환 금지
- 심사기준에 근거 없는 관용문단 창작 금지
```

## 10. 핵심 설계 결정

### Q1: 관용문단을 LLM이 생성해야 하나 vs 라이브러리에서 선택해야 하나?
**결정: 라이브러리 선택 방식** — 관용문단은 법적 효력을 가진 상투 단락이므로 창작이 불필요하고 오히려 위험. LLM은 발명 유형과 스타일 프리셋에 맞는 단락 선택 조건 판단에만 개입.

### Q2: 복합문단(T28, S1_T1, E5_leeandmok)을 분리해야 하나?
**결정: 분리하지 않고 단락 전체 보존** — 복합문단은 실제 명세서에서 하나의 bracket 번호([0028], [0030], [0097]) 아래 등장한다. 문장 단위로 쪼개면 단락 경계가 사라져 단락 구분 DB 연동 후 매칭이 불가능해진다. 라이브러리에 복합문단 ID(T28, S1_T1)를 별도로 등록하고 구성 요소는 `composite_of` 필드로만 기록.

### Q3: 기재불비 방어 커버리지를 어떻게 보장하나?
**결정: legal_coverage 필드로 검증** — 출력에 포함된 paragraph_ids가 42조 3항/4항을 모두 커버하는지 자동 체크. 미커버 시 E1 강제 포함.

### Q4: style_preset 기본값은?
**결정: `cluster_A`** — 3,192건 분석 결과 리앤목 등 대형 사무소가 5~6개 관용문단으로 기재불비 방어 커버리지가 가장 높음. 단락 배치 순서 실례: 1020220060454A [0026]~[0030], [0097].

### Q5: DB 단락 구분 추가 전까지 중간 대응은?
**결정: 앵커 문구 탐지 유지** — 현재 DB는 단락 구분 없이 Embodiment 전문을 저장. 정규표현식으로 각 관용문단의 앵커 문구(첫 문장 도입부)를 탐지하는 방식을 임시 사용. 단락 구분 추가 후 `[NNNN]` 경계 직접 인식으로 교체.

## 11. Embodiment 기술 단락 작성 권장사항

> B1·B2는 독립 단락(관용문단)이 아니라 **기술 단락 내 인라인 기재 패턴**이다.  
> 부모 에이전트(Embodiment 생성)가 구성요소·예시 단락을 생성할 때 아래 규칙을 적용한다.

### 11-1. 개방형 구성요소 기재 (B1 — 813건, 25.5%)

**원칙**: 청구항의 개방형 기재(`포함하는`)를 본문 단락에서 미러링한다.

**적용 위치**: 구성요소 열거 단락의 **마지막 문장** 또는 **독립 문장**으로 삽입.

**법적 근거**: 특허법 42조 4항 1호 (뒷받침) — 청구항 상위개념의 추가 구성요소 포함 가능성 보전.

**권장 표현 패턴**:
```
[구성요소명]을 더 포함할 수 있다.
[구성요소명]를 추가로 구비할 수 있다.
[구성요소명]을 더 포함하거나 생략할 수도 있다.
```

**적용 조건**:
- `claim_scope_type == 'open'` 인 경우 필수
- 주요 구성요소 단락마다 1회 — 모든 단락에 반복 삽입하면 과잉
- 실시예가 복수인 경우(has_multiple_embodiments) 각 실시예별 1회

**주의**: "더 포함할 수 있다" 자체가 문장 전체를 구성하는 경우(독립 단락 형태)도 허용되나,
구성요소 설명 문장 뒤에 자연스럽게 이어 붙이는 것이 가독성 측면에서 우선.

---

### 11-2. 열거 예시 한정력 차단 (B2 — 556건, 17.4%)

**원칙**: 예시 또는 실시예 열거 후 해당 열거가 발명을 한정하지 않음을 명시한다.

**적용 위치**: 예시·열거 단락의 **문장 말미** 또는 **바로 다음 문장**.

**법적 근거**: 특허법 42조 4항 1호 (뒷받침) — 실시예 열거가 청구항 범위를 축소하지 않도록 방어.

**권장 표현 패턴**:
```
본 발명이 이에 한정되는 것은 아니다.
이에 한정되지 않으며, 다양한 변형이 가능하다.
이에 제한되지 않고 다른 [구성·방법·재료] 도 사용할 수 있다.
상기 실시예에 한정되지 않는다.
```

**적용 조건**:
- "예를 들어", "예컨대", "일 실시예로서" 등 예시 표현이 포함된 단락 뒤에 삽입
- 수치 범위·재료 선택지를 열거한 단락 뒤에 삽입
- 복수의 선택지 중 하나를 "바람직하게는"으로 특정한 경우 반드시 삽입

**주의**: B2를 모든 단락에 기계적으로 반복하면 명세서 품질이 저하되고
심사관에게 "구성이 불확정"이라는 인상을 줄 수 있다. 열거·예시가 있는 단락에만 선택적으로 적용.

---

### 11-3. 부모 에이전트 연동 규칙

| 조건 | 적용 패턴 | 삽입 방식 |
|---|---|---|
| 구성요소 열거 단락 + `claim_scope_type == 'open'` | B1 | 단락 말미 문장 추가 |
| "예를 들어"·"예컨대" 포함 단락 | B2 | 단락 직후 독립 문장 |
| 수치 범위 열거 단락 | B2 | 단락 말미 또는 직후 |
| 바람직한 실시예 특정 단락 | B2 | 단락 말미 |
