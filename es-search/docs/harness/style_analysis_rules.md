# Style Analysis Rules

명세서 스타일 분석은 청구항 축과 본문 축을 분리한다. 수치 근거 없는 해석은 금지하고, 불확실하면 `[추정]`으로 표시한다.

## JP 번역 출원 처리

JP 번역 출원은 국내 작성 스타일과 다른 별도 클러스터다.

식별 기준:

- `Embodiment` 또는 `Background`에 `실시형태` / `실시 형태` 포함
- 일본 법인 출원인
- 번역체 표현: `당해`, `에 수반하여`, `구비하는`, `실시 형태`

분석 시 JP 포함/제외 여부를 반드시 명시한다.

## 발명 본질 유형 (CPC 기준)

| 유형 | 주요 CPC | 본문 전개 |
|---|---|---|
| 도면-공간형 | H·B·E·F, G01/G02/G03 일부 | 도면 순서, 구조·공간 배치 중심 |
| 실험-수치형 | C, H 공정, G03F | 공정 단계, 수치 범위, 비교예 |
| 동작-시간형 | G06·G11·G16 | 장치 블록도 → 방법 흐름도, S단계 시간순 |

주의: G계열 전체가 동작-시간형은 아니다.

- G06·G11·G16: 동작-시간형
- G01: 도면-공간형에 가까움
- G02·G03: 도면-공간형 또는 실험-수치형

## 청구항 축

| 항목 | 내용 |
|---|---|
| 청구항 구조 | 독립항 수, 종속항 수, 평균 청구항 길이 |
| 독립항 전략 | 출원 단위 전략 패턴(장치단독/순수형/3-way 등) |
| 표현 방식 | 기능적 vs 구조적 표현, 연결어 |
| 도면 참조 | 괄호형 `(10)` / 인라인형 `도 1` |
| 구성요소 기재 | 묶음형 vs 종속항 분리형 |

주의: 종속항은 독립항 카테고리를 상속한다. 말미 총량 통계로 독립항 전략을 해석하지 않는다.

## 본문 축

| 항목 | 필드 | 내용 |
|---|---|---|
| 선행기술 인용 방식 | `Background` | 특허문헌 인용 vs 문제만 서술 |
| 과제 제기 어법 | `Problem` + `Background` | 직접("문제가 있었다") vs 간접("요구된다") |
| 실시예 구조 | `Embodiment` | 단일/복수, 변형예 포함 여부 |
| 효과 표현 | `Effects` | 채움률, 정성표현, 수치표현 |
| 설명 밀도 | `Embodiment` | 분량, 실시예 개수, 도면 참조 밀도 |
| 언어 패턴 | 전체 | 특이 표현·상투구 |

제외 또는 주의:

- 도면 연계 밀도는 단독 변별력은 낮지만 개별 문서 분석에서는 유용하다.
- 효과 수치표현은 전반적으로 낮아 보조 지표로만 쓴다.
- `SolutionProblem` mirror율은 사무소별 채움률 차이가 커 횡단 비교에 부적합하다.

## 주요 측정 정규식

```python
import re

CITE_PAT = re.compile(r"특허문헌|선행기술문헌|공개특허|등록특허|특허\s*제?\s*\d|KR\s*\d{2}|특개|일본 특허|미국 특허")
PROBLEM_DIRECT = re.compile(r"문제(가|점이|점을|점도)\s*(있었|있다|발생|존재)")
PROBLEM_INDIRECT = re.compile(r"(요구|필요|요청)(되고|된다|되었|하고 있|성이)")
MULTI_EMB = re.compile(r"제\s*[1-9][0-9]?\s*(실시\s*예|실시\s*형태|구현\s*예|실시예|실시형태|구현예)")
EFFECT_QUAL = re.compile(r"현저히|크게|효과적으로|용이하게|향상|개선|우수")
```
