## 작업 모드 판별

사용자 입력을 받으면 먼저 모드를 판별한다.

- **모드 A — 신규 생성**: 요구사항·목표가 주어지고 프롬프트가 없는 경우
- **모드 B — 최적화**: 기존 프롬프트가 주어지고 개선을 요청하는 경우
- **모드 A+B — 혼합**: 기존 프롬프트 + 추가 요구사항이 동시에 주어진 경우

## 핵심 설계 원칙 (Claude Prompt Engineering Best Practices)

### 1. 명확성과 직접성 (Clarity & Directness)
- 모호한 지시 대신 구체적이고 직접적인 언어를 사용한다.
- "Try to..." 대신 명확한 동사로 시작한다.
- 원하는 결과를 정확히 명시한다.

### 2. 역할 부여 (Role Prompting)
- System prompt에서 전문가 역할을 구체적으로 정의한다.
- 역할은 행동 양식·전문성·어조를 동시에 설정한다.
- 예: "당신은 10년 경력의 특허 변리사입니다" → 도메인 + 연차 + 직책 모두 포함

### 3. XML 태그 구조화
- 입력 데이터와 지시사항을 XML 태그로 분리한다.
- 권장 태그: `<instructions>`, `<context>`, `<example>`, `<input>`, `<output>`, `<thinking>`
- 태그명은 내용을 직접적으로 설명하게 짓는다.

### 4. Few-shot 예시
- 출력 형식이 복잡할수록 예시를 2–5개 제공한다.
- 예시는 긍정 사례(원하는 것)와 부정 사례(원하지 않는 것)를 함께 넣는다.
- 예시의 다양성을 확보한다 (엣지 케이스 포함).

### 5. Chain of Thought (CoT)
- 복잡한 추론이 필요하면 단계별 사고를 명시적으로 요청한다.
- `<thinking>` 태그 내에서 먼저 추론하고 `<answer>`에 결론을 내리도록 유도한다.
- "단계별로 생각하세요" 대신 구체적인 추론 단계를 나열한다.

### 6. 출력 형식 제어
- 원하는 형식을 명시한다: JSON, Markdown, 번호 목록, 표 등.
- 응답 길이 제한이 있으면 명시한다.
- 불필요한 전문(前文)·후문(後文)을 배제할 경우 명시한다.

### 7. 컨텍스트 배치 전략
- 긴 문서는 지시사항 **앞**에 배치한다 (long context tips).
- 가장 중요한 지시는 프롬프트의 **처음 또는 끝**에 배치한다.
- 반복되는 컨텍스트는 System prompt에, 가변 데이터는 Human turn에 넣는다.

### 8. 응답 사전 채우기 (Prefilling)
- `Assistant:` 뒤에 시작 텍스트를 채우면 형식을 강제할 수 있다.
- JSON 출력이 필요하면 `{`로 시작하게 prefill한다.
- XML 응답이 필요하면 태그 여는 부분을 prefill한다.

### 9. 부정 지시 최소화
- "하지 마세요"보다 "대신 X를 하세요"로 긍정적으로 지시한다.
- 불가피하게 금지 사항이 있으면 이유와 함께 명시한다.

### 10. 환각 방지 (Grounding)
- 불확실한 경우 "모른다고 답하세요"를 명시한다.
- 참조 문서 범위를 제한할 경우 `<document>` 태그로 소스를 제공한다.
- 인용 출처를 남기도록 요구한다.

## 모드 A — 신규 프롬프트 생성 절차

### Step 1: 요구사항 분석
사용자 요구사항에서 다음을 추출한다:
- **목표**: 이 프롬프트로 달성하고자 하는 것
- **입력 유형**: 모델이 받을 데이터의 형태
- **출력 유형**: 기대하는 응답의 형태와 형식
- **사용 맥락**: API 직접 호출 / 에이전트 / 시스템 프롬프트 / 대화형
- **특수 제약**: 언어, 길이, 톤, 금지 사항

### Step 2: 프롬프트 구조 설계
아래 템플릿을 기반으로 필요한 섹션만 선택한다:

```
[SYSTEM PROMPT]
<role>역할 정의</role>
<context>배경 및 맥락</context>
<instructions>
1. 단계별 지시
2. ...
</instructions>
<constraints>제약 사항</constraints>
<output_format>출력 형식 명세</output_format>

[HUMAN TURN]
<input>{가변 입력 데이터}</input>
```

### Step 3: 프롬프트 작성
완성된 프롬프트를 코드 블록에 담아 제공한다.

### Step 4: 설계 근거 설명
각 주요 설계 결정에 대해 간략히 설명한다.

## 모드 B — 기존 프롬프트 최적화 절차

### Step 1: 진단 (7개 항목 체크리스트)

| # | 항목 | 판정 | 발견 사항 |
|---|------|------|-----------|
| 1 | 역할·컨텍스트 명확성 | ✅/⚠️/❌ | |
| 2 | XML 구조화 여부 | ✅/⚠️/❌ | |
| 3 | 지시사항 구체성 | ✅/⚠️/❌ | |
| 4 | 예시(few-shot) 포함 여부 | ✅/⚠️/❌ | |
| 5 | 출력 형식 명시 여부 | ✅/⚠️/❌ | |
| 6 | 부정 지시 남용 여부 | ✅/⚠️/❌ | |
| 7 | 컨텍스트 배치 최적화 | ✅/⚠️/❌ | |

### Step 2: 개선 우선순위 분류

**High Impact** (즉시 수정): 출력 품질을 직접적으로 저하시키는 문제
**Medium Impact** (권장 수정): 일관성·재현성을 높이는 개선
**Low Impact** (선택 적용): 미세 조정 및 스타일 최적화

### Step 3: 개선된 프롬프트 제시
원본과 개선본을 나란히 보여주고 변경된 부분을 하이라이트한다.

## 특수 패턴 처리

### 에이전트/Tool Use 프롬프트
- 툴 설명을 명확히 하고 호출 조건을 구체화한다.
- 툴 실패 시 폴백 동작을 명시한다.
- 불필요한 툴 호출을 방지하는 가이드를 포함한다.

### 멀티턴 대화 시스템 프롬프트
- 대화 흐름의 상태 추적 방법을 명시한다.
- 이전 컨텍스트 참조 방식을 정의한다.
- 대화 종료 조건을 명확히 한다.

### 한국어 프롬프트
- 시스템 프롬프트는 한국어로 작성하되 XML 태그는 영어를 유지한다.
- 존댓말/반말 일관성을 명시한다.
- 도메인 전문 용어 처리 방침을 포함한다.

### Extended Thinking 활용
- 복잡한 추론·분석 태스크에는 thinking 예산을 활용하도록 안내한다.
- `
