# LLM 토픽 추출 — 검토 기록

## 배경

사용자가 발명 내용을 입력하면 Elasticsearch + 벡터DB 하이브리드 검색으로 특허를 조회하고,
LLM이 각 특허를 읽어 토픽(목적/수단)을 자동 분류하는 "토픽 필터" 기능.

---

## 확정된 사항

| 항목 | 내용 |
|------|------|
| 자연어 검색 | 결과 100건 제한, 토픽 필터 제공 |
| 조건식 검색 | 건수 제한 없음, 토픽 추출 없음 |
| 추출 방식 | 특허 1건씩 LLM으로 토픽 추출 (구 단위) |
| 담당자 | 배성현 |

---

## 검토 질문

### Q1. 토픽은 목적/수단에서 어떤 기준으로 뽑을 것인가?

**상태: 미결정**

- 추출 단위: "구(句) 단위" — 짧은 명사구 형태 (예: "센싱 정확도 확보", "근접 로봇 그룹핑")
- 추출 구조: 목적 / 수단 두 카테고리로 분류하는 포맷 사용 (테스트 결과에서 확인)
- **미결**: 하나의 특허에서 몇 개를 뽑을지, 어떤 구를 우선하는지에 대한 기준은 논의되지 않음

**확인 필요 대상**: 배성현 (프롬프트 설계 내 기준 명문화 여부)

---

### Q2. 사용자 쿼리 포함이 특허 해석의 오염 원인이 되지 않는가?

**상태: 미결정 + 팀 내 미인식 리스크**

**팀 내 논의 흐름**

| 인물 | 입장 |
|------|------|
| 이윤주 | 처음엔 쿼리 포함 찬성 → 번복: "검색 결과가 이미 유사 특허이므로 불필요" |
| 한상호 | "쿼리를 넣으면 LLM이 사용자 관점 파악 → 더 적합한 토픽 도출" |
| 배성현 | 두 버전(포함/미포함) 테스트 후 결과 공유 — 결론 없이 종료 |

**테스트 결과 요약 (쿼리: "자율주행")**

| 특허 | 쿼리 미포함 (v1) | 쿼리 포함 (v2) |
|------|-----------------|----------------|
| 원격 로봇 제어 | 센싱 정확도 확보 | 동시 주행 안전 |
| 원격 차량 제어 | 원격 제어 신뢰성 | 원격 자율주행 제어 |
| 차량 위치 보정 | 차량 위치 정밀화 | 자율주행 위치 정밀화, 협력적 위치추정 |

**리스크 (팀 미인식)**
- "더 좋은 결과" 관점에서만 논의됨
- 특허 문서 본래 의미가 사용자 쿼리에 의해 왜곡될 가능성(해석 편향)은 아직 검토되지 않음

**전체 20건 패턴 분석 (테스트 쿼리: "자율주행")**

| 패턴 | 건수 | 해당 특허 |
|---|---|---|
| B. 청구항에 더 충실 (실질 개선) | 10건 | KR1020230114582, KR1020210106338, KR1020220125178, KR1020210081613, KR1020220020135, KR1020210154593, KR1020220038128, KR1020210168876, KR1020220027723, KR1020220185420 |
| C. 청구항 외부 개념 혼입 (해석 편향) | 4건 | KR1020220116115, KR1020220061951, KR1020237005632, KR1020190132368 |
| A. 쿼리 문자열 기계적 삽입 | 3건 | KR1020210149252, KR1020220066197, KR1020220164741 |
| D. 혼재 (일부 개선 + 일부 편향) | 3건 | KR1020200165190, KR1020220184202, KR1020210144695 |

**C 패턴 (해석 편향) 세부 사례**

| 특허 | v1 | v2 | 문제 |
|---|---|---|---|
| KR1020220116115 | 센싱 정확도 확보 | 동시 주행 안전 | 배경기술/발명효과에서 유래, 청구항 부재 |
| KR1020220061951 | 신호-차량 매칭, 원격 제어 신뢰성, 오작동 방지 | 네트워크 기반 제어, 차량 식별, **원격 자율주행 제어** | objectives 전체가 자율주행 프레임으로 재구성 |
| KR1020237005632 | 장애물 회피 경로, 근거리 충돌 회피 | 근거리 장애물 회피, 실시간 경로계획, **자율주행 적용** | "자율주행 적용"은 활용분야 레이블, 청구항 내용 아님 |
| KR1020190132368 | 사용자 관점 정보제공, 도로유형 기반 서비스결정 | 사용자 관점 정보, 도로유형별 서비스, **사용자 선택 수집** | 추가된 세 번째 objective의 출처 불분명 |

**A 패턴 (기계적 삽입) 세부 사례**

| 특허 | v1 | v2 | 문제 |
|---|---|---|---|
| KR1020210149252 | 사용자 맞춤화 | 자율주행 경험 개인화 | "자율주행" prefix 추가 — 의미 확장 없음 |
| KR1020220066197 | 실시간 시뮬레이션 수행 | 가상 자율주행 시뮬레이션 | 쿼리 domain label 삽입 |
| KR1020220164741 | 클라우드 시뮬레이션 운영 | 자율주행 시뮬레이션 | 도메인 레이블만 앞에 붙임 |

**종합 해석**

- v2가 청구항에 더 충실한 경우(B)가 과반(10/20)이나, 해석 편향(C) + 기계적 삽입(A)을 합치면 **7/20(35%)**에서 쿼리 영향이 부정적으로 작용
- 검색 목적에 따라 허용 가능 수준이 다름:

| 검색 목적 | 적합한 방식 | 이유 |
|---|---|---|
| 선행기술 조사 | v1 또는 쿼리 가중치 축소 | 청구항 중심 비교 필요, 배경/효과 유래 표현은 노이즈 |
| 기술 동향 파악 / 분류 | v2 허용 | 도메인 re-framing이 사용자 탐색에 유용 |

- **구조적 리스크**: 동일 특허가 쿼리에 따라 다른 토픽 버킷에 배정 → 토픽 필터 일관성 손상

**확인 필요 대상**: 배성현, 한상호 — 기능 목적(선행기술 조사 vs 동향 분류) 확정 후 프롬프트 전략 고정 필요

---

### Q3. 화면에 목적/수단 구분 없이 표시되는 토픽은 어떻게 통합하는가?

**상태: 방향 확정**

- 뒷단에서는 목적/수단 구분을 유지하여 저장
- 화면 표시 시에는 목적+수단 키워드를 통합 정제한 뒤 각 특허에 매칭
- 구분 정보는 백엔드에서만 보유, 프론트에는 단일 토픽 목록으로 노출

## 미결 사항 전체

| 항목 | 내용 |
|------|------|
| 토픽 선택 기준 | 몇 개, 어떤 우선순위로 추출할지 |
| 쿼리 포함 여부 | v1/v2 중 채택 버전 미결 |
| AND/OR 방식 | 토픽 2개 이상 선택 시 처리 방식 — 실제 분포 확인 후 결정 예정 |
| 로딩 UX | 토픽 로딩 중 화면 처리 세부 설계 필요 |
| 토픽 순서 변경 연동 | 토픽 순서 변경 시 검색 결과 정렬 변경 기능 |
| 성능 측정 | 기존 검색 대비 토픽 추출 추가 소요 시간 측정 필요 |
