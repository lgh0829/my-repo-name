## 1. 개요

### 1.1 목적

변리사가 PatSol 명세서 작성 페이지에서 **챗 패널을 통해 AI 에이전트와 대화하며** 명세서 각 섹션의 초안을 생성·검토·수정한다. 에이전트는 변리사의 메시지 맥락을 이해하여 적절한 도구(기재요건 검토, 섹션 생성, 수정 제안 등)를 선택·호출하며, 명세서 전체를 컨텍스트로 유지한다.

### 1.2 배경

- 명세서는 각 섹션이 독립적으로 잘 쓰인다고 기재요건이 충족되지 않음. 배경기술 문제점 → 해결과제 → 해결수단(청구항 연결) → 실시예(구성요소 전부 뒷받침)의 연쇄가 일관되어야 함
- 섹션별 인라인 AI 보조 방식은 이 연쇄를 확인할 수 없고, 섹션마다 생성 근거가 분산됨
- 기재요건 검토는 청구항별·명세서 전체·도면 단위로 별도 기능화되어 있음. AI 어시스턴트는 이를 독립 기능으로 포함하는 것이 아니라, 챗 맥락에 따라 해당 도구를 에이전트로서 호출하는 구조
- PatSol 핵심 포지셔닝("검색 → 청구항 → 명세서 연결")의 최종 구간

### 1.3 AI 역할 한계점 설정 (명시적 원칙)

- AI는 **초안 생성·도구 실행 보조 에이전트**이며, 기재요건 충족 여부 판단·법적 표현의 최종 결정은 변리사에게 있음
- 명세서 편집 영역은 변리사가 직접 소유·편집하는 주 공간이며, 챗 패널은 보조 인터페이스임
- 도구 실행 결과는 변리사가 명시적으로 반영 액션을 취해야만 명세서에 적용됨
- AI는 제안형 표현을 사용하며 법적 결론을 단정하지 않음

## 2. 용어 정의

### 2.1 포함 관계

```
명세서 작성 페이지 (/document)
├── 명세서 편집 영역  ← 변리사의 주 편집 공간
│   ├── 섹션 블록 × N
│   │   ├── 섹션 텍스트 (AI 생성 또는 수동)
│   │   ├── AI 생성 레이블
│   │   └── 섹션 생성 버튼
│   └── 드래그 선택 핸들  → 챗 연동 트리거
└── 챗 패널  ← AI 에이전트 인터페이스 (상시 표시, 토글 가능)
    ├── 대화 메시지 목록
    │   ├── 사용자 메시지 (직접 입력 / 드래그 인용 포함)
    │   └── AI 응답
    │       ├── AI 작업 내역 (접을 수 있음)
    │       ├── 응답 텍스트
    │       ├── 섹션 앵커 배지 (해당 시)
    │       ├── 수정 전후 비교 (수정 제안 시)
    │       │   └── 반영 버튼
    │       └── 변경 항목 목록 (T-10 실행 시)  #현재 개발 대상에서 제외
    │           ├── 개별 항목 체크박스 + [이동] 버튼
    │           └── [선택 항목 반영]  [전체 반영]  [전체 취소]
    ├── 메시지 입력창
    └── 생성 횟수 안내 (T-01·T-02 실행 직전)
```

### 2.2 용어 정의

| 용어 | 정의 |
| --- | --- |
| **에이전트** | 변리사의 메시지 의도를 파악하여 적절한 기능을 선택·실행하고 결과를 종합하여 응답하는 AI. 단순 텍스트 생성과 달리 기재요건 검토·섹션 생성 등 실제 기능 실행을 포함함 |
| **챗 패널** | 명세서 전체 내용을 참조하며 AI와 대화하는 인터페이스. 상시 표시가 기본이며 토글로 닫을 수 있음 |
| **AI 작업 내역** | 에이전트가 기능을 실행하는 과정을 단계별로 챗 패널에 표시하는 영역. 완료 후 접힌 상태로 유지되며, 문제 발견 시 전면 노출 |
| **드래그 인용** | 명세서 편집 영역에서 텍스트를 선택하면 해당 내용이 챗 패널 입력창에 인용 블록으로 자동 삽입되는 기능 |
| **섹션 앵커 배지** | 챗 응답이 명세서의 특정 섹션을 참조할 때 응답 내에 삽입되는 클릭 가능한 링크. 클릭 시 편집 영역의 해당 섹션으로 이동 |
| **수정 전후 비교** | AI가 수정을 제안할 때 현재 텍스트와 제안 텍스트를 나란히 보여주는 확인 화면. 변리사가 반영 여부를 결정하는 단계 |
| **수정 반영** | 수정 전후 비교에서 AI 제안 내용을 명세서에 실제로 적용 |
| **변경 항목 목록** | 선호 표현 일괄 적용 시 수정 반영 결정을 위한 챗 패널에 표시되는 변경 항목 목록 |
| **생성 횟수 안내** | 명세서 생성 기능 실행 전 잔여 생성 횟수와 차감 횟수를 표시하는 안내 |
| **AI 생성 레이블** | 섹션이 AI에 의해 생성되었음을 표시하는 UI 표식. 변리사가 직접 수정 시 "수동 수정됨"으로 전환 |
| **수동 수정 보호** | 변리사가 직접 수정한 섹션에 AI가 재생성·수정할 때, 수정 전·후 내용을 프롬프트 컨텍스트로 전달하여 AI가 변리사의 수정 방향성을 인식하고 존중하도록 하는 정책 |
| **기재불비 기준** | 명세서 기재가 법적 요건을 충족하지 못하는 패턴 및 판단 기준. 에이전트가 생성·검토 전 과정에서 참조하는 기준점 |
| **기재요건** | 청구항과 명세서 본문 간 용어 일관성, 실시예의 청구항 구성요소 전부 뒷받침 등 법적 충족 조건 |
| **사무소 스타일** | 특허사무소별 어체·항목 순서·표현 방식 설정값 |
| **작성전략** | 명세서 스타일 설정에 포함된 작성 방향·강조 포인트 등 전략적 지침 |
| **선호 표현 규칙** | 치환 규칙 단위로 등록된 표현 선호 설정 (예: 않다 → 아니하다). 스타일 설정과 충돌 시 우선 적용 |

## 3. 사용자 스토리

1. 변리사로서, 자연어 메시지로 AI에게 명세서 작성·검토·수정을 지시할 수 있다, 매번 컨텍스트를 설명하지 않아도 되도록
2. 변리사로서, AI 에이전트가 어떤 도구를 실행하는지 단계별로 볼 수 있다, 에이전트의 동작을 신뢰하고 이해할 수 있도록
3. 변리사로서, AI의 수정 제안을 반영 전에 수정 전후 비교로 확인할 수 있다, 원치 않는 변경이 적용되지 않도록
4. 변리사로서, 선호 표현 일괄 적용 시 변경 목록에서 항목별로 수락/거절할 수 있다, 규칙의 예외 케이스를 직접 제어하도록
5. 변리사로서, 의도가 모호한 지시를 입력했을 때 AI가 선택지를 제시해준다, 의도하지 않은 작업이 실행되지 않도록
6. 변리사로서, 기재요건 검토 결과를 챗 패널에서 바로 확인하고 수정 적용 버튼을 클릭할 수 있다, 편집 영역과 챗을 쉽게 오갈 수 있도록
7. 변리사로서, 명세서 생성 전 잔여 횟수를 확인할 수 있다, 예상치 못한 사용량 소비를 방지하도록
8. 변리사로서, 챗 대화 히스토리가 프로젝트 기간 동안 유지된다, 이전 지시와 수정 내역을 참조할 수 있도록
9. 변리사로서, 명세서 텍스트를 드래그 선택하여 해당 부분에 대한 질문을 AI에게 바로 던질 수 있다, 문서와 대화 사이의 맥락 전환을 줄이도록
10. 변리사로서, 새 대화 시작 옵션으로 이전 맥락 없이 깨끗하게 대화를 시작할 수 있다, 누적된 맥락이 응답에 과하게 영향을 줄 때 초기화하도록

## 4. 정책 및 규칙

| # | 정책 | 내용 |
| --- | --- | --- |
| P-1 | AI 생성 배지 표시 | AI가 생성한 섹션에는 AI 생성 배지를 표시한다. 변리사가 직접 수정하면 "수동 수정됨"으로 전환된다 |
| P-2 | 수동 수정 보호 | 변리사가 직접 수정한 섹션에 AI가 다시 수정(명세서 섹션 단일 생성, 일괄 생성)할 경우, 수정 전·후 내용을 프롬프트 컨텍스트로 전달하여 AI가 변리사의 수정 방향성을 인식하고 존중하는 방향으로 생성한다. |
| P-3 | 생성 비활성화 조건 | 청구항이 1개 이상 저장되지 않은 상태에서는 섹션 생성 기능과 챗 패널의 생성 명령이 disabled 처리된다. |
| P-4 | 법적 단정 표현 금지 | AI 응답에 "신규성이 있다", "등록 가능하다" 등 법적 결론을 단정하는 표현을 포함하지 않는다 |
| P-5 | 컨텍스트 부족 고지 | 청구항 미입력 등 필수 컨텍스트 누락 시 생성을 중단하고 누락 항목을 챗 패널에 안내한다 |
| P-6 | 스타일·선호 표현 적용 순서 | 사무소 스타일 + 작성전략을 기본으로 적용하고, 선호 표현 규칙이 충돌 시 우선 적용한다 |
| P-7 | 챗 컨텍스트 동기화 | 명세서 편집 영역의 내용이 변경되면 챗 패널의 명세서 상태가 자동으로 갱신된다 |
| P-8 | 섹션 앵커 배지 포함 | 챗 응답이 명세서의 특정 섹션을 참조할 때는 반드시 섹션 앵커 배지를 포함한다 |
| P-9 | 생성 요약 챗 출력 | 섹션 생성(명세서 섹션 단일 생성, 일괄 생성) 완료 시 챗 패널에 고려사항 요약을 자동 출력한다 |
| P-10 | 생성 횟수 안내 필수 | 생성 실행 전 항상 잔여 생성 횟수와 1회 차감 사실을 인라인으로 표시한다. 의도가 명확한 경우(즉시 실행)라도 예외 없이 적용된다 |
| P-11 | 기재요건 검토 구독 게이트 | 기재요건 검토(T-03~T-06)는 구독 사용자가 명시적으로 버튼을 클릭하거나 챗에서 요청한 경우에만 실행된다. 에이전트가 임의로 자동 실행하지 않는다. 미구독자에게는 검토 버튼 4개를 locked 상태로 노출한다(숨기지 않음). locked 버튼 클릭 시 구독 유도 안내를 1회 표시하며 반복 노출은 금지한다. |
| P-12 | 의도 분류 실행 원칙 | 변리사 메시지의 의도가 명확하면 즉시 실행(인라인 표시 후 즉시 실행), 모호하면 의도 확인(선택지 제시)로 처리한다. 도구 유형(읽기 전용/문서 변경) 무관하게 적용 |
| P-13 | 다중 도구 실행 방식 | 읽기 전용 도구는 자동 실행한다. 문서 변경 도구 실행 직전에는 반드시 멈추고 변경 미리보기 또는 횟수 고지를 표시한다 |
| P-14 | 챗 히스토리 영속 | 챗 대화 히스토리는 프로젝트 기간 전체 동안 유지된다. 변리사는 "새 대화 시작" 옵션으로 이전 맥락 없이 새 대화를 시작할 수 있다. 새 대화 시작이 선호 표현·스타일 설정을 초기화하지는 않는다 |
| P-15 | LLM 동시 실행 금지 | LLM을 사용하는 기능(T-01~T-10 중 LLM 호출 포함 도구)이 실행 중인 동안 다른 LLM 기능은 실행할 수 없다. 실행 중에는 편집 영역의 검토 버튼 4개 및 챗 패널 입력창이 모두 disabled 처리된다. |

## 5. AI 동작 정의

### 5.1 입력 컨텍스트 구성

챗 패널은 항상 아래 입력 컨텍스트를 LLM에 전달한다. 도구 호출과 달리 입력 컨텍스트는 실행되지 않으며, 에이전트가 읽고 참조하는 배경 데이터다.

| 항목             | 설명                                                                           | 청구항 생성 | 명세서 본문 생성 | 명세서 도면 생성 |
| -------------- | ---------------------------------------------------------------------------- | ------ | --------- | --------- |
| 청구항 목록         | 독립항·종속항 전체 텍스트 및 계층 구조                                                       | 필수     | 필수        | 필수        |
| 현재 명세서 전체 텍스트  | 작성 완료·미작성 포함한 모든 섹션 현재 상태                                                    | 권장     | 필수        | 필수        |
| 기재불비 기준        | 기재불비 해당 패턴 및 판단 기준. 생성·검토 전 과정에서 에이전트가 내재화하는 기준점                             | 필수     | 필수        | 필수        |
| 비교발명 분석        | 선행특허와 내 발명의 구성요소 대비 분석 결과. 매칭된 선행특허 리스트·검색 결과 요약 포함. 회피 포인트 및 차별점 기술의 근거로 활용 | 필수     | 권장        | 선택        |
| 발명 요약 (직무발명서)  | 발명의 배경·목적·효과 원문                                                              | 필수     | 권장        | 권장        |
| 발명 자료          | 발명자가 제공한 원본 도면·구성도·발명 설명 자료                                                  | 권장     | 필수        | 필수        |
| 명세서 스타일 + 작성전략 | 적용 중인 스타일 설정 전문 및 작성 방향·전략적 지침                                               | 선택     | 선택        | 선택        |
| 선호 표현 규칙       | 등록된 치환 규칙 목록. 생성·수정 전 과정에 자동 반영                                              | 선택     | 선택        | 선택        |
| 드래그 인용 텍스트     | 드래그 인용 사용 시 선택된 텍스트 + 섹션 ID                                                  | 상황별    | 상황별       | 상황별       |
| 수동 수정 보호 섹션 목록 | 반영 버튼 적용 시 스킵할 섹션 ID 목록                                                      | 자동     | 자동        | 자동        |


**컨텍스트 트런케이션 우선순위**

우선순위 기본값 (기획자 판단 기준): 기재불비 기준 > 청구항 > 드래그 인용 텍스트 > 현재 명세서 > 발명 요약 > 비교발명 분석

- 명세서 전체 포함 시 토큰 한도 초과 대응 방안 (요약 압축 vs 해당 섹션 + 인접 섹션만 포함) (→ OQ-7)

### 5.3 의도 분류 기준

변리사 메시지를 수신하면 에이전트는 의도 명확성을 판단하여 실행 방식을 결정한다. (→ P-12)

| 조건 | 방식 | 동작 |
| --- | --- | --- |
| 의도가 명확한 메시지 | **즉시 실행** | AI 작업 내역에 실행 도구를 인라인 표시 후 즉시 실행 |
| 의도가 모호한 메시지 | **의도 확인** | 선택지를 챗에 먼저 제시. 변리사가 선택 후 실행 |
| 기재요건 검토 범위 불명확 ("기재요건 검토해줘" 등) | **의도 확인** | C-04 선택지 카드 제시: ① 청구항만 ② 발명의 설명만 ③ 도면만 ④ 전체 검토 |
| 명세서 단일/일괄 섹션 생성 실행 (의도 불문) | **즉시 실행 예외** | 항상 생성 횟수 안내 후 확인 버튼 표시 (→ P-10) |

**즉시 실행 예시:**

```
변리사: "청구항 기재요건 검토해줘"
  → ⟳ 기재요건 검토 — 청구항 (T-03) 실행 중...
  → [결과 출력]
```

**의도 확인 예시:**

```
변리사: "기재요건 검토해줘" (범위 불명확)
  → 어떤 범위를 검토할까요?
    ① 청구항만
    ② 발명의 설명만
    ③ 도면만
    ④ 전체 검토 (청구항 + 발명의 설명 + 도면 + 교차 항목)

변리사: "배경기술 좀 봐줘"
  → 어떤 작업을 원하시나요?
    ① 기재요건 검토
    ② 수정 제안 작성
    ③ 섹션 다시 생성
```

### 5.4 다중 도구 실행 원칙

하나의 메시지가 여러 도구의 순차 실행을 요구할 때 다음 방식으로 처리한다. (→ P-13)

```
예: "청구항 기재요건 확인하고 문제 있으면 수정해줘"
    ↓
1단계: T-03 기재요건 검토 — 청구항 (읽기 전용 → 자동 실행)
    ↓
에이전트: 문제 발견. 수정 제안을 보여줍니다.
    ↓
2단계: T-07 섹션 수정 제안 (문서 변경 → 변경 미리보기 표시 후 중단)
    ↓
변리사: 변경 미리보기 확인 후 [반영] 클릭
```

**전체 검토 실행 순서:**

전체 검토 버튼 클릭 시 T-03 → T-06 → T-05 → T-04 순서로 순차 실행한다. T-03 Fresh 캐시가 있을 경우 LLM 호출을 스킵하되 캐시 결과는 T-04에 컨텍스트로 전달한다. (→ 7.8)

```
전체 검토 버튼 클릭
    ↓
T-03 청구항 검토
  → Fresh 캐시 있음: ✓ 청구항 검토 — 이전 결과 사용 (LLM 스킵, 결과 T-04에 전달)
  → Fresh 캐시 없음: LLM 실행
    ↓
T-06 발명의 설명 검토 (LLM 실행)
    ↓
T-05 도면 검토
  → 도면 없음: 스킵 + "도면 없음 — 도면 검토 생략" 표시
    ↓
T-04 교차 항목 검토 (T-03·T-06 결과를 컨텍스트로 수신)
```

- 읽기 전용 도구는 자동으로 연속 실행
- 문서 변경 도구 직전에는 반드시 멈추고 변경 미리보기 또는 생성 횟수 안내를 표시
- 중간 단계의 읽기 전용 결과는 다음 도구의 인풋으로 자동 전달

### 5.5 출력 스펙

**생성 대상 섹션**

| 섹션 | 한국 출원 기준 항목명 |
| --- | --- |
| 기술분야 | 발명이 속하는 기술 분야 |
| 배경기술 | 종래 기술 및 문제점 |
| 해결과제 | 발명이 이루고자 하는 기술적 과제 |
| 해결수단 | 과제 해결을 위한 수단 (청구항 재기술 포함) |
| 발명의 효과 | 발명의 효과 |
| 실시예 | 발명의 실시를 위한 구체적 내용 |
| 도면 설명 | 도면의 간단한 설명 (도면 있는 경우) |

**출력 포맷 기준**

- T-01·T-02 생성 결과는 명세서 편집 영역에 직접 반영 + 챗 패널에 생성 요약 자동 출력 (→ P-9)
- T-07 수정 제안은 챗 패널 수정 전후 비교로 출력 + 반영 버튼 제공. 변리사가 반영 버튼을 눌러야만 적용됨
- 청구항에서 사용된 구성요소 명칭을 그대로 사용 (임의 동의어 치환 금지)
- 실시예는 독립항의 모든 구성요소를 최소 1회 이상 포함
- 섹션별 출력 길이 기준은 현재 상태 유지

### 5.6 프롬프트 설계 원칙

- **변리사 주체성 유지**: "~로 판단됩니다", "~를 고려하여 작성하였습니다" 등 제안형 어미 사용
- **청구항 용어 준수**: 입력된 청구항 텍스트의 구성요소 명칭을 임의 변경하지 않음
- **기재불비 기준 준수**: 입력 컨텍스트의 기재불비 기준을 생성 단계부터 반영하여 기재불비를 사전에 방지
- **섹션 간 일관성 지시**: 생성·수정 시 전체 명세서 컨텍스트를 참조하여 다른 섹션과의 논리적 연결 유지
- **선호 표현 우선 적용**: 선호 표현 규칙이 있을 경우 사무소 스타일보다 우선 적용 (→ P-6)
- **사무소 스타일 주입**: 스타일 + 작성전략 설정이 있는 경우 시스템 프롬프트 최상단에 삽입
- **할루시네이션 억제**: 입력 컨텍스트에 없는 기술적 사실 생성 금지 지시

### 5.7 폴백 및 불확실성 처리

| 상황 | 동작 |
| --- | --- |
| 청구항 미입력 | T-01·T-02 disabled. 챗 패널에 이유 안내 (→ P-3, P-5) |
| 발명 요약 누락 | 청구항만으로 생성 진행. 생성 요약에 "발명 요약 없음 — 구체성 보완 권장" 표시 |
| 비교발명 분석 누락 | 배경기술 생성 후 챗에 "비교발명 분석 미연결 — 배경기술 구체화에 한계 있음" 안내 |
| 기재요건 검토 미구독 | T-03~T-06 검토 버튼을 locked 상태로 노출. 클릭 시 구독 유도 안내 1회 표시. 반복 노출 금지 (→ P-11) |
| 잔여 생성 횟수 소진 | T-01·T-02 disabled. 챗 패널에 소진 안내 + 플랜 업그레이드 링크 제공 |
| 생성 실패 (API 오류) | 챗 패널에 오류 메시지 + 재시도 버튼 |
| 토큰 한도 초과 | TBD: 트런케이션 적용 후 "일부 컨텍스트가 축소되었습니다" 안내 (→ OQ-7) |

## 6. AI 품질 기준

### 6.1 정확성 목표

| 지표 | 목표값 | 측정 방법 |
| --- | --- | --- |
| 청구항-명세서 용어 일관성 | 미정 | 자동 용어 추출 후 불일치 항목 수 카운트 |
| 실시예의 청구항 구성요소 뒷받침율 | 미정 | 독립항 구성요소 대비 실시예 포함 비율 |
| 변리사 수정 없이 사용 비율 | 미정 | 생성 후 수정 없이 저장된 섹션 비율 |
| 기재요건 검토 결과 정확도 | 미정 | 플래그 발생 건 중 변리사가 실제 수정한 비율 |

### 6.2 허용 불가 출력 유형

- 청구항에 없는 구성요소를 명세서에서 신규로 추가·발명하는 경우
- "본 발명은 신규성이 있다", "등록 가능하다" 등 법적 결론을 단정하는 표현
- 특정 선행특허를 근거 없이 무효 또는 회피 완료로 단정하는 표현
- 청구항의 구성요소 명칭을 임의로 변경하거나 동의어로 치환한 경우
- 기재불비 기준에 해당하는 표현을 포함한 출력

## 7. 기능 상세

### 7.1 챗 패널

**위치:** document 페이지 > 우측 사이드 패널 (상시 표시 기본, 토글 버튼으로 닫기 가능. 마지막 상태 프로젝트별 기억)

**설명:** 명세서 전체를 컨텍스트로 유지하는 AI 에이전트 인터페이스. 에이전트는 메시지 의도를 파악하여 도구를 선택·호출하고 결과를 종합하여 응답한다.

**구성 요소**

| 요소 | 설명 |

| 대화 메시지 목록 | 사용자·AI 메시지의 시간순 스크롤 목록. 프로젝트 기간 전체 유지 |

| 메시지 입력창 | 자유 텍스트 입력. 드래그 인용 시 인용 블록 자동 삽입 |
| 새 대화 시작 버튼 | 이전 히스토리 맥락 없이 새 대화 스레드 시작. 스타일·선호 표현 설정은 유지 |
| 토글 버튼 | 챗 패널 열기/닫기. 닫으면 명세서 편집 영역이 전체 폭으로 확장 |

**동작 흐름**

```
변리사: 메시지 입력 후 전송
    ↓
의도 명확성 판단 (→ P-12)
    ↓ (명확 — 즉시 실행)           ↓ (모호 — 의도 확인)
도구 선택                       선택지 제시
    ↓                               ↓
AI 작업 내역 표시                변리사: 선택
    ↓
T-01·T-02일 경우 → 생성 횟수 안내 (→ 7.10)
T-03~T-09일 경우 → 자동 실행 (읽기 전용)
T-07·T-10일 경우 → 변경 미리보기 표시 후 중단 (→ 7.5, 7.9)
    ↓
응답 텍스트 스트리밍 출력
    ↓
섹션 참조 감지 시 → 섹션 앵커 배지 삽입 (→ 7.6)
```

**상태 조건**

| 조건 | 상태 |
| --- | --- |
| 청구항 1개 이상 저장됨 | T-01·T-02 생성 요청 enabled |

### 7.2 챗 응답 컴포넌트 유형

**위치:** document 페이지 > 챗 패널 > AI 응답 버블 내

**설명:** AI 응답은 응답 텍스트 외에 상황에 따라 인터랙티브 컴포넌트를 포함한다. 에이전트가 조건에 따라 0개 이상의 컴포넌트를 렌더링하며, 변리사의 선택을 받거나 명세서 변경을 적용하는 데 사용된다.

**AI 응답 구조**

```
AI 응답
├── AI 작업 내역 (항상 포함, 기본 접힘)         → C-01
├── 응답 텍스트 (항상 포함)
└── 인터랙티브 컴포넌트 (조건부, 0개 이상)
    ├── 섹션 앵커 배지                           → C-02
    ├── 수정 전후 비교                            → C-03
    │   └── 반영 버튼
    ├── 선택지 카드                               → C-04
    └── 변경 항목 목록  [개발 대상 제외]           → C-05
        ├── 개별 항목 체크박스 + [이동] 버튼
        └── [선택 항목 반영]  [전체 반영]  [전체 취소]
```

**컴포넌트 목록**

| ID | 컴포넌트 | 렌더 조건 | 인터랙션 | 상세 스펙 |
| --- | --- | --- | --- | --- |
| C-01 | AI 작업 내역 | 모든 AI 응답 | 클릭으로 펼침/접힘 | → 7.3 |
| C-02 | 섹션 앵커 배지 | 응답이 명세서 특정 섹션 참조 시 | 클릭 시 편집 영역 해당 섹션으로 스크롤 | → 7.6 |
| C-03 | 수정 전후 비교 | T-07 수정 제안 시 | [반영] 클릭 시 명세서에 적용 | → 7.5 |
| C-04 | 선택지 카드 | 에이전트가 의도 확인 필요 판단 시 (→ P-12) | 선택지 클릭 시 해당 도구 실행 트리거 | 하단 참조 |
| C-05 | 변경 항목 목록 | T-09 선호 표현 일괄 적용 시 | 항목별 수락/거절, 일괄 반영/취소 | → 7.9 / [개발 대상 제외] |

**C-04 선택지 카드**

렌더 조건: 에이전트가 P-12에 따라 의도 확인이 필요하다고 판단한 경우

구성 요소:
- 질문 텍스트 (에이전트 생성)
- 선택지 버튼 1~5개

동작 흐름:

```
에이전트: 의도 모호 → 의도 확인 방식 선택 (→ P-12)
    ↓
C-04 렌더 (응답 텍스트 하단)
    ↓
변리사: 선택지 클릭
    ↓
선택 결과를 컨텍스트로 추가 후 해당 도구 실행
```

상태:
- 선택 전: 모든 선택지 버튼 enabled
- 선택 후: 선택 항목 강조, 나머지 disabled (선택 취소 불가)

### 7.3 AI 작업 내역

**위치:** document 페이지 > 챗 패널 > AI 응답 버블 내 상단

**설명:** 에이전트가 도구를 실행하는 과정을 단계별로 표시한다. 실행 중에는 진행 상태를, 완료 후에는 접힌 상태로 유지된다. 이슈 발견 시 접히지 않고 전면 노출된다.

**구성 요소**

| 요소 | 설명 |
| --- | --- |
| 단계 항목 | 실행 중인 도구명 + 상태 아이콘 (⟳ 실행 중 / ✓ 완료 / ⚠ 이슈) |
| 접기/펼치기 | 완료 후 기본 접힘. 클릭으로 전체 단계 확인 가능 |

**동작 흐름**

```
에이전트: 도구 실행 시작
    ↓
단계 시각화 펼침 — 해당 단계 ⟳ 표시
    ↓
도구 실행 완료
    ↓ (이슈 없음)             ↓ (이슈 있음)
"✓ [도구명] 완료" 표시      "⚠ [도구명] — [이슈 요약]" 표시
접힌 상태로 전환            전면 노출 유지
```

**표시 예시:**

```
▾ 에이전트 작업 내역
  ✓ 의도 파악 — 기재요건 검토 (명세서 전체)
  ✓ 청구항 구조 확인
  ⚠ 실시예 뒷받침 검토 — 문제 1건
  ✓ 용어 일관성 검토
```

### 7.4 드래그 인용

**위치:** document 페이지 > 명세서 편집 영역

**설명:** 명세서 텍스트를 드래그하여 선택하면 챗 패널에 해당 텍스트를 인용한 상태로 질문을 입력할 수 있다. 드래그 인용으로 선택된 텍스트에 대한 수정 제안은 항상 부분 교체 변경 미리보기로 처리된다.

**구성 요소**

| 요소 | 설명 |
| --- | --- |
| 드래그 액션 버튼 | 텍스트 선택 시 선택 영역 근처에 플로팅 표시 (→ OQ-5) |
| 챗 인용 블록 | 선택 텍스트 + 섹션명이 입력창 상단에 인용 형태로 삽입 |

**동작 흐름**

```
변리사: 명세서 텍스트 드래그 선택
    ↓
플로팅 액션 버튼 표시
    ↓
변리사: 버튼 클릭
    ↓
챗 패널 포커스 이동
선택 텍스트 + 섹션 ID → 입력창 상단에 인용 블록 삽입
    ↓
변리사: 질문 입력 후 전송
    ↓
LLM 호출 (인용 텍스트 + 섹션 위치 + 전체 명세서 컨텍스트)
    ↓
챗 패널에 응답 출력
수정 제안 시 → 부분 교체 수정 전후 비교 표시 (→ 7.5)
```

### 7.5 챗 응답 반영 (수정 전후 비교)

**위치:** document 페이지 > 챗 패널 > AI 응답 버블

**설명:** 에이전트가 섹션 수정을 제안할 때 현재 텍스트와 제안 텍스트를 변경 미리보기 형식으로 보여준다. 섹션 전체 교체와 부분(문장·구) 교체를 모두 지원하며, 에이전트가 변경 범위를 자동 판단한다.

**변경 범위 판단 기준**

| 메시지 유형 | 적용 변경 미리보기 범위 |
| --- | --- |
| "배경기술을 다시 써줘" 등 섹션 전체 재작성 지시 | 섹션 전체 교체 변경 미리보기 |
| "이 문장 고쳐줘" 등 부분 수정 지시 | 문장·구 단위 부분 교체 변경 미리보기 |
| 드래그 인용으로 특정 텍스트 선택 후 수정 요청 | 선택 범위 부분 교체 변경 미리보기 (항상) |

**구성 요소**

| 요소 | 설명 |
| --- | --- |
| 수정 전후 비교 | `-` 삭제 텍스트(붉은색) + `+` 추가 텍스트(초록색) |
| 적용 대상 표시 | 수정 전후 비교 상단에 "[배경기술에 반영]" 형태로 섹션 명시 |
| 반영 버튼 | 변경 미리보기 확인 후 명세서에 적용 |
| 취소 버튼 | 변경 미리보기 무시 |

**동작 흐름**

```
에이전트: 수정 제안 생성
    ↓
변경 범위 자동 판단 (섹션 전체 / 부분)
    ↓
챗 패널에 수정 전후 비교 출력
[배경기술에 반영]
  - 종래 기술에서 알려진 X 장치는...
  + 종래 기술로는 Y 방법이 알려져 있으나...
[반영]  [취소]
    ↓
변리사: [반영] 클릭
    ↓
수동 수정 보호 여부 확인 (→ P-2)
    ↓ (보호 있음)
"수동 수정 내용을 덮어씁니다" 확인 다이얼로그
    ↓ (확인 / 보호 없음)
명세서 편집 영역 해당 섹션에 적용
AI 생성 레이블 표시 (→ P-1)
```

### 7.7 명세서 섹션 생성

**위치:** 트리거 진입점 2곳 — 명세서 편집 영역 섹션 헤더 / 챗 패널 메시지

**설명:** 단일 섹션 또는 전체 명세서를 생성한다 (T-01·T-02). 어디서 트리거하든 결과는 명세서 편집 영역에 반영되고, 생성 요약이 챗 패널에 자동 출력된다. 실행 전 항상 생성 횟수 안내를 거친다. (→ P-10)

**구성 요소**

| 요소 | 설명 |
| --- | --- |
| 섹션 생성 버튼 | 각 섹션 헤더 우측 배치. 청구항 없을 시 disabled |
| 전체 생성 버튼 | 편집 영역 상단 툴바. 청구항 없을 시 disabled |
| 생성 진행 표시 | 일괄 생성 시 섹션별 순차 완료 상태 표시 |
| 챗 생성 요약 | 생성 완료 후 챗 패널에 자동 출력 (→ P-9) |

**동작 흐름 — 단일 섹션**

```
트리거: 섹션 생성 버튼 클릭 또는 챗에서 "○○ 섹션 작성해줘"
    ↓
청구항 존재 여부 확인 (→ P-3)
    ↓ (없음)
챗 패널에 안내 메시지 출력 (→ P-5)
    ↓ (있음)
생성 횟수 안내 표시 (→ 7.10, P-10)
    ↓
변리사: [생성] 클릭
    ↓
AI 작업 내역 표시 (→ 7.3)
입력 컨텍스트 구성 (→ 5.1)
    ↓
LLM 호출 → 섹션 텍스트 생성
    ↓
명세서 편집 영역 해당 섹션에 반영 + AI 생성 레이블 (→ P-1)
챗 패널에 생성 요약 자동 출력 (→ P-9)
    ↓
기재요건 자동 검토 opt-in 설정 활성화 시 → 전체 검토 실행 (T-03→T-06→T-05→T-04) (→ P-12)
```

**동작 흐름 — 전체 일괄 생성**

```
트리거: 전체 생성 버튼 클릭 또는 챗에서 "명세서 전체 작성해줘"
    ↓
수동 수정 보호 섹션 확인 (→ P-2)
    ↓
생성 횟수 안내 (1회 차감 예정) (→ 7.10, P-10)
    ↓
변리사: [생성] 클릭
    ↓
보호 섹션 스킵 안내 + 섹션 순서대로 순차 LLM 호출
(기술분야 → 배경기술 → 해결과제 → 해결수단 → 효과 → 실시예)
    ↓
각 섹션 완료 시 즉시 반영
    ↓
전체 완료 후 챗 패널에 통합 생성 요약 출력 (→ P-9)
```

### 7.8 기재요건 검토

**위치:**
- 트리거 A (시스템 트리거): 청구항 섹션 헤더 버튼(T-03) / 발명의 설명 섹션 헤더 버튼(T-06) / 도면 섹션 헤더 버튼(T-05) / 편집 영역 상단 툴바 전체 검토 버튼(T-03→T-06→T-05→T-04)
- 트리거 B (에이전트 위임): 챗 패널 메시지 의도 분류 후 C-04 선택 결과
- 결과 출력: document 페이지 > 챗 패널 > AI 응답 버블

**설명:** 변리사가 버튼 클릭 또는 챗 요청으로 기재요건 검토를 실행하면 결과가 챗 패널에 출력된다. 각 T 도구의 검증 항목 상세는 frd-spec-requirements를 참조한다. (→ P-11)

**버튼 상태**

| 상태 | 조건 | 표시 |
| --- | --- | --- |
| enabled | 구독 중, LLM 미실행 | 기본 버튼 |
| loading | LLM 실행 중 | 스피너. 검토 버튼 4개 + 챗 입력창 전체 disabled (→ P-15) |
| fresh | 마지막 검토 이후 해당 범위 명세서 미변경 | 체크 배지 |
| stale | 마지막 검토 이후 해당 범위 명세서 변경됨 | 경고 배지 (케이스별 메시지 상이 — 하단 참조) |
| locked | 미구독자 | 자물쇠 아이콘. 클릭 시 구독 유도 안내 1회 표시 (→ P-11) |

**Stale 배지 케이스**

| 원인 | 배지 메시지 | 배지 색조 | 권장 액션 |
| --- | --- | --- | --- |
