## 1. 개요

### 1.1 목적

변리사가 PatSol 명세서 작성 페이지에서 **챗 패널을 통해 AI와 대화하며** 명세서 각 섹션(기술분야, 배경기술, 해결과제, 해결수단, 효과, 실시예 등)의 초안을 생성·검토·수정한다. AI는 명세서 전체를 컨텍스트로 유지하여 섹션 간 기재요건 충족 여부를 통합적으로 지원한다.

### 1.2 배경

- 명세서는 각 섹션이 독립적으로 잘 쓰인다고 기재요건이 충족되지 않음. 배경기술에서 제기한 문제 → 해결과제 → 해결수단(청구항 연결) → 실시예(구성요소 전부 뒷받침)의 연쇄가 일관되어야 함
- 섹션별 인라인 AI 보조 방식은 이 연쇄를 확인할 수 없고, 섹션마다 생성 근거가 분산되어 변리사가 전체 맥락을 파악하기 어려움
- PatSol은 청구항·선행기술 컨텍스트를 이미 보유한 vertical AI이므로, 변리사는 도메인 배경을 매번 설명할 필요 없이 자연어 지시만으로 명세서를 다듬을 수 있어야 함
- PatSol 핵심 포지셔닝("검색 → 청구항 → 명세서 연결")의 최종 구간

### 1.3 AI 역할 한계 (명시적 원칙)

> AI는 **초안 생성 보조 도구**이며, 기재요건 충족 여부 판단·법적 표현의 최종 결정은 변리사에게 있다.

- 명세서 편집 영역은 변리사가 직접 소유·편집하는 주 공간이며, 챗 패널은 보조 인터페이스
- AI 응답의 수정 제안은 변리사가 명시적으로 반영 액션을 취해야만 명세서에 적용됨
- AI는 제안형 표현을 사용하며 법적 결론을 단정하지 않음

## 2. 용어 정의

### 2.1 포함 관계

```
명세서 작성 페이지 (/document)
├── 명세서 편집 영역  ← 변리사의 주 편집 공간
│   ├── 섹션 블록 × N
│   │   ├── 섹션 텍스트 (AI 생성 또는 수동)
│   │   ├── AI 생성 배지  (AI가 생성한 섹션에 표시)
│   │   └── 섹션 생성 버튼
│   └── 드래그 선택 핸들  → 챗 연동 트리거
└── 챗 패널  ← AI 어시스턴트 인터페이스
    ├── 대화 메시지 목록
    │   ├── 사용자 메시지 (직접 입력 / 드래그 인용 포함)
    │   └── AI 응답
    │       ├── 응답 텍스트
    │       ├── 섹션 앵커 배지  (해당 섹션으로 이동)
    │       └── 반영 버튼  (수정 제안 → 명세서 적용)
    └── 메시지 입력창
```

### 2.2 용어 정의

| 용어 | 정의 |
| --- | --- |
| **챗 패널** | 명세서 전체를 컨텍스트로 유지하는 AI 대화 인터페이스. 생성 요청·질문·수정 지시·기재요건 확인을 단일 채널로 처리 |
| **드래그-투-ask** | 명세서 편집 영역에서 텍스트를 선택한 후 챗 패널에 인용하여 AI에게 질문하는 인터랙션 패턴 |
| **섹션 앵커 배지** | 챗 응답이 명세서 특정 섹션·위치를 참조할 때 표시하는 클릭 가능한 링크 배지. 클릭 시 편집 영역 해당 위치로 이동 |
| **반영 버튼** | 챗 응답의 수정 제안 텍스트를 명세서 해당 섹션에 적용하는 액션 버튼 |
| **AI 생성 배지** | 명세서 섹션이 AI에 의해 생성되었음을 표시하는 UI 표식. 변리사가 직접 수정 시 "수동 수정됨"으로 전환 |
| **기재요건 플래그** | 섹션 간 일관성 문제(용어 불일치, 실시예 미뒷받침 등)를 챗 패널에 자동 고지하는 알림 |
| **컨텍스트 패키지** | LLM에 전달하는 입력 데이터 묶음. 청구항 + 현재 명세서 전체 + 선행기술 + 발명서 + 스타일 설정 포함 |
| **수동 수정 보호** | 변리사가 직접 편집한 섹션을 이후 AI 생성·반영이 덮어쓰지 않는 정책 |
| **사무소 스타일** | 특허사무소별 어체·항목 순서·표현 방식 설정값 |
| **기재요건** | 청구항과 명세서 본문 간 용어 일관성, 실시예의 청구항 구성요소 전부 뒷받침 등 법적 충족 조건 |

## 3. 정책 및 규칙

| # | 정책 | 내용 |
| --- | --- | --- |
| P-1 | AI 생성 배지 표시 | AI가 생성한 섹션에는 AI 생성 배지를 표시한다. 변리사가 직접 수정하면 "수동 수정됨"으로 전환된다 |
| P-2 | 수동 수정 보호 | 변리사가 수동 수정한 섹션은 챗에서 반영 버튼을 눌러도 덮어쓰지 않는다. 적용하려면 변리사가 보호 해제를 명시적으로 확인해야 한다 |
| P-3 | 생성 비활성화 조건 | 청구항이 1개 이상 저장되지 않은 상태에서는 섹션 생성 기능과 챗 패널의 생성 명령이 disabled 처리된다 |
| P-4 | 법적 단정 표현 금지 | AI 응답에 "신규성이 있다", "등록 가능하다" 등 법적 결론을 단정하는 표현을 포함하지 않는다 |
| P-5 | 컨텍스트 부족 고지 | 청구항 미입력 등 필수 컨텍스트 누락 시 생성을 중단하고 누락 항목을 챗 패널에 안내한다 |
| P-6 | 사무소 스타일 우선 | 사무소 스타일 설정이 존재하는 경우 기본 프롬프트보다 우선 적용한다 |
| P-7 | 챗 컨텍스트 동기화 | 명세서 편집 영역의 내용이 변경되면 챗 패널의 명세서 상태가 자동으로 갱신된다. 챗 패널은 항상 최신 명세서 전체를 컨텍스트로 유지한다 |
| P-8 | 섹션 앵커 배지 포함 | 챗 응답이 명세서의 특정 섹션 또는 텍스트를 참조할 때는 반드시 섹션 앵커 배지를 포함한다 |
| P-9 | 생성 요약 챗 출력 | 섹션 생성(단일·일괄) 완료 시 챗 패널에 고려사항 요약을 자동으로 출력한다 (어떤 청구항을 중심으로 썼는지, 컨텍스트 부족 항목은 무엇인지 등) |

## 4. AI 동작 정의

### 4.1 입력 컨텍스트 구성

챗 패널은 명세서 전체를 컨텍스트로 유지한다. 단일 섹션만 넘기는 것이 아니라, 현재까지 작성된 명세서 전체 + 청구항 구조 + 선행기술을 통합하여 섹션 간 관계를 파악한 응답이 가능하다.

#### 컨텍스트 패키지 구성 항목

| 항목 | 필수 여부 | 설명 |
| --- | --- | --- |
| 청구항 목록 | 필수 | 독립항·종속항 전체 텍스트 및 계층 구조 |
| 현재 명세서 전체 텍스트 | 필수 | 작성 완료·미작성 포함한 모든 섹션 현재 상태 |
| 발명 요약 (직무발명서) | 권장 | 발명의 배경·목적·효과 원문 |
| 선행기술 조사 결과 | 권장 | 매칭된 선행특허 리스트 및 비교발명 분석 요약 |
| 사무소 스타일 설정 | 선택 | 어체·항목 순서·특정 표현 규칙 |
| 드래그 인용 텍스트 | 상황별 | 드래그-투-ask 사용 시 선택된 텍스트 + 섹션 ID |
| 수동 수정 보호 섹션 목록 | 자동 | 반영 버튼 적용 시 스킵할 섹션 ID 목록 |

#### 컨텍스트 트런케이션 정책

- TBD: 명세서 전체 포함 시 토큰 한도 초과 대응 방안 (요약 압축 vs. 해당 섹션 + 인접 섹션만 포함)
- 우선순위 기본값: 청구항 > 드래그 인용 텍스트 > 현재 명세서 > 발명 요약 > 선행기술

### 4.2 출력 스펙

#### 생성 대상 섹션

| 섹션 | 한국 출원 기준 항목명 |
| --- | --- |
| 기술분야 | 발명이 속하는 기술 분야 |
| 배경기술 | 종래 기술 및 문제점 |
| 해결과제 | 발명이 이루고자 하는 기술적 과제 |
| 해결수단 | 과제 해결을 위한 수단 (청구항 재기술 포함) |
| 발명의 효과 | 발명의 효과 |
| 실시예 | 발명의 실시를 위한 구체적 내용 |
| 도면 설명 | 도면의 간단한 설명 (도면 있는 경우) |

#### 출력 포맷 기준

- 섹션 생성 결과는 명세서 편집 영역에 직접 반영하고, 챗 패널에 생성 요약을 동시 출력 (→ P-9)
- 수정 제안은 챗 패널에 텍스트로 출력 + 반영 버튼 제공. 변리사가 반영 버튼을 눌러야만 명세서에 적용됨
- 청구항에서 사용된 구성요소 명칭을 그대로 사용 (임의 동의어 치환 금지)
- 실시예는 독립항의 모든 구성요소를 최소 1회 이상 포함
- TBD: 섹션별 출력 길이 기준

### 4.3 프롬프트 설계 원칙

- **변리사 주체성 유지**: "~로 판단됩니다", "~를 고려하여 작성하였습니다" 등 제안형 어미 사용. 변리사의 최종 판단을 전제하는 표현 유지
- **청구항 용어 준수**: 입력된 청구항 텍스트의 구성요소 명칭을 임의 변경하지 않음
- **섹션 간 일관성 지시**: 생성·수정 시 전체 명세서 컨텍스트를 참조하여 다른 섹션과의 논리적 연결을 유지하도록 지시
- **사무소 스타일 주입**: 스타일 설정이 있는 경우 시스템 프롬프트 최상단에 삽입 (→ P-6)
- **할루시네이션 억제**: 입력 컨텍스트에 없는 기술적 사실 생성 금지 지시 (TBD: 구체적 인스트럭션 문구)

### 4.4 폴백 및 불확실성 처리

| 상황 | 동작 |
| --- | --- |
| 청구항 미입력 | 챗 패널과 섹션 생성 버튼 모두 disabled. 챗 패널에 이유 안내 (→ P-3, P-5) |
| 발명 요약 누락 | 청구항만으로 생성 진행. 챗 패널 생성 요약에 "발명 요약 없음 — 구체성 보완 권장" 표시 |
| 선행기술 누락 | 배경기술 생성 후 챗에 "선행기술 미연결 — 배경기술 구체화에 한계 있음" 플래그 출력 |
| 생성 실패 (API 오류) | 챗 패널에 오류 메시지 + 재시도 버튼 표시 |
| 토큰 한도 초과 | TBD: 트런케이션 적용 후 챗 패널에 "일부 컨텍스트가 축소되었습니다" 안내 |

## 6. 기능 상세

### 6-1. 챗 패널 (IA TBD)

**위치:** document 페이지 > 우측 사이드 패널 (TBD: 위치·토글 방식)

**설명:** 명세서 전체를 컨텍스트로 유지하는 AI 대화 인터페이스. 생성 요청·질문·수정 지시·기재요건 확인을 단일 채널로 처리한다.

#### 구성 요소

| 요소 | 설명 |
| --- | --- |
| 대화 메시지 목록 | 사용자·AI 메시지의 시간순 스크롤 목록 |
| AI 응답 버블 | 응답 텍스트 + 섹션 앵커 배지(해당 시) + 반영 버튼(해당 시) |
| 메시지 입력창 | 자유 텍스트 입력. 드래그 인용 시 인용 블록이 자동 삽입됨 |
| 컨텍스트 상태 표시 | 현재 챗이 참조 중인 명세서 버전 및 누락 컨텍스트 항목 표시 (TBD) |

#### 동작 흐름

```
변리사: 메시지 입력 후 전송
    ↓
컨텍스트 패키지 구성 (청구항 + 현재 명세서 전체 + ...) (→ 4.1)
    ↓
LLM 호출
    ↓
응답 텍스트 스트리밍 출력
    ↓
섹션 참조 감지 시 → 섹션 앵커 배지 삽입 (→ 6-5)
수정 제안 감지 시 → 반영 버튼 삽입 (→ 6-4)
기재요건 이슈 감지 시 → 플래그 메시지 출력 (→ 6-6)
```

#### 상태 조건

| 조건 | 상태 |
| --- | --- |
| 청구항 1개 이상 저장됨 | 생성 요청 enabled |
| 청구항 없음 | 생성 요청 disabled. 질문·기재요건 확인은 enabled |
| 응답 생성 중 | 입력창 disabled (전송 불가) |

### 6-2. 드래그-투-ask (IA TBD)

**위치:** document 페이지 > 명세서 편집 영역

**설명:** 명세서 텍스트를 드래그하여 선택하면 챗 패널에 해당 텍스트를 인용한 상태로 질문을 입력할 수 있다. AI는 선택된 텍스트의 위치·섹션 정보를 컨텍스트로 포함하여 응답한다.

#### 구성 요소

| 요소 | 설명 |
| --- | --- |
| 드래그 액션 버튼 | 텍스트 선택 시 선택 영역 근처에 플로팅으로 표시. "AI에게 묻기" (TBD: 아이콘·레이블) |
| 챗 인용 블록 | 선택 텍스트 + 섹션명이 입력창 상단에 인용 형태로 삽입됨 |

#### 동작 흐름

```
변리사: 명세서 텍스트 드래그 선택
    ↓
플로팅 액션 버튼 표시
    ↓
변리사: "AI에게 묻기" 클릭
    ↓
챗 패널 포커스 이동
선택 텍스트 + 섹션 ID → 입력창 상단에 인용 블록 삽입
    ↓
변리사: 질문 입력 후 전송
    ↓
LLM 호출 (인용 텍스트 + 섹션 위치 + 전체 명세서 컨텍스트 포함)
    ↓
챗 패널에 응답 출력 (섹션 앵커 배지 포함)
```

### 6-3. 명세서 섹션 생성 (IA TBD)

**위치:** 트리거 진입점 2곳 — 명세서 편집 영역 섹션 헤더 / 챗 패널 메시지

**설명:** 단일 섹션 또는 전체 명세서를 생성한다. 어디서 트리거하든 결과는 명세서 편집 영역에 반영되고, 생성 요약이 챗 패널에 자동 출력된다. (→ P-9)

#### 구성 요소

| 요소 | 설명 |
| --- | --- |
| 섹션 생성 버튼 | 각 섹션 헤더 우측 배치. 청구항 없을 시 disabled |
| 전체 생성 버튼 | 편집 영역 상단 툴바. 청구항 없을 시 disabled |
| 생성 진행 표시 | 일괄 생성 시 섹션별 순차 완료 상태 표시 |
| 챗 생성 요약 | 생성 완료 후 챗 패널에 자동 출력되는 고려사항 요약 (→ P-9) |

#### 동작 흐름 — 단일 섹션

```
트리거: 섹션 생성 버튼 클릭 또는 챗에서 "○○ 섹션 작성해줘"
    ↓
청구항 존재 여부 확인 (→ P-3)
    ↓ (없음)
챗 패널에 안내 메시지 출력 (→ P-5)
    ↓ (있음)
컨텍스트 패키지 구성 (→ 4.1)
    ↓
LLM 호출 → 섹션 텍스트 생성
    ↓
명세서 편집 영역 해당 섹션에 반영 + AI 생성 배지 표시 (→ P-1)
    ↓
챗 패널에 생성 요약 자동 출력 (→ P-9)
```

#### 동작 흐름 — 전체 일괄 생성

```
트리거: 전체 생성 버튼 클릭
    ↓
수동 수정 보호 섹션 확인 (→ P-2)
    ↓
보호 섹션 스킵 안내 + 계속 여부 확인
    ↓
섹션 순서대로 순차 LLM 호출
(기술분야 → 배경기술 → 해결과제 → 해결수단 → 효과 → 실시예)
    ↓
각 섹션 완료 시 즉시 반영
    ↓
전체 완료 후 챗 패널에 통합 생성 요약 출력 (→ P-9)
```

#### 상태 조건

| 조건 | 상태 |
| --- | --- |
| 청구항 1개 이상 저장됨 | enabled |
| 청구항 없음 | disabled |
| 생성 진행 중 | disabled (중복 실행 방지) |
| 수동 수정 보호 섹션 (단일 생성 시) | 보호 해제 확인 후 진행 |

### 6-4. 챗 응답 반영 (IA TBD)

**위치:** document 페이지 > 챗 패널 > AI 응답 버블

**설명:** AI가 챗에서 특정 섹션의 수정 텍스트를 제안할 때 반영 버튼이 표시된다. 변리사가 반영 버튼을 클릭하면 해당 텍스트가 명세서 편집 영역의 지정 섹션에 적용된다.

#### 구성 요소

| 요소 | 설명 |
| --- | --- |
| 반영 버튼 | AI 응답 버블 하단 배치. 수정 제안이 포함된 응답에만 표시 |
| 적용 대상 섹션 표시 | 반영 버튼 옆에 "[배경기술에 반영]" 형태로 대상 명시 |

#### 동작 흐름

```
변리사: 챗 응답 확인 후 반영 버튼 클릭
    ↓
수동 수정 보호 여부 확인 (→ P-2)
    ↓ (보호 있음)
"수동 수정 내용을 덮어씁니다" 확인 다이얼로그
    ↓ (확인 / 보호 없음)
명세서 편집 영역 해당 섹션에 적용
AI 생성 배지 유지 (→ P-1)
```

### 6-5. 섹션 앵커 배지 (IA TBD)

**위치:** document 페이지 > 챗 패널 > AI 응답 버블 내

**설명:** 챗 AI 응답이 명세서의 특정 섹션 또는 텍스트를 참조할 때 해당 위치로 이동할 수 있는 앵커 배지를 응답 내에 삽입한다. 챗 대화와 명세서 편집 영역 간의 시각적 연결 고리 역할을 한다.

#### 구성 요소

| 요소 | 설명 |
| --- | --- |
| 섹션 앵커 배지 | "[배경기술]", "[실시예 §2]" 형태의 클릭 가능한 인라인 배지 |

#### 동작 흐름

```
AI 응답 생성 중 섹션 참조 감지 (→ P-8)
    ↓
해당 위치에 섹션 앵커 배지 삽입
    ↓
변리사: 배지 클릭
    ↓
명세서 편집 영역의 해당 섹션으로 스크롤 이동 + 일시 하이라이트
```

### 6-6. 기재요건 자동 플래그 (IA TBD)

**위치:** document 페이지 > 챗 패널 (시스템 메시지 형태)

**설명:** 섹션 생성 또는 반영 후 AI가 명세서 전체를 검토하여 기재요건 충족 여부를 확인하고, 문제가 감지되면 챗 패널에 플래그 메시지를 자동 출력한다.

#### 검증 항목

| 항목 | 설명 |
| --- | --- |
| 실시예 뒷받침 | 독립항의 모든 구성요소가 실시예에 1회 이상 포함되는지 확인 |
| 용어 일관성 | 청구항 구성요소 명칭과 명세서 전체 텍스트 간 불일치 탐지 |
| 섹션 간 논리 연결 | 배경기술 문제점 → 해결과제 → 해결수단 간 논리적 흐름 확인 |

#### 구성 요소

| 요소 | 설명 |
| --- | --- |
| 플래그 메시지 | 챗 패널 내 시스템 메시지 스타일. 이슈 항목 목록 + 섹션 앵커 배지 포함 |

#### 동작 흐름

```
트리거: 섹션 생성 완료 / 반영 버튼 적용 완료 / 수동 트리거 (TBD)
    ↓
명세서 전체 텍스트 + 청구항 기준으로 검증 항목 확인
    ↓ (이슈 없음)
별도 출력 없음
    ↓ (이슈 있음)
챗 패널에 플래그 메시지 출력
  예) "실시예에서 독립항 1항의 구성요소 'X 모듈'이 명시되지 않았습니다.
       [실시예 이동 →]"
```
