> **1차 타깃 사용자 범위**: 명세서 작성 업무를 수행하는 **특허법인 소속 개인 변리사**를 핵심 페르소나로 한정한다. R&D 실무자는 보조 페르소나로 별도 기술한다.

---

## 1. 페르소나

### Persona A — 변리사 (핵심 페르소나)

| 속성 | 내용 |
|------|------|
| 직군 | 특허법인 소속 변리사 (어소시에이트 ~ 파트너) |
| 핵심 업무 | 의뢰인의 발명을 기반으로 특허 명세서(청구항·발명의 설명) 작성 |
| 검색을 하는 이유 | 명세서 작성 전·중에 선행기술 조사를 수행하여 청구항 범위를 설계하기 위해 |
| 검색 빈도 | 건당 최소 1회 이상, 주요 사건은 수 차례 반복 조사 |
| 도구 숙련도 | 특허 검색 전문가 수준 — 검색식(Boolean·CPC) 작성 가능, 윈텔윕스 검색식 사용에 매우 익숙, 키프리스/키워트 DB 경험 있음 |
| 진입 조건 | 변리사 자격증 업로드 시 AI 명세서 기능 전체 이용 가능 |
| 플랜 | 입문형(10만/월) · 기본형(30만/월) · 고급형(50만/월) — 청구항 생성 횟수 차등 |
| 작업 단위 | 고객 케이스 단위로 반복 (건별 15~50분) |

**특징적 행동 패턴:**
- 고객으로부터 불완전한 소재(아이디어 메모, 청구항만, 직무발명서, 선행특허 목록)를 받아 작업 시작
- 비교 발명을 최대 3~5건 선택해 선행기술조사 수행
- 회피 설계 결과로 생성된 청구항을 검토·수정 후 명세서 초안 생성
- 재검색(Re-search)으로 청구항 → 검색 → 청구항 사이클을 필요에 따라 반복

### Persona B — R&D 실무자 (보조 페르소나)

| 속성 | 내용 |
|------|------|
| 직군 | 기업 내부 연구개발 담당자 |
| 핵심 업무 | 발명 아이디어 탐색, 기술 트렌드 파악, 경쟁사 특허 분석 |
| 진입 조건 | 일반 회원가입으로 검색·분석 기능 이용 가능 (AI 명세서 기능은 변리사 전용) |
| 플랜 | 모든 사용자(30만/월) |
| 작업 단위 | 프로젝트 탐색 단위, 비정기적 |

**특징적 행동 패턴:**
- 기술 트렌드 키워드나 발명 아이디어 메모로 자연어 검색 시작
- OS 매트릭스로 경쟁사 특허 공백 기술 시각화
- 검색 결과를 Excel로 다운로드해 내부 보고 자료로 활용
- 분석 결과를 변리사에게 전달 후 출원 의뢰

## 2. 명세서 작성 흐름 속 검색의 위치

변리사의 명세서 작성은 아래 흐름으로 진행된다. 검색은 단일 이벤트가 아니라 **반복적으로 끼어드는 업무**다.

```
발명 상담·OA 접수
       │
       ▼
[선행기술 조사 ①] ← 청구항 초안 설계 전 — 넓은 범위 탐색
       │
       ▼
청구항 초안 작성
       │
       ▼
[선행기술 조사 ②] ← 특정 구성요소 신규성·진보성 확인 — 좁은 범위 정밀 검색
       │
       ▼
명세서 본문 작성
       │
       ▼
[선행기술 조사 ③] ← OA 대응 또는 보정 시 추가 확인 (필요 시)
       │
       ▼
출원·제출
```

각 단계별로 검색 목적과 요구 정밀도가 다르기 때문에, 검색 도구는 **탐색형(①)과 정밀형(②③)을 모두 지원**해야 한다.

---

## 3. 검색 입력 방식별 핵심 목적

### 3-A. 검색식 입력 (정밀 모드)

변리사는 체계적 조사가 필요한 경우 직접 Boolean 검색식 또는 CPC 코드를 입력한다.

**핵심 목적: "내가 설계한 조건 그대로 처리하고, 빠짐없이 가져와라."**

- **정밀도(precision)** 와 **재현 가능성(reproducibility)** 이 최우선 — 동일 식으로 나중에 재실행해도 같은 결과셋이 나와야 한다
- LLM이 검색식을 임의로 변형하는 것을 **극도로 불신**한다
- 오류가 있을 경우, 조용히 수정하는 것보다 **명시적으로 보여주고 승인을 받는 것**을 선호한다

**결과에서 기대하는 것:**
1. 검색식이 **그대로 실행되었다는 확인** — 수정이 있었다면 before/after를 나란히 표시
2. **히트 건수와 조건별 분포** — 어떤 항(AND/OR/NOT, CPC)이 결과를 어떻게 좁혔는지
3. **검색식 저장·재실행** 기능 — 조사 이력 관리와 보고서 첨부용
4. 결과 목록에서 **청구항 상태(등록·소멸·심사중)** 즉시 확인

**spec 연결:** Query Builder Assistant(How 3) — 오류를 조용히 수정하지 않고 원문과 수정안을 병렬 표시, 변리사 승인 후 적용

---

### 3-B. 발명 자유 서술 입력 (탐색 모드)

의뢰인의 발명 설명을 받은 직후, 기술 영역의 전체 지형을 빠르게 파악할 때 사용한다.

**핵심 목적: "이 기술 영역에 어떤 선행기술이 깔려 있는지 빠르게 조감하고 싶다."**

- 초기 탐색이므로 **재현율(recall)** 우선 — 좁게 걸러서 놓치는 것이 더 위험하다
- 결과를 보고 청구항의 차별화 포인트를 **스스로 도출**하는 것이 목적

**결과에서 기대하는 것:**
1. 기술 개념 레벨의 유사 특허 목록 — 단순 키워드 매칭이 아닌 의미 기반 유사도
2. **어떤 구성요소가 겹치는지** 근거 제시 — 명세서 차별화 포인트 설계에 바로 활용
3. **선행기술 출원일 분포** — 어느 시점부터 기술이 집중 출원됐는지 타임라인
4. **화이트스페이스 힌트** — 아직 청구되지 않은 구성요소 조합 제안
5. 탐색 결과를 기반으로 **정밀 검색식으로 전환** 제안 (Post-Search Refinement)

**spec 연결:** 3-Way Sufficiency(How 2) — 발명 서술이 충분히 구체적이면 Vector DB Top50으로 직행, 범위가 너무 넓으면 CPC 코드/키워드 필터 제안으로 Q&A 루프 제거

---

### 3-C. 문서 업로드 (발명신고서·기존 특허)

의뢰인이 제출한 발명신고서, 또는 회피 설계 기준이 되는 경쟁사 특허를 그대로 올리는 경우다.

**핵심 목적: "이 문서와 기술적으로 얼마나 가까운 특허들이 존재하는가?"**

문서 자체가 이미 구체적인 기술 컨텍스트를 담고 있으므로, 변리사는 쿼리를 별도로 작성하지 않고 **문서 전체를 검색 입력으로** 쓰고 싶어 한다.

**결과에서 기대하는 것:**
1. **유사도 기반 정렬된 특허 목록** — 청구항 단위로 어느 항이 가장 근접한지 대응 표시
2. **특허 패밀리·국가 확장** — 동일 발명의 PCT/해외 패밀리 포함 여부 (회피 설계 범위 확인)
3. **출원일 선후관계** — 업로드 문서 대비 선행기술인지 후행 특허인지
4. **청구항 생존 상태** — 등록·소멸·심사중 여부 (회피 필요성 판단)

**spec 연결:** Intent Classifier(How 1) — 문서 임베딩 직접 벡터 검색 경로 별도 필요. 텍스트 flatten 시 문서 구조 정보 소실 주의

---

### 입력 방식 비교

| 구분 | 검색식 | 자유 서술 | 문서 업로드 |
|------|--------|-----------|------------|
| 주요 발생 시점 | 조사 ②③ (정밀) | 조사 ① (탐색) | 조사 ① 또는 회피 설계 시 |
| 결과 우선 지표 | 정밀도 + 재현성 | 재현율 + 커버리지 | 유사도 + 청구항 대응 근거 |
| LLM 개입 허용도 | 낮음 — 승인 후 수정만 허용 | 높음 — 범위·필터 제안 환영 | 중간 — 임베딩은 위임, 결과 해석은 직접 |
| spec 연결 경로 | Query Builder Assistant (How 3) | 3-Way Sufficiency (How 2) | Intent Classifier 신규 경로 필요 |

## 4. 유저 저니 (소재 유형별)

팟솔 작업은 고객이 제공하는 소재 유형에 따라 4가지 진입 경로로 분기된다.

```
소재 유형
├── 아이디어만 있음              → Journey 1  (자유 서술 입력)
├── 문서/청구범위 있음           → Journey 2  (자유 서술 or 문서 업로드)
├── 선행특허/기술동향 있음       → Journey 3  (OS 매트릭스 경유)
└── 도면/설명서 있음            → Journey 4  (문서 업로드)
```

---

### Journey 1 — 아이디어 기반

**적용 상황:** 고객이 아이디어 메모·구두 설명만 전달 / 발명 미완성 / 부분 발명 설명만 존재
**검색 입력 방식:** 3-B 자유 서술 → 3-A 검색식(재검색) 순서로 전환
**예상 소요 시간:** 20~40분
**결과물:** 청구항 세트 + AI 명세서 초안

```
1. 로그인
2. 특허 검색 — 아이디어/키워드를 자유 서술로 입력  ← 3-B 탐색 모드
3. 검색 실행
4. 검색 결과에서 비교 발명 선택 (최대 3~5건)
5. 비교 발명 분석하기 → 선행기술조사 (1차)
6. 회피 설계 → 청구항 생성 (1차)
   ├── [만족] → Step 8
   └── [불만족] → Step 7 재검색
7. 재검색 — 생성된 청구항이 검색창에 자동 입력    ← 3-A 정밀 모드 전환
   → 비교 발명 선택 → 선행기술조사 (2차) → 청구항 생성 (2차)
8. 최종 청구항 확인 및 수정
9. 명세서 초안 생성
10. AI 명세서 초안 확인 및 수정
```

---

### Journey 2 — 문서·청구 범위 기반

**적용 상황:** 청구항만 받은 경우 / 직무발명서·기술 매뉴얼·논문을 받은 경우

#### 2A — 청구 범위만 받은 경우

**검색 입력 방식:** 3-B 자유 서술
**예상 소요 시간:** 15~30분

```
1. 로그인
2. 특허 검색 → 청구 범위 내용을 자연어로 입력
3. 검색 실행
4. → Journey 2 공통 작업
```

#### 2B — 직무발명서 / 매뉴얼 / 논문 기반

**검색 입력 방식:** 3-C 문서 업로드
**예상 소요 시간:** 30~50분

```
1. 로그인
2. 특허 검색 → 발명 문서 업로드 버튼 클릭
3. 파일 업로드 (PDF, DOC, DOCX, PPT, PPTX, HWP 등)
4. 업로드 완료 → 문서 내용이 검색창에 자동 표시
   └── 문서 내 도면이 있는 경우 하단에 도면 자동 표시
5. 검색 실행
6. → Journey 2 공통 작업
```

#### Journey 2 공통 작업

```
(공통) 1. 검색 결과에서 비교 발명 선택
(공통) 2. 비교 발명 분석하기 → 선행기술조사
(공통) 3. 회피 설계 → 청구항 생성 (종속항 포함)
(공통) 4. 명세서 초안 생성
(공통) 5. AI 명세서 초안 확인 및 수정
```

---

### Journey 3 — 선행특허·기술동향 기반

**적용 상황:** 경쟁사 유사 선행특허를 고객이 제공한 경우 / 기술 트렌드 자료만 받은 경우

#### 3A — 비슷한 선행특허가 많은 경우 (6건 이상)

**예상 소요 시간:** 30~40분

```
1. 로그인
2. 특허 분석 > OS 매트릭스 → 엑셀 양식 다운로드
3. 팟솔 검색 결과 또는 타사 특허검색 데이터로 엑셀 작성
4. 엑셀 파일 업로드
5. OS 매트릭스 생성
6. 공백 기술 확인 (숫자 적거나 없는 셀 = 공백 기술)
7. 특허 검색 → 공백 기술의 목적·수단 입력 → 검색   ← 3-B 자유 서술
8. 비교 발명 선택 → 비교 발명 분석하기
9. → Journey 3 공통 작업
```

#### 3B — 선행특허가 5건 이하인 경우

**예상 소요 시간:** 20~40분

```
1. 로그인
2. 특허 검색 → 발명 요지 또는 청구 범위로 검색   ← 3-B 자유 서술
3. 출원번호로 특허 직접 추가 (비교 발명 추가란)   ← 3-A 검색식 방식 활용
4. 선택 특허 AI 분석하기 → 비교 발명 분석
5. → Journey 3 공통 작업
```

#### 3C — 기술 트렌드만 받은 경우 (R&D 실무자 주요 저니)

**예상 소요 시간:** 20~40분

```
1. 로그인
2. 특허 검색 → 기술 트렌드 키워드 입력          ← 3-B 자유 서술
3. 검색 결과 Excel로 다운로드
4. 특허 분석 > OS 매트릭스 → 다운로드 파일 업로드
5. OS 매트릭스 생성 → 공백 기술 확인
6. 공백 기술 목적·수단으로 재검색
7. 비교 발명 선택 → 비교 발명 분석하기
8. → Journey 3 공통 작업
```

#### Journey 3 공통 작업

```
(공통) 1. 회피 설계 → 청구항 생성
(공통) 2. 명세서 초안 생성
(공통) 3. AI 명세서 초안 확인 및 수정
```


## 6. 핵심 관찰

**변리사의 모든 검색은 명세서 작성이라는 단일 목적에 귀결된다.** 검색 결과를 보는 행위 자체가 목적이 아니라, 결과를 해석하여 청구항 범위를 확정하고 명세서 본문을 작성하는 것이 최종 아웃풋이다. 이 때문에 세 가지 입력 방식 모두 **"결과 → 다음 명세서 작업"으로 이어지는 연결 동선**이 필수적이다.

- 탐색 결과는 → 청구항 차별화 포인트 메모 또는 정밀 검색식 전환으로
- 정밀 검색 결과는 → 선행기술 리스트 저장 및 명세서 선행기술란 초안으로
- 문서 비교 결과는 → 회피 설계 방향 메모 또는 청구항 보정 근거로

spec의 How 4(Post-Search Refinement)와 How 5(세션 컨텍스트)가 이 연결 동선을 담당해야 하며, 특히 **검색 세션과 명세서 작업 세션 간 컨텍스트 전달**이 변리사 워크플로우의 핵심 마찰 지점이다.
