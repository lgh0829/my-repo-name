## Slide 1 — 왜 지금 이 기능인가

**PatSol의 마지막 미완성 구간**

```
검색  ──────────────  완료
청구항 구조화  ───────  완료
명세서 작성  ─────────  ← 지금 여기
```

변리사의 실제 문제:
- 섹션마다 따로 AI를 쓰면 기재요건 연쇄를 확인할 수 없음
  - 배경기술 문제점 → 해결과제 → 실시예 뒷받침이 일관되어야 등록 가능
- 선행기술 조사 결과가 명세서 초안에 자동으로 반영되지 않음
- 초안 생성은 빠르지만, 사무소 형식·기재요건 맞추는 후처리에 시간이 집중됨

> **이 기능은 PatSol 포지셔닝("검색 → 청구항 → 명세서 연결")의 최종 구간을 채운다.**

## Slide 2 — 기획 배경

**왜 AI 채팅 패널인가**

| 문제점 | 해결 전략 | 수단 |
|---|---|---|
| 변리사가 최종 결정권자 — AI가 근거 없이 결과만 제시하면 변리사가 판단 자체를 할 수 없다 | 생성·수정 제안마다 기재요건 기준의 근거를 함께 출력 | AI 에이전트 + 채팅 패널 |
| 기재요건 검토 도구가 청구항별·명세서 전체·도면 단위로 분산 구현 — 사용자가 맥락에 맞는 도구를 매번 직접 찾아 실행해야 한다 | 사용자가 챗으로 의도를 전달하면 에이전트가 적절한 도구를 자동 선택·호출 | ↑ |
| 섹션 단위 완성도만으로 불충분 — 섹션 간 불일치도 기재불비 | 명세서 전체를 컨텍스트로 유지 → 에이전트가 일관성 유지 기준을 판단 → 기재요건 검토 도구 호출 | ↑ |

## Slide 3 — 한 줄 정의

**"변리사가 챗으로 AI에게 명세서 작성·검토·수정을 지시한다"**

명세서 전체를 컨텍스트로 유지하는 AI 에이전트가 챗 패널에 상주.
변리사의 메시지 의도를 파악해 적절한 도구를 선택·실행하고 결과를 돌려준다.

**주요 원칙:**
- AI는 초안 생성 보조 도구. 기재요건 충족 판단·법적 결론은 변리사가 최종 결정
- 도구 실행 결과는 변리사가 명시적으로 [반영]을 눌러야만 명세서에 적용됨
- AI 응답에 "등록 가능하다", "신규성이 있다" 등 법적 단정 표현 금지

## Slide 6 — 구현 단계

### Phase 1 — AI 코파일럿 챗 기본

**범위:** T-07(선행기술 조회) + T-08(청구항 구조 조회) + 의도 분류 + 챗 히스토리
**명세서 편집 영역은 건드리지 않음**

```
데모 시나리오:
"이 청구항에서 독립항 구조 설명해줘"
  → AI 작업 내역 표시 (⟳ 청구항 구조 분석 중...)
  → 텍스트 응답 출력
```

---

### Phase 1.5 — 기재요건 검토 연동

**범위:** T-03~T-05(기존 기재요건 기능 챗 호출) + 섹션 앵커 배지
**전제:** 기존 기재요건 검토 기능 완료 필요

```
데모 시나리오:
"기재요건 검토해줘"
  → ⚠ 실시예 뒷받침 검토 — 문제 1건
  → "독립항 1항의 구성요소 'X 모듈'이 실시예에 없습니다. [실시예로 이동 →]"
```

---

### Phase 2 — 에이전트 고도화 (문서 변경)

**범위:** T-01, T-02, T-06, T-09 + 수정 전후 비교 UI + 드래그 인용

```
데모 시나리오:
"배경기술 다시 써줘"
  → 수정 전후 비교 출력:
      - 종래 기술에서 알려진 X 장치는...
      + 종래 기술로는 Y 방법이 알려져 있으나...
  → 변리사: [반영] 클릭 → 명세서에 적용
```

## Slide 7 — 주요 UX 컴포넌트 3가지

*(Figma 참조: https://www.figma.com/design/XD5TxmzMB68uXpKRmrgHpt)*

**① AI 작업 내역** — 에이전트가 무엇을 하는지 단계별로 보임
```
▾ 에이전트 작업 내역
  ✓ 의도 파악 — 기재요건 검토 (명세서 전체)
  ✓ 청구항 구조 확인
  ⚠ 실시예 뒷받침 검토 — 문제 1건
  ✓ 용어 일관성 검토
```
→ 완료 후 자동 접힘. 이슈 발견 시 전면 노출. 변리사가 AI를 신뢰할 수 있는 근거

**② 수정 전후 비교** — AI 제안을 반영 전에 눈으로 확인
```
[배경기술에 반영]
  - 종래 기술에서 알려진 X 장치는...
  + 종래 기술로는 Y 방법이 알려져 있으나...
[반영]  [취소]
```
→ [반영] 클릭 전까지 명세서 변경 없음. 변리사가 제어권을 유지하는 핵심 장치

**③ 선택지 카드** — 의도가 모호하면 AI가 먼저 묻는다
```
어떤 작업을 원하시나요?
  ① 기재요건 검토
  ② 수정 제안 작성
  ③ 섹션 다시 생성
```
→ 의도하지 않은 작업 실행 방지

## Slide 9 — 오늘 합의할 것

### 기술 검토 (개발팀이 답해야 할 것)

- **에이전트 구조**: 단일 에이전트로 가능한가, 멀티 에이전트가 필요한가
- **기존 Q&A 챗봇과 공통화**: 현재 "특허 자세히 보기" 챗봇 구조 재활용 범위
- **토큰 비용 추산**: 입력 컨텍스트 조합별 예상 비용 (청구항 + 명세서 전체 + 기재불비 기준)
- **Phase 1 예상 기간**

### 기획 미결 (회의 후 업데이트)

- **OQ-7**: 토큰 한도 초과 시 컨텍스트 어떻게 줄일 것인가 (섹션 요약 vs 인접 섹션만 포함)
- **OQ-8**: AI 품질 평가 기준 — 테스트셋 구성 방법 (선행기술조사문헌 + 출원 명세서 수집 방향 논의 중)

- [ ] 기술 기획 작성 담당자 지정

## Slide 10 — 앞으로의 진행방향

**Phase 로드맵**

```
[지금] 킥오프
    ↓
[단기] POC — 에이전트 구조 검증 + 입력 컨텍스트 토큰 비용 측정
    ↓
Phase 1 — AI 코파일럿 챗 기본
  T-07·T-08, 의도 분류, 챗 히스토리
  명세서 섹션 생성(T-01·T-02), 수정 제안(T-06)
    ↓
Phase 1.5 — 기재요건 검토 연동        ← 기존 기재요건 기능 완료 전제
  T-03~T-05, 섹션 앵커 배지
    ↓
Phase 2 — 에이전트 고도화
  드래그 인용, 선호 표현 일괄 적용(T-09), 다중 도구 오케스트레이션
```

**Phase 간 전제조건**

| Phase | 착수 전제 |
|---|---|
| Phase 1 | POC에서 에이전트 구조 방향 확정 |
| Phase 1.5 | 기존 기재요건 검토 기능(T-03~T-05) 개발 완료 |
| Phase 2 | Phase 1 완료 |

**킥오프 후 단기 액션 (1~2주)**

- [ ] 정기 기술 리뷰 첫 회 일정 확정
