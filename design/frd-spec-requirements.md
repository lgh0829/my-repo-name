## 1. 개요

### 1.1 목적

**해결하려는 문제**

변리사는 명세서 작성 과정에서 두 종류의 기재요건 위반을 관리해야 한다.

| 구분 | 설명 | 해당 법조항 |
| --- | --- | --- |
| **절차적 요건** | 다항제 기재방법 준수 — 독립항·종속항 형식, 인용 번호 명시, 택일적 기재, 다중인용 재인용 금지, 인용 순서 등. 형식 규칙이므로 패턴 탐지로 확정 검출 가능하나, 청구항 수가 늘수록 수동 확인 누락이 발생한다 | §42조8항 동령5조 |
| **실체적 요건** | 발명 내용의 기재 적법성 — 청구범위의 명확·간결 기재, 발명의 설명 뒷받침, 실시가능성, 기재요소 완비 등. 일부 표현 패턴은 자동 탐지 가능하나, 대부분은 발명의 설명과의 의미론적 비교 판단이 필요하다 | §42조4항2호, §42조4항1호, §42조3항1호, §42조6항 |

두 종류 모두 현재 변리사가 수동으로 확인한다. 출원 후 심사관 거절이유통지에서 발견하거나 무효심판까지 이어질 경우 보정 비용과 권리 손실이 발생한다.

**이 시스템이 하는 일**

룰베이스 검증과 AI 검증을 조합하여 변리사가 출원 전에 기재불비를 발견·수정할 수 있도록 지원한다.

| 검증 방식 | 담당 항목 | 특징 |
| --- | --- | --- |
| **룰베이스 검증** | 패턴으로 확정 탐지 가능한 항목 | 즉시 실행, AI 호출 없음 |
| **AI 검증** | 발명의 설명과의 의미론적 비교가 필요한 항목 | LLM 호출, 버튼 클릭 시 실행 |

> 룰베이스/AI 구분은 절차적/실체적 요건과 일대일로 대응하지 않는다. 각 검증 항목의 특성에 따라 독립적으로 결정된다.

검증 결과는 청구항 / 발명의 설명 / 도면 / 전체 검토 4종 기능을 통해 변리사에게 제공된다. (→ frd-spec-ai-assistant)

### 1.2 배경

- 기재불비는 거절이유통지(§42조3항1호, §42조4항)·무효사유로 이어지며, 출원 후 발견 시 보정 비용 발생
- 기계적으로 잡을 수 있는 형식 위반(인용번호 누락, 택일적 기재 오류, 다중인용 재인용 등)도 현재는 변리사가 수동으로 확인
- AI로 판단해야 하는 내용 검증(청구범위-실시예 뒷받침 여부, 실시가능성)은 LLM 없이 자동화 불가
- 룰베이스 검증과 AI 검증을 분리하면 속도·비용·정확도 트레이드오프를 각 방식에서 독립적으로 최적화 가능

## 2. 시스템 관점

### 2.1 검증 방식 정의

**룰베이스 검증**
규칙 기반(Regex / 사전 매칭 / 구조 분석)으로 즉시 탐지. AI 호출 없음.

**AI 검증**
LLM이 의미론적으로 판단. 룰베이스가 구조적으로 탐지 불가한 내용 요건 담당.

#### 2.1.1 검증 투입 원칙

> **룰베이스가 확실히 잡을 수 있는 범위는 룰베이스만 사용한다. AI는 룰베이스가 구조적으로 탐지 불가한 케이스에만 투입한다.**

- 룰베이스로 탐지 가능한 케이스에 AI를 중복 투입하지 않는다 (비용 최적화)
- 룰베이스 false positive는 탐지 범위를 좁혀 해결한다. AI 재검증으로 해결하지 않는다
- 하나의 검증 항목에 룰베이스와 AI가 병존하는 경우, 탐지 범위가 겹치지 않아야 한다
- 발생 빈도가 낮다고 판단되는 케이스는 이번 구현 범위에서 제외한다
- 구현 복잡성이 높은 경우, 빠르고 간단하게 구현 가능한 탐지 방식을 우선 채택한다

#### 2.1.2 시스템 구현 방향

##### 실행 시점

룰베이스·AI 검증 모두 **버튼 클릭 시에만** 실행한다. 저장·편집 완료 등 자동 실행은 허용하지 않는다.

##### 결과 표시 순서

룰베이스 검증 결과를 먼저 표시하고, AI 검증 결과를 항목별로 순차 추가한다. 변리사는 룰베이스 결과를 즉시 확인하면서 AI 결과를 기다릴 수 있다.

##### AI 호출 최적화

AI 검증 항목은 **입력 유형별로 묶어 처리**한다. 동일한 입력 데이터를 사용하는 항목을 하나의 LLM 호출로 처리하여 총 호출 횟수를 최소화한다.

예시 (T-03 청구항 검토 기준):
- 그룹 1: `ClaimItem.content`만 사용하는 항목 → 1회 호출
- 그룹 2: `ClaimItem.content + Document.content(실시예)` 사용 항목 → 1회 호출
- 그룹 3: 추가 입력 조합 항목 → 1회 호출

##### 룰베이스 실행 위치

초기 구현은 방식 A로 시작하되, 방식 C로 전환 가능한 구조(인터페이스 추상화)로 설계한다.

| 방식 | 설명 | 비고 |
| --- | --- | --- |
| **A — 메인 서버 내 모듈** | 기존 애플리케이션 서버에 룰 엔진 모듈로 통합. 별도 네트워크 통신 없음 | MVP 기준 초기 구현 방향 |
| **C — 별도 룰 엔진 서비스** | 룰 엔진을 독립 서비스로 분리. 검증 요청을 API로 처리 | 독립 배포·언어 유연성·확장성 확보 시 전환 |

##### T-04 의존성 처리

T-04(교차 항목 검토)는 T-03·T-06 결과를 입력으로 사용한다. T-04 단독 클릭 시 T-03·T-06이 미실행 상태라면 **자동으로 선행 실행** 후 T-04를 연결한다.

##### LLM 호출 실패 처리 (MVP)

LLM 호출 실패·타임아웃 발생 시 해당 T 도구 전체 결과를 오류 상태로 표시하고 재실행을 유도한다. 부분 성공 결과는 보존하지 않는다.

##### 룰베이스 규칙 업데이트 방식

법령 개정 등으로 룰베이스 규칙 변경이 필요한 경우 아래 세 방식 중 하나를 선택한다. 개발팀이 구현 시점의 상황에 맞게 택일한다.

| 방식 | 규칙 저장 위치 | 업데이트 절차 | 비고 |
| --- | --- | --- | --- |
| **A — 코드 하드코딩** | 소스코드 내 상수·패턴 | 코드 수정 → 배포 | 구현 단순, 규칙 변경마다 배포 필요 |
| **B — 설정 파일** | 서버 내 YAML/JSON config | 파일 교체 → 재시작 또는 hot reload | 배포 없이 규칙 교체 가능 |
| **C — DB 관리** | 룰 테이블 | 관리자 UI 또는 SQL | 런타임 변경 가능, 설계 비용 ↑ |

## 5. 사용자 스토리

1. 변리사로서, 기재요건 검토 버튼 클릭 시 다항제 기재방법 위반이 즉시 표시된다, 출원 전에 형식 오류를 놓치지 않도록
2. 변리사로서, 택일적 기재 위반·이중인용 위반 등 룰베이스로 잡을 수 있는 오류는 AI 응답을 기다리지 않고 즉시 확인할 수 있다, 빠르게 수정할 수 있도록
3. 변리사로서, 청구범위가 발명의 설명에 의해 뒷받침되는지 AI 검토를 요청할 수 있다, 심사관이 거절이유로 지적하기 전에 파악할 수 있도록
4. 변리사로서, 기재요건 검토 결과가 청구항 / 발명의 설명 / 도면 / 교차 항목 / 전체 단위로 분리 제공된다, 작업 범위에 따라 필요한 검토만 선택 실행할 수 있도록
5. 변리사로서, 검토 결과에 해당 법조항과 거절이유 해당 여부가 함께 표시된다, 심사관의 관점에서 문제를 이해할 수 있도록

### 이번 작업에서 구현 방향성

- 빠르게 적용할 수 있는 방향으로 기능 구현 방향을 잡을 것
- 상대적으로 발생되는 케이스가 드물다고 판단될 경우, 이번 작업 범위에서 제외함
- 구현 복잡성이 높을 때, 빠르고 간단하게 구현 가능한 방법을 이번 작업 범위로 설정함
- 검증 기능 단위로 기재불비 요건을 구현하며, 검증 기능 단위별 구현 우선순위를 설정함

| 발명의 설명 미작성 상태에서 T-03 요청 | "뒷받침 검토를 위해 발명의 설명 작성이 필요합니다" 안내 |
| 도면 미등록 상태에서 T-05 요청 | "도면이 등록되지 않았습니다" 안내 |

### 7.5 실질 독립항 판별 — 룰베이스·AI 역할 분담 (§42조8항 동령5조1항)

#### 역할 분담 원칙

| 케이스 | 처리 방식 | 탐지 방법 |
| --- | --- | --- |
| 말미어 카테고리 불일치 (부모와 발명 카테고리 다름) | 룰베이스 | 말미어 패턴 테이블 + 부모 카테고리 비교 |
| 치환형 — 기계 ("대신", "대체하여" 키워드) | 룰베이스 | 키워드 사전 탐지 |
| 삭제형 ("만으로 된", "제외" 키워드) | 룰베이스 | 키워드 탐지 + 부모 구성요소 목록 비교 |
| 치환형 — 화학 (치환기 상위/하위 개념 특정) | AI | 의미론적 판단 필요 |
| 룰베이스 키워드 미탐지 케이스 (문맥 의존 구성요소 증감) | AI | 룰베이스 플래그를 컨텍스트로 수신 후 보완 판단 |

> 룰베이스는 구조·키워드 기반으로 먼저 플래그. AI는 룰베이스 결과를 컨텍스트로 받아 예외 해소 및 의미 판단 케이스를 담당한다.

#### (A) 독립항이 다른 청구항을 인용하는 허용 형식 (5조1항 단서)

동일 사항의 중복 기재를 피하기 위해 발명이 명확하게 파악되는 범위 내에서 타항을 인용하는 독립항 형식은 허용된다. 말미어 카테고리가 부모와 다르면 룰베이스에서 참고 레벨로 표시, AI가 최종 확인.

| 예시 형식 | 말미어 카테고리 판단 |
| --- | --- |
| 청구항 N의 방법으로 제조된 …… 물건 | 방법발명(N) → 물건발명 → 룰베이스 참고 플래그 |
| ……하여 청구항 N의 물건을 제조하는 …… 방법 | 물건발명(N) → 제법발명 → 룰베이스 참고 플래그 |
| 청구항 N의 방법으로 제조된 물건을 이용하여 ……하는 방법 | 방법·물건 연쇄 인용 → 룰베이스 참고 플래그 |
| 청구항 N의 장치로 제조된 물건 | 동일 카테고리지만 중복 회피 목적 → AI 최종 판단 |

#### (B) 종속항 형식이나 실질 독립항으로 취급하는 케이스

어두에 타항을 인용하지만 종속항으로 보지 않고 **독립항으로 취급**. 다중인용 재인용 판단 시 독립항과 동일 처리.

**삭제형** — 룰베이스 키워드 탐지 → AI 보완

| 예시 | 판단 근거 |
| --- | --- |
| 청구항 1: 물탱크+밸브+노즐로 구성된 수분공급장치 → 청구항 2: 청구항 1에 있어서, 물탱크+밸브만으로 된 것 | 노즐 삭제 → 실질 독립항 [대법원 2005.11.10, 2004후3546] |

> **AI 판단 기준 (삭제형):** 인용 항의 구성요소를 한정하거나 부가하여 구체화한 것이 아니라면 실질 독립항. 역방향 기재(권리범위 넓은 항을 종속항 자리에 배치)가 바람직하다는 점도 수정 제안 시 참고.

**치환형 — 기계** (룰베이스 탐지 가능)

| 예시 | 판단 근거 |
| --- | --- |
| 청구항 1에 있어서, 기어 전동기구 대신 벨트전동기구를 구비한 … | "대신" 키워드 → 룰베이스 참고 플래그 → 실질 독립항 |

**치환형 — 화학** (AI 전담)

| 예시 | 판단 근거 |
| --- | --- |
| 청구항 1: 화학식(Ⅰ) 중 R1은 C1~6 알킬인 화합물 → 청구항 2: 청구항 1에 있어서, R1은 페닐인 화합물 | 상위 개념 치환기(C1~6알킬) → 하위 특정 치환기(페닐) → AI 의미 판단 필요 |

#### (C) 5조1항 임의규정 처리 원칙

동령5조1항은 임의규정으로 이 조항 단독 위반은 거절이유가 아님 [91후578]. 룰베이스·AI 모두 5조1항 관련 검출 결과를 "참고" 레벨로만 표시하고 거절이유 항목에 포함하지 않는다.

### 7.6 검증 입력 데이터 매핑 — ERD 필드 기준

검증 엔진이 읽어야 하는 DB 필드를 검증 항목별로 정의한다. 데이터 모델은 `write_document → write_claim → write_claimitem ↔ write_claimitem_parent_items` 구조 기준.

#### 룰베이스 검증 필드 매핑

| 검증 항목                  | 모델                 | 사용 필드                                                                                                                          |
| ---------------------- | ------------------ | ------------------------------------------------------------------------------------------------------------------------------ |
| 문언적 중복                 | ClaimItem          | `content` (전체 항목 교차 비교)                                                                                                        |
| 카테고리 다른 발명 병렬 기재       | ClaimItem          | `content`, `category`                                                                                                          |
| 이중인용 금지 위반             | ClaimItem (parent_items M2M) | `from_claimitem_id`, `to_claimitem_id` — 그래프 순회: from의 to가 >1개이고, 그 to 중 하나도 to가 >1개이면 위반                                      |
| 인용 순서 위반               | ClaimItem (parent_items M2M) | child.`order` < parent.`order` 이면 위반                                                                                           |
| 실질 독립항 탐지 (말미어)        | ClaimItem (parent_items M2M) | `parent_items` M2M으로 루트 독립항 재귀 탐색 (`parent_items`가 없는 조상까지) → 루트 독립항 `category` / 대상 ClaimItem `content` 말미어 파싱으로 카테고리 추론 후 비교 |
| 치환형 탐지                   | ClaimItem (parent_items M2M) | `parent_items` M2M (종속항 확인) + 대상 ClaimItem `content` (키워드 탐지)                                                                  |
| 삭제형 탐지                   | ClaimItem (parent_items M2M) | `parent_items` M2M (직접 부모 탐색) + 대상 ClaimItem `content` + 직접 부모 ClaimItem `content` (구성요소 비교)                                    |
| 인용 번호 미기재              | ClaimItem          | `content` (Regex)                                                                                                              |
| 택일적 기재 위반              | ClaimItem          | `content` (Regex)                                                                                                              |
| 표현 위반 (임의부가·불명확·부정·수치) | ClaimItem          | `content`                                                                                                                      |
| 배경기술 섹션 누락             | Document           | `content` (섹션 구조 파싱)                                                                                                           |

#### AI 검증 필드 매핑

| 검증 항목 | 모델 | 사용 필드 |
| --- | --- | --- |
| 뒷받침 요건 | ClaimItem + Document | ClaimItem.`content` (전체 항목) ↔ Document.`content` (실시예 섹션) |
| 실시가능 요건 | Document | `content` (발명의 설명 전체) |
| 용어 일관성 | ClaimItem + Document | ClaimItem.`content` + Document.`content` (실시예 + 도면 설명) |
| 복수 발명 의미 판단 | ClaimItem | `content`, `category` |
| 치환형 화학 (의미 판단) | ClaimItem (parent_items M2M) | 대상 ClaimItem.`content` + parent ClaimItem.`content` |

#### 미확인 전제 (→ OQ-8, OQ-9)

- **`category` 필드 채워지는 방식**: 명세서 저장 시 말미어 파싱으로 자동 분류되는지, 수동 입력인지 확인 필요. 자동 분류면 룰베이스 검증에서 직접 사용 가능; 아니면 `content`에서 직접 파생해야 함
- **`parent_items` M2M과 텍스트 인용의 동기화 보장 여부**: "제3항에 있어서"라고 텍스트에 있는데 M2M 연결이 없거나 반대인 경우, 어느 쪽을 신뢰 소스로 볼지 정책 결정 필요. 이중인용·순서 위반 룰베이스 검증 정확도가 여기에 달림

## 9. 기능 상세

### 9.1 룰베이스 검증 결과 표시 (T-03)

**위치:** document > 챗 패널 + 청구항 에디터 인라인

**설명:** 변리사가 T-03 버튼을 클릭하면 룰베이스 검증이 실행되고, 위반 항목이 챗 패널 결과 목록과 해당 청구항 인라인 배지로 표시된다.

**구성 요소**

| 요소 | 설명 |
| --- | --- |
| 위반 배지 | 청구항 텍스트 내 위반 구간 하이라이트 + 위반 유형 툴팁 |
| 위반 목록 패널 | 챗 패널 내 T-03 실행 결과. 전체 위반 항목 목록 |
| 법조항 표시 | 각 위반 항목에 해당 법조항 + 거절이유 여부 표기 |

**동작 흐름**

```
변리사: [기재요건 검토 — 청구항] 버튼 클릭
    ↓
룰베이스 검증 실행 (동기적)
    ↓
위반 항목 없음 → 정상 표시
위반 항목 있음 → 챗 패널 결과 목록 + 해당 청구항 인라인 배지
    ↓
변리사: 배지 클릭 → 위반 상세 + 해당 조항 표시
```

**상태 조건**

| 조건 | 상태 |
| --- | --- |
| 위반 없음 | 배지 없음 |
| 거절이유 위반 | 경고 배지 (빨간색 계열) |
| 임의규정 위반 또는 품질 참고 | 참고 배지 (회색 계열) |

### 9.2 기재요건 검토 — 청구항 (T-03)

**위치:** document > 챗 패널 (AI 어시스턴트 연동)

**설명:** 청구항에 대해 룰베이스 + AI 검토를 실행한다. 룰베이스 결과를 먼저 표시하고, AI 검증 결과를 순차 추가한다.

**구성 요소**

| 요소 | 설명 |
| --- | --- |
| AI 작업 내역 | 에이전트가 무엇을 분석하는지 단계별 표시 (→ C-01) |
| 검토 결과 목록 | 룰베이스 형식 위반 + AI 뒷받침·구조 검토 결과 |
| 섹션 앵커 | 해당 실시예 위치로 이동하는 클릭 가능 링크 |
| [수정 제안] 버튼 | 클릭 시 T-07 실행 → 수정 전후 비교 표시 |

**동작 흐름**

```
변리사: "청구항 N항 기재요건 검토해줘" (또는 [기재요건 검토] 클릭)
    ↓
컨텍스트 구성: 대상 청구항 + 전체 청구항 계층 + 발명의 설명 + 기재불비 기준(청구항 카테고리)
    ↓
T-03 실행: 룰베이스 먼저 → 결과 즉시 표시
           AI 검증 (입력 유형별 묶음) → 순차 추가
    ↓
수정 필요 항목: [수정 제안] 버튼 → T-07 실행 → [반영] / [취소]
```

### 특허법 제42조 (특허출원)

**제1항**
특허를 받으려는 자는 다음 각 호의 사항을 적은 특허출원서를 특허청장에게 제출하여야 한다.
1. 특허출원인의 성명 및 주소(법인인 경우에는 그 명칭 및 영업소의 소재지를 말한다)
2. 특허출원인의 대리인이 있는 경우에는 그 대리인의 성명 및 주소나 영업소의 소재지(대리인이 특허법인·특허법인(유한)인 경우에는 그 명칭, 사무소의 소재지 및 지정된 변리사의 성명을 말한다)
3. 발명의 명칭
4. 발명자의 성명 및 주소

**제2항**
제1항의 특허출원서에는 발명의 설명, 청구범위를 적은 명세서와 필요한 도면 및 요약서를 첨부하여야 한다.

**제3항** (발명의 설명 기재요건)
제2항에 따른 발명의 설명은 다음 각 호의 요건을 모두 충족하여야 한다.
1. 그 발명이 속하는 기술분야에서 통상의 지식을 가진 사람이 그 발명을 쉽게 실시할 수 있도록 명확하고 상세하게 적을 것
2. 그 발명의 배경이 되는 기술을 적을 것

**제4항** (청구범위 기재요건)
제2항에 따른 청구범위에는 보호를 받으려는 사항을 적은 항(이하 "청구항"이라 한다)이 하나 이상 있어야 하며, 그 청구항은 다음 각 호의 요건을 모두 충족하여야 한다.
1. 발명의 설명에 의하여 뒷받침될 것
2. 발명이 명확하고 간결하게 기재될 것

**제5항** (삭제, 2014.6.11.)

**제6항** (청구범위 기재 요소 완비 요건 — 거절이유·무효사유 O)
청구범위에는 발명을 특정하는 데 필요한 구조·방법·기능·물질 또는 이들의 결합관계 등을 기재하여야 한다.

**제7항** (삭제, 2014.6.11.)

**제8항** (청구범위 기재방법 위임)
제2항에 따른 청구범위의 기재방법에 관하여 필요한 사항은 대통령령으로 정한다.

**제9항** (명세서·도면 기재방법 위임)
제2항에 따른 명세서, 요약서 및 도면의 기재방법·서식 등에 관하여 필요한 사항은 산업통상자원부령으로 정한다.

---

### 특허법 시행령 제5조 (청구범위의 기재방법)

**제1항** (독립항·종속항 형식 — 임의규정 [91후578])
청구항은 독립항(이하 "독립항"이라 한다)으로 기재함을 원칙으로 하되, 그 독립항을 한정하거나 부가하여 구체화하는 종속항(이하 "종속항"이라 한다)을 기재할 수 있다. 이 경우 종속항은 독립항 또는 다른 종속항을 인용하는 형식으로 기재한다.

**제2항** (적정 수 기재 — 거절이유 O [98허8571, 88후967])
청구항은 발명의 성질에 따라 적정한 수로 기재하여야 한다.

**제4항** (인용 번호 기재 — 거절이유 O)
2이상의 항을 인용하는 청구항에 있어서 인용되는 청구항을 적을 때에는 그 번호를 적어야 한다.

**제5항** (택일적 기재 — 거절이유 O)
2이상의 항을 인용하는 청구항은 인용되는 항 중 어느 하나를 택일적으로 인용하는 것으로 기재하여야 한다.

**제6항** (다중인용 금지 — 거절이유 O)
2이상의 항을 인용하는 청구항(이하 "다중인용청구항"이라 한다)은 2이상의 항을 인용한 다른 청구항을 인용하여서는 아니 된다.

**제7항** (인용 순서 — 거절이유 O)
인용되는 청구항은 그 청구항보다 앞서 기재된 청구항만을 인용할 수 있다.

| 말미어 패턴 | 카테고리 판정 | 비고 |
| --- | --- | --- |
| "청구항 N의 방법으로 제조된 …… **물건**" | 물건 | 방법발명(N) → 물건발명 연계 — 룰베이스 참고 플래그, AI 최종 확인 |
| "……하여 청구항 N의 물건을 제조하는 …… **방법**" | 제법발명 | 물건발명(N) → 제법발명 연계 — 룰베이스 참고 플래그, AI 최종 확인 |
| "청구항 N의 방법으로 제조된 물건을 이용하여 ……하는 **방법**" | 방법 | 방법·물건 연쇄 인용 — 룰베이스 참고 플래그, AI 최종 확인 |
| "청구항 N의 장치로 제조된 **물건**" | 물건 | 제조장치(N) → 물건발명 연계 — AI 판단 |

### C.3 5조2항 이종 카테고리 위반 예시

> 심사기준 doc 2420 (5조2항 예시)

| 위반 표현 예 | 카테고리 조합 |
| --- | --- |
| "… 고분자 화합물 **및** 그 고분자 화합물을 이용한 컨텍트 렌즈" | 물건 + 물건(응용품) |
| "… 하는 **방법 및** 그 방법에 의해 제조된 **물건**" | 방법 + 물건 |
| "… **장치 및** 그 장치를 이용한 **방법**" | 장치(물건) + 방법 |

**하이브리드 방식 정의** (14~16 적용 시):
- 1단계: Regex로 표현 후보 탐지
- 2단계: 탐지된 후보에 대해 LLM이 "발명의 설명 뒷받침 여부" 판단 → 예외 해당 시 탐지 제외

**선택 기준**:
- 하이브리드: LLM 호출 최소화 우선 시
- 전체 LLM: 구현 단순화 우선 시

### D.3 도메인 모델 명세 확인 필요

검증 리스트에 나열된 입력 데이터(`ClaimItem.content`, `parent_items M2M`, `Document.content` 등)가 **현재 백엔드에 어떻게 존재하는지** 확인이 필요하다. 없는 필드가 있으면 구현 비용이 크게 달라진다.

확인이 필요한 모델 및 필드:

```
ClaimItem
  - id
  - order          (청구항 번호 — 존재 여부?)
  - content        (전문 텍스트)
  - category       (물건/방법/용도 등 — 현재 저장되는가?)
  - parent_items   (다대다 인용 관계 — M2M 테이블 존재 여부?)

Document
  - content        (발명의 설명 전체 텍스트)
  - figure_description  (도면 설명)
  - sections       ({ 기술분야, 배경기술, 해결수단, 실시예, 효과 }
                    섹션별 구분이 현재 되는가?)
```

> 기술 검토 의제 1순위. `parent_items M2M` 또는 `Document.sections` 구분이 없다면 해당 검증 항목 전체의 구현 비용이 달라진다.

### D.5 결과 표시 포맷

챗 패널에서 검증 결과를 어떻게 보여줄지 LLM 출력 형식을 정의한다. 이 포맷은 LLM 시스템 프롬프트에 삽입하여 출력 형식을 지시하는 용도로 사용한다.

#### 템플릿

```
{{검토등급}}: {{#청구항번호}} / {{위반_항목명}}
근거 | {{법조항}}
내용 | {{위반_설명}}
제안 | {{수정_방향}}
```

- `{{검토등급}}`: `거절이유` · `주의` · `참고` 중 하나
- `{{#청구항번호}}`: `#N` 형식. 복수 청구항은 `#5, #7`
- 제안이 없는 항목: `제안 | —`
- 복수 결과: `---`로 항목 구분

#### 단일 위반 예시

```
거절이유: #5 / 다중인용항 재인용 금지 위반
근거 | 특허법시행령 제5조제6항
내용 | 청구항 5가 다중인용항인 청구항 3을 재인용하고 있습니다.
제안 | 청구항 3의 인용을 단일 항으로 분리하거나 청구항 5의 인용 구조를 변경하세요.
```

#### 복수 위반 예시

```
거절이유: #5 / 다중인용항 재인용 금지 위반
근거 | 특허법시행령 제5조제6항
내용 | 청구항 5가 다중인용항인 청구항 3을 재인용하고 있습니다.
제안 | 청구항 3의 인용을 단일 항으로 분리하거나 청구항 5의 인용 구조를 변경하세요.
---
주의: #14 / 임의부가적 표현 사용
근거 | 특허법 제42조제4항제2호
내용 | 「필요에 따라」 표현이 발명의 설명으로 뒷받침되지 않으면 구성요소 범위가 불명확해질 수 있습니다.
제안 | 해당 표현을 삭제하거나 발명의 설명에 선택적 첨가 근거를 보완하세요.
---
참고: #6 / 명확성 목적 동시인용
근거 | 시행령 제5조제5항1호②
내용 | 거절이유에 해당하지 않으나 동시인용 구문이 확인되었습니다.
제안 | —
```
