# LLM 토픽 추출 구조 평가

## 대상 기능 개요

쿼리(기술 설명)로 특허 출원건을 검색한 뒤, 각 출원건에 대해 LLM이 아래 필드를 생성한다.

| 필드 | 설명 |
|---|---|
| `objectives` | 이 출원의 목적 키워드 (2–3개) |
| `solutions` | 이 출원의 수단 키워드 (2–3개) |
| `matched_objective_topic` | 전체 출원 집합을 목적 기준으로 분류한 토픽 |
| `matched_solution_topic` | 전체 출원 집합을 수단 기준으로 분류한 토픽 |

---

## 평가 대상 쿼리

> 전기차 주요 장치의 데이터 스캔 장치로부터 데이터를 획득하고, 데이터 DB를 통해 선별 압축 데이터를 획득하고, 선별 압축 데이터를 클라우드로 전송하여, 클라우드는 OBD 기반 데이터를 획득하고, 선별 압축 데이터 및 OBD 기반 데이터를 포함하는 전기차 관련 데이터에 기초하여 이슈 데이터를 추출하고, 추출된 이슈 데이터를 통해 이슈 이벤트를 설정하여 실시간 알림 및 경고를 제공하는 기술

**쿼리의 수단 파이프라인:**

```
[1] 데이터 스캔 장치 → [2] 선별 압축 → [3] 클라우드 전송 + OBD 통합
→ [4] 이슈 모델링 → [5] 실시간 알림/경고
```

## Q1. objectives / solutions 추출 품질

### 결론: objectives 양호 / solutions 보통

#### 잘 된 점
- 쿼리와 직접 대응되는 출원(KR1020230148876)은 objectives/solutions 모두 쿼리 파이프라인을 잘 포착
- objectives는 청구항의 목적 구문("결정하는 단계", "점검 경로 생성", "판단하는")을 충실히 반영

#### 문제점

| 문제 유형 | 예시 |
|---|---|
| **objectives ↔ solutions 경계 불명확** | KR1020220161437: objective "점검 경로 최적화" vs solution "점검 경로 생성" → 실질적으로 동일 |
| **solutions이 컴포넌트 명칭 그대로** | "관리 판단부", "터미널 이벤트 검출", "경보정보관리부" → 수단의 본질(행위)이 아닌 시스템 명칭 |
| **objectives 내 중복** | KR1020230171112: "클라우드 사전보안 점검"과 "보안 위험 판단"은 실질적으로 같은 내용 |
| **쿼리와의 접점이 반영 안 됨** | KR1020230171112는 "클라우드 전송" 측면이 쿼리와 겹치지만 solutions에 나타나지 않음 |

## Q2. matched_objective_topic / matched_solution_topic 분류 품질

### 결론: objective_topic 대체로 양호 / solution_topic 문제 있음

#### matched_objective_topic
- 대부분 올바른 목적 카테고리에 배정
- 단, 특이성 소실 케이스: US12566012의 "지역(경보지역) 기반 가입자 식별"이라는 핵심 특성이 "알림 전송 신뢰성 및 메시지 처리"로 뭉뚱그려짐

#### matched_solution_topic 주요 문제

| 출원 | solutions | matched_solution_topic | 문제 |
|---|---|---|---|
| KR1020230148876 | 데이터 스캔 장치, 선별 압축 데이터, **이슈 모델링** | 데이터 선별·압축 및 이상데이터 수집 | 쿼리 핵심 수단 "이슈 모델링"이 토픽에서 누락 |
| KR1020180026513 | 보안엘리먼트 알림설정, 셀룰러 통신 회로, **알림 재시도 로직** | 사용자·가입자·인증 관리 | 핵심 수단이 재시도/신뢰성인데 "인증 관리"로 잘못 분류 |
| KR1020210066563 vs KR1020220061051 | 유사한 빅데이터 웨어하우스 구조 | "원인분석 및 이슈 모델링" vs "데이터 추출·전처리 모듈" | 동일 구조에 다른 토픽 배정 — 일관성 문제 |

## 구조 설계 이슈 및 권장안

### 현재 구조의 본질적 문제

objectives/solutions가 **쿼리를 고려해서 추출**되기 때문에 두 가지 관심사가 하나의 필드에 뒤섞인다.

| 관심사 | 질문 |
|---|---|
| **특허 표현** | 이 특허는 무엇을 하는가? |
| **검색 관련성** | 이 특허가 왜 쿼리에 매칭됐는가? |

### 권장 구조 (관심사 분리)

```
claims (독립 추출)         +         쿼리
         ↓                                ↓
objectives / solutions            매핑 레이어
(순수 특허 표현 — 안정, 재사용 가능)      ↓
              ↘                 matched_aspect  ← 쿼리와의 접점 명시
                ↘              ↙
                matched_objective_topic
                matched_solution_topic
```

**예시 (KR1020230148876):**

```json
{
  "applicationNumber": "KR1020230148876",
  "objectives": ["플릿 통합 관제", "실시간 이슈 알림", "비OBD 데이터 통합"],
  "solutions": ["데이터 스캔 장치", "선별 압축 데이터", "이슈 모델링", "클라우드 전송"],
  "matched_aspect": "비OBD 데이터 선별 압축 → 클라우드 전송 → 이슈 모델링",
  "matched_objective_topic": "플릿 통합 관제 및 다중 차량 운영",
  "matched_solution_topic": "데이터 선별·압축 및 이슈 모델링"
}
```

### 구조 비교

| 항목 | 현재 (쿼리-aware 혼합) | 권장 (관심사 분리) |
|---|---|---|
| objectives/solutions 재사용 | ❌ 쿼리 바뀌면 재추출 필요 | ✅ 고정, 재사용 가능 |
| 쿼리 관련성 명시 | ❌ 암묵적 | ✅ `matched_aspect`로 명시 |
| matched_topic 정확도 | ⚠️ 쿼리 편향 objectives 기반 | ✅ 풍부한 objectives + 쿼리 맥락 동시 고려 |
| 특허 전체 표현 | ⚠️ 쿼리 관련 부분만 부각 | ✅ 특허 본질 그대로 |

## 개선 우선순위

1. **`matched_aspect` 필드 추가** — 쿼리와의 접점을 별도로 명시 (가장 임팩트 큼)
2. **objectives/solutions를 claims에서 독립 추출**로 전환 — 재사용성 확보
3. **matched_solution_topic 분류 기준 정비** — 쿼리 파이프라인 단계를 taxonomy 축으로 사용
4. **solutions 추출 방식 개선** — 컴포넌트 명칭이 아닌 기술적 행위(동사+명사) 형태로 추출
