## 1. 개요

### 1.1 목적

PatSol 명세서 작성 흐름에서 **어느 시점에 어떤 항목을 AI로 생성할지**를 명확히 정의하고, **각 시점·각 항목에 대한 기재요건 검토 규칙**을 규정한다. 이 문서는 `frd-spec-ai-assistant.md`의 도구(T-01~T-09)가 언제, 무엇을, 어떤 기준으로 생성·검토하는지를 결정하는 상위 오케스트레이션 명세다.

### 1.2 배경

- 명세서 12개 항목은 서로 다른 입력 소스에 의존한다. 청구항이 확정되기 전에 과제의 해결 수단을 생성하면 G-03 시점에 전면 재작성이 발생한다
- 기재요건 검토는 단순 오탈자 검수가 아니라 청구항↔명세서↔도면의 3방향 정합성 검증이다. 검토 기준(기재불비 유형)과 검토 시점이 사전에 명세되지 않으면 에이전트가 일관되지 않은 피드백을 생성한다
- `frd-spec-ai-assistant.md`는 챗 패널 UX 인터랙션을 다루며, 본 문서는 그 이면의 생성 로직과 검토 규칙을 다룬다

### 1.3 AI 역할 한계 (명시적 원칙)

> AI는 입력 컨텍스트가 충분한 항목만 생성하며, 입력이 없거나 확정되지 않은 항목은 초안 또는 생성 불가 상태로 처리한다. 기재요건 충족 여부의 최종 판단은 변리사에게 있다. (P-201, P-103)

### 1.4 관련 기능 위치

| 위치 | 기능 |
|---|---|
| /document > 챗 패널 | AI 에이전트 인터페이스 (→ `frd-spec-ai-assistant.md`) |
| /document > 명세서 편집 영역 | 생성 결과 반영, 기재요건 인라인 표시 |
| /document > 명세서 편집 영역 > 섹션 헤더 | 섹션별 생성 버튼 및 상태 배지 |

## 2. 용어 정의

### 2.1 생성 파이프라인 구조

```
[선행기술조사]          [발명신고서]
      ↓                      ↓
   G-01                   G-02
청구항 생성           명세서 본문 생성
(+ 발명의 명칭 초안)   (기술분야·배경기술·과제·효과·
 (+ 배경기술 초안)      구체적내용·실시예·해결수단 초안)
      ↓                      ↓
      └──────────┬───────────┘
                 ↓
              G-03
         통합 초안 생성
  (청구항 확정·과제의 해결 수단 확정·
   요약 생성·배경기술 통합·
   도면의 간단한 설명 뼈대 생성)
                 ↓
         [도면 업로드 / AI 생성]
                 ↓
              G-04
         도면 설명 확정
  (도면의 간단한 설명 내용 채우기·
   구체적내용·실시예 도면 부호 삽입)
```

### 2.2 용어 정의

| 용어 | 정의 |
|---|---|
| **생성 시점(G-NN)** | 명세서 작성 흐름에서 AI 생성이 일어나는 주요 단계. G-01~G-04로 구분 |
| **초안 항목** | 해당 생성 시점에서 생성되지만, 이후 시점에서 재생성·확정될 것이 예정된 항목. "초안" 배지 표시 |
| **확정 항목** | 해당 생성 시점에서 최종 버전으로 생성되는 항목 |
| **생성 불가** | 필수 입력 컨텍스트가 없어 이 시점에 생성할 수 없는 항목 |
| **유지** | 이전 생성 시점의 결과를 그대로 사용하며 재생성하지 않는 항목 |
| **기재요건 검토(R-NN)** | 명세서가 법적·기술적 기재 요건을 충족하는지 검토하는 기능 분류. R-01~R-04로 구분 |
| **기재불비(BQ-NN)** | 심사관이 지적하는 명세서 기재의 법적 불비 유형. BQ-01~BQ-08로 분류 |
| **도면 부호** | 도면에서 각 구성요소를 식별하는 숫자 또는 기호 (예: 제어 유닛(100)) |
| **뼈대** | 도면의 간단한 설명에서 "도 N은 …이다" 형식의 목록만 있고 내용은 미작성인 상태 |
| **발명신고서** | 발명자가 발명 내용을 정리한 내부 문서. 기술분야·배경·목적·효과·실시예 포함 |

## 3. 사용자 스토리

1. 변리사로서, 선행기술조사 결과를 바탕으로 청구항을 AI가 생성해준다, 선행기술과 충돌하지 않는 권리범위를 빠르게 확보할 수 있도록
2. 변리사로서, 발명신고서를 업로드하면 명세서 본문 6개 항목이 일괄 생성된다, 초안 작성에 드는 시간을 줄일 수 있도록
3. 변리사로서, 청구항과 본문이 모두 준비된 후 통합 초안을 한 번에 완성할 수 있다, 섹션 간 일관성을 자동으로 보장받을 수 있도록
4. 변리사로서, 도면을 업로드하면 도면의 간단한 설명과 도면 부호가 자동으로 삽입된다, 부호 불일치로 인한 기재불비를 사전에 방지하도록
5. 변리사로서, 어떤 항목이 아직 초안 상태인지 한눈에 볼 수 있다, 제출 전 최종 검토 누락을 방지하도록
6. 변리사로서, 청구항을 수정했을 때 영향받는 항목(과제의 해결 수단·요약)에 재생성 권고 알림이 뜬다, 일관성 깨짐을 즉시 인지할 수 있도록
7. 변리사로서, 기재불비 유형별로 문제 위치와 수정 방향을 안내받는다, 심사 단계에서 거절이유통지를 받기 전에 스스로 수정할 수 있도록
8. 변리사로서, 기재요건 검토 결과의 심각도(오류·주의·권고)를 구분해서 볼 수 있다, 제출 전 우선순위를 정해 대응할 수 있도록
9. 변리사로서, 도면 부호 불일치가 발견되면 명세서 내 해당 위치로 바로 이동할 수 있다, 부호 수정 작업을 빠르게 완료하도록
10. 변리사로서, G-01만 완료된 상태에서도 G-02 없이 G-03 진행이 가능하다, 발명신고서가 없는 경우에도 최소한의 초안을 확보할 수 있도록

## 4. 정책 및 규칙

| # | 정책 | 내용 |
|---|---|---|
| P-501 | 생성 시점 제약 | 각 생성 시점(G-NN)은 해당 시점의 입력 컨텍스트가 충분한 항목만 생성한다. 입력이 없는 항목은 생성하지 않으며, 상태를 "생성 불가"로 표시한다 |
| P-502 | 초안 배지 표시 | 초안 항목은 섹션 헤더에 "초안" 배지를 표시한다. G-03 확정 생성 후 배지가 제거된다 |
| P-503 | 청구항-해결수단-요약 동기화 | G-03 이후 청구항이 수정되면 과제의 해결 수단과 요약에 "재생성 권고" 알림을 표시한다. 자동 재생성은 하지 않는다 |
| P-504 | 도면 설명 편집 경고 | G-03 완료 후 G-04 전까지 도면의 간단한 설명 섹션을 수동 편집하면 "도면 업로드 후 자동 완성됩니다. 지금 수동 편집하면 G-04에서 덮어쓸 수 있습니다" 경고를 표시한다 |
| P-505 | 요약 생성 시점 제한 | 요약은 G-03(청구항 확정) 이후에만 생성 가능하다. G-01·G-02 시점에서 요약 생성은 disabled 처리한다 |
| P-506 | 기재불비 기준 사전 반영 | 모든 생성 시점에서 기재불비 기준(BQ-01~BQ-08)을 프롬프트에 내재화하여 생성 단계에서 기재불비를 사전 방지한다 |
| P-507 | 검토 레벨 3단계 | 기재요건 검토 결과는 오류(제출 전 필수 해결) / 주의(강하게 권고) / 권고(선택적 개선) 3단계로 구분한다 |
| P-508 | G-01·G-02 독립 실행 허용 | G-01과 G-02는 순서 무관하게 독립 실행 가능하다. 어느 하나가 없어도 G-03 진행은 허용하되, 누락 항목을 명시한다 |
| P-509 | G-04 도면 부호 일괄 삽입 | G-04에서 도면 부호 삽입은 구체적인 내용·실시예 섹션에 일괄 적용한다. 기존 도면 부호가 있는 경우 충돌 여부를 먼저 표시한다 |

## 5. 기능 상세

---

### 5-1. G-01 — 선행기술조사 → 청구항 생성

**위치:** /document > 명세서 편집 영역 > 챗 패널 또는 청구항 섹션 생성 버튼

**설명:** 선행기술조사 결과를 기반으로 청구항을 생성한다. 발명의 명칭·배경기술·해결과제는 이 시점에서 초안으로만 생성된다.

#### 입력 컨텍스트

| 항목 | 필수 여부 |
|---|---|
| 선행기술조사 결과 (매칭 선행특허 + 비교발명 분석) | 필수 |
| 발명 요약 (직무발명서 / 발명 설명) | 권장 |

#### 생성 항목

| 항목 | 처리 방식 | 배지 |
|---|---|---|
| 청구항 | **생성 (확정)** — 독립항 1개 이상, 종속항 구조 포함 | AI 생성 |
| 발명의 명칭 | **초안 생성** — 청구항 내용에서 도출 | 초안 |
| 배경기술 | **초안 생성** — 선행기술 분석 기반 | 초안 |
| 해결하는 과제 | **초안 생성** — 선행기술의 한계에서 도출 | 초안 |
| 나머지 항목 | **생성 불가** | — |

#### 동작 흐름

```
트리거: 선행기술조사 연결 상태에서 [청구항 생성] 실행
    ↓
선행기술조사 결과 존재 여부 확인 (→ P-501)
    ↓ (없음)
"선행기술조사 결과를 연결해주세요" 안내 + 생성 disabled
    ↓ (있음)
생성 횟수 안내 (→ frd-spec-ai-assistant 7-9)
    ↓
AI 작업 내역 표시
LLM 호출 — 기재불비 기준 내재화 (→ P-506)
    ↓
청구항 생성 → 명세서 편집 영역 반영 + AI 생성 배지
발명의 명칭 / 배경기술 / 해결과제 초안 생성 → 각 섹션 반영 + 초안 배지 (→ P-502)
    ↓
챗 패널에 생성 요약 출력:
  - 독립항 N개, 종속항 M개 생성
  - 초안 생성 항목: 발명의 명칭, 배경기술, 해결과제
  - "G-03 통합 초안 생성 시 확정됩니다"
```

#### 상태 조건

| 조건 | 상태 |
|---|---|
| 선행기술조사 미연결 | 생성 버튼 disabled |
| 발명 요약 없음 | 생성 진행, 챗에 "발명 요약 없음 — 청구항 구체성 보완 권장" 표시 |
| G-01 이미 실행됨 | 재실행 시 "청구항을 덮어씁니다" 확인 다이얼로그 |

### 5-2. G-02 — 발명신고서 → 명세서 본문 생성

**위치:** /document > 명세서 편집 영역 > 챗 패널 또는 [본문 생성] 버튼

**설명:** 발명신고서를 기반으로 명세서 본문 6개 항목을 일괄 생성한다. 과제의 해결 수단은 청구항 미확정 상태이므로 초안으로만 생성한다.

#### 입력 컨텍스트

| 항목 | 필수 여부 |
|---|---|
| 발명신고서 (직무발명서 / 발명 설명 텍스트) | 필수 |
| G-01 청구항 | 권장 (없어도 진행 가능, → P-508) |

#### 생성 항목

| 항목 | 처리 방식 | 비고 |
|---|---|---|
| 기술분야 | **생성 (확정)** | |
| 배경기술 | **생성 (확정)** — G-01 초안이 있으면 통합, 없으면 발명신고서 기반 | G-01 결과와 통합은 G-03에서 최종 조정 |
| 해결하는 과제 | **생성 (확정)** | G-01 초안이 있으면 정합성 확인 후 반영 |
| 발명의 효과 | **생성 (확정)** | |
| 발명을 실시하기 위한 구체적인 내용 | **생성 (확정)** | 도면 부호는 G-04에서 삽입 |
| 실시예 | **생성 (확정)** | 도면 부호는 G-04에서 삽입 |
| 과제의 해결 수단 | **초안 생성** — 발명 내용 기반 | 청구항 확정(G-03) 후 재생성 예정 |
| 청구항 | **생성 불가** (G-01 결과 유지) | |
| 도면의 간단한 설명 | **생성 불가** | G-03에서 뼈대 생성 |
| 요약 | **생성 불가** (→ P-505) | |
| 도면 | **생성 불가** | G-04 대상 |

#### 동작 흐름

```
트리거: 발명신고서 업로드 완료 후 [본문 생성] 실행
    ↓
발명신고서 존재 여부 확인 (→ P-501)
    ↓ (없음)
"발명신고서를 업로드해주세요" 안내
    ↓ (있음)
G-01 청구항 존재 여부 확인
    ↓ (있음)                          ↓ (없음)
청구항 컨텍스트 포함하여 생성       "청구항 없음 — 발명신고서만으로 생성합니다.
                                     G-01 완료 후 G-03에서 정합성 확인을 권장합니다" 안내
    ↓
생성 횟수 안내 (1회 차감)
    ↓
섹션 순서대로 순차 생성 및 반영
  기술분야 → 배경기술 → 해결과제 → 효과 → 구체적내용 → 실시예 → 과제의 해결 수단(초안)
    ↓
챗 패널에 생성 요약 출력:
  - 생성 완료 항목 목록
  - 초안 항목: 과제의 해결 수단 (G-03에서 청구항과 동기화 예정)
  - 미생성 항목: 도면의 간단한 설명, 요약, 도면
```

#### 상태 조건

| 조건 | 상태 |
|---|---|
| 발명신고서 없음 | 생성 버튼 disabled |
| G-01 청구항 없음 | 생성 진행, 챗에 경고 안내 (→ P-508) |
| G-02 이미 실행됨 | 재실행 시 수동 수정 보호 섹션 스킵 안내 |

### 5-3. G-03 — 통합 초안 생성

**위치:** /document > 명세서 편집 영역 상단 > [통합 초안 완성] 버튼

**설명:** G-01 청구항과 G-02 본문을 통합하여 명세서 초안을 완성한다. 과제의 해결 수단·요약을 확정 생성하고, 초안 배지를 제거한다. 도면의 간단한 설명은 뼈대만 생성한다.

#### 사전 조건

| 조건 | 처리 |
|---|---|
| G-01 + G-02 모두 있음 | 정상 실행 |
| G-01만 있음 | 실행 허용, 본문 항목 누락 안내 (→ P-508) |
| G-02만 있음 | 실행 허용, 청구항 누락 안내 |
| 둘 다 없음 | [통합 초안 완성] disabled |

#### 생성/처리 항목

| 항목 | 처리 방식 | 배지 변화 |
|---|---|---|
| 청구항 | **정합성 검토 후 확정** — 본문과 용어 일관성 점검, 변경 필요 시 diff 제시 | 초안 → AI 생성 |
| 발명의 명칭 | **확정** — G-01 초안 유지 또는 본문 반영하여 개선 | 초안 → AI 생성 |
| 과제의 해결 수단 | **확정 생성** — 청구항 기반 재생성 (G-02 초안 교체) | 초안 → AI 생성 |
| 요약 | **생성** — 청구항 기반 신규 생성 (→ P-505) | — → AI 생성 |
| 배경기술 | **통합·확정** — G-01 선행기술 초안 + G-02 발명신고서 내용 합성 | 초안 → AI 생성 |
| 해결하는 과제 | **확정** — G-01·G-02 결과 정합성 검토 후 확정 | 초안 → AI 생성 |
| 도면의 간단한 설명 | **뼈대 생성** — "도 N은 …이다" 목록만. 내용 미작성 (G-04 대기) | — → 뼈대(잠금) |
| 기술분야·효과·구체적내용·실시예 | **유지** — G-02 결과 그대로 | 변화 없음 |
| 도면 | **생성 불가** | — |

#### 동작 흐름

```
트리거: [통합 초안 완성] 버튼 클릭
    ↓
사전 조건 확인 — 없는 입력 항목 안내
    ↓
생성 횟수 안내 (1회 차감)
    ↓
AI 작업 내역 표시
    ↓
① 청구항 ↔ 본문 용어 일관성 점검
    ↓ (불일치 발견)
챗 패널에 diff 표시:
  "[청구항 2항] '구동 모터' → 명세서에서 '구동부'로 혼용"
  [청구항 수정]  [본문 수정]  [무시]
    ↓ (선택 후)
② 과제의 해결 수단 확정 생성 (청구항 기반)
③ 요약 생성 (청구항 기반)
④ 배경기술 통합
⑤ 발명의 명칭·해결과제 확정
⑥ 도면의 간단한 설명 뼈대 생성
    ↓
도면 설명 섹션 편집 경고 설정 (→ P-504)
    ↓
챗 패널에 통합 생성 요약 출력:
  - 확정 완료 항목 목록
  - 뼈대 생성: 도면의 간단한 설명 (도면 업로드 후 G-04 실행 필요)
  - 미생성 항목: 도면
  - 기재요건 자동 검토 opt-in 설정 활성화 시 → R-02 전체 자동 실행
```

#### 후처리 — 청구항 수정 감지

```
G-03 완료 이후 변리사가 청구항을 수동 수정
    ↓
과제의 해결 수단 + 요약 섹션 헤더에 "재생성 권고" 알림 표시 (→ P-503)
  "청구항이 수정되었습니다. 과제의 해결 수단·요약 재생성을 권고합니다. [재생성]"
```

### 5-4. G-04 — 도면 설명 확정

**위치:** /document > 명세서 편집 영역 > 도면 업로드 또는 도면 AI 생성 완료 후 트리거

**설명:** 실제 도면을 기반으로 도면의 간단한 설명을 완성하고, 구체적인 내용·실시예에 도면 부호를 삽입한다. G-03의 뼈대 상태를 최종 완성한다.

#### 입력 컨텍스트

| 항목 | 필수 여부 |
|---|---|
| 도면 파일 (업로드 또는 AI 생성 결과) | 필수 |
| G-03 통합 초안 (도면의 간단한 설명 뼈대 포함) | 필수 |

#### 생성/처리 항목

| 항목 | 처리 방식 |
|---|---|
| 도면의 간단한 설명 | **내용 채우기** — 각 도면의 구성요소와 설명 작성. G-03 뼈대 교체 |
| 발명을 실시하기 위한 구체적인 내용 | **도면 부호 삽입** — 구성요소명에 도면 부호 괄호 추가 (예: 제어 유닛(100)) |
| 실시예 | **도면 부호 삽입** — 동일 규칙 적용 |
| 도면 | **등록** — 업로드 또는 AI 생성 결과를 명세서에 연결 |

#### 동작 흐름

```
트리거: 도면 업로드 완료 또는 도면 AI 생성 완료
    ↓
G-03 완료 여부 확인
    ↓ (미완료)
"통합 초안(G-03) 완성 후 진행해주세요" 안내
    ↓ (완료)
도면 파일 분석 — 구성요소 목록 추출
    ↓
기존 도면 부호 존재 여부 확인
    ↓ (충돌 있음)
챗 패널에 충돌 목록 표시:
  "기존 부호 100 → 새 도면에서 200으로 변경 감지"
  [기존 유지]  [새 부호 적용]  [항목별 선택]
    ↓ (확인 또는 충돌 없음)
① 도면의 간단한 설명 내용 채우기 (뼈대 교체) + 편집 경고 해제
② 구체적인 내용 도면 부호 일괄 삽입 (diff 표시 후 [반영] 확인)
③ 실시예 도면 부호 일괄 삽입 (diff 표시 후 [반영] 확인)
    ↓
BQ-07 도면 부호 불일치 자동 검토 실행 (→ 5-6)
    ↓
챗 패널에 완료 요약 출력:
  - 도면의 간단한 설명 완성 (도 N개)
  - 부호 삽입 위치 수: 구체적인 내용 N곳, 실시예 M곳
  - 부호 불일치 검토 결과
```

### 5-5. R-01 — 항목별 필수 기재요건

**위치:** 검토 결과는 /document > 챗 패널 (T-04 연동) + 명세서 편집 영역 인라인 표시

**설명:** 각 명세서 항목이 법적으로 반드시 충족해야 하는 최소 기재요건을 정의한다. R-01은 규칙 정의이며, 실행은 T-03~T-05(→ `frd-spec-ai-assistant.md`)를 통해 이루어진다.

#### 항목별 기재요건 정의

| 항목 | 필수 기재요건 | 검토 레벨 |
|---|---|---|
| 청구항 | 독립항 1개 이상 존재 | 오류 |
| 청구항 | 각 청구항은 단일 발명을 정의할 것 | 주의 |
| 청구항 | 종속항은 인용 청구항 번호를 명시할 것 | 오류 |
| 발명의 명칭 | 발명의 기술 특징을 포함한 명사형 표현 | 주의 |
| 배경기술 | 종래 기술의 문제점이 1개 이상 명시될 것 | 주의 |
| 해결하는 과제 | 배경기술의 문제점과 연결된 과제가 기술될 것 | 주의 |
| 과제의 해결 수단 | 청구항의 모든 독립항 구성요소가 포함될 것 | 오류 |
| 발명의 효과 | 해결 수단에서 도출되는 효과가 기술될 것 | 주의 |
| 발명을 실시하기 위한 구체적인 내용 | 독립항의 모든 구성요소가 1회 이상 설명될 것 | 오류 |
| 실시예 | 독립항의 모든 구성요소를 1회 이상 포함할 것 | 오류 |
| 요약 | 발명의 구성 및 효과가 포함될 것 | 주의 |
| 도면의 간단한 설명 | 등록된 모든 도면에 대한 설명이 존재할 것 | 오류 |

### 5-6. R-02 — 기재불비 유형 분류 및 감지 로직

**위치:** 검토 결과는 챗 패널 이슈 목록 + 명세서 편집 영역 인라인 하이라이트

**설명:** 심사관이 실제로 지적하는 기재불비 패턴을 분류하고, 각 패턴의 감지 방법과 피드백 방식을 정의한다.

#### 기재불비 유형 정의

| BQ-ID | 유형명 | 정의 | 감지 방법 | 레벨 |
|---|---|---|---|---|
| BQ-01 | 독립항 미존재 | 청구항에 독립항이 없음 | 청구항 구조 파싱 | 오류 |
| BQ-02 | 용어 불일치 | 청구항 구성요소 명칭이 명세서 본문에서 다르게 사용됨 | 청구항 명사 추출 → 본문 전체 검색 | 오류 |
| BQ-03 | 실시예 미지지 | 독립항 구성요소가 실시예에 1회 이상 포함되지 않음 | 독립항 구성요소 목록 → 실시예 텍스트 매핑 | 오류 |
| BQ-04 | 해결수단-청구항 불일치 | 과제의 해결 수단에 청구항 독립항 구성요소가 누락됨 | 독립항 구성요소 목록 → 해결수단 텍스트 매핑 | 오류 |
| BQ-05 | 섹션 간 논리 단절 | 배경기술 문제점 → 해결과제 → 해결수단 간 논리 연결이 불명확함 | LLM 논리 흐름 평가 | 주의 |
| BQ-06 | 효과 미도출 | 발명의 효과가 청구항 구성요소에서 도출되지 않고 외부 개념을 언급함 | LLM 효과-구성요소 연결 평가 | 주의 |
| BQ-07 | 도면 부호 불일치 | 명세서에 기재된 부호가 도면에 존재하지 않거나 도면 부호가 명세서에 미기재 | 도면 부호 목록 ↔ 명세서 부호 표기 교차 검증 | 오류 |
| BQ-08 | 신규사항 추가 | 청구항에 기재된 내용이 명세서 본문에 전혀 없음 (출원 후 보정 불가 영역) | 청구항 구성요소 → 본문 전체 검색 (BQ-02와 구분: 없는 것 vs 다르게 쓴 것) | 오류 |

#### 기재불비 카드 상세 — BQ-03 예시

```
BQ-03 실시예 미지지
────────────────────────────────────────────
정의     청구항 독립항의 구성요소가 실시예 섹션에
         1회 이상 명시적으로 설명되지 않은 경우

감지     1. 독립항 구성요소 명사 추출
         2. 실시예 전체 텍스트에서 각 명사 존재 여부 확인
         3. 미매핑 구성요소 목록 생성

피드백   레벨: 오류
         위치: 실시예 섹션 상단 인라인 배지 + 챗 패널 이슈 목록
         메시지: "독립항 1항의 'X 모듈'이 실시예에 포함되지 않았습니다"
         액션: [실시예로 이동] [수정 제안 보기 → T-06 실행]
```

### 5-7. R-03 — 템플릿 기반 검토 규칙

**위치:** 챗 패널 검토 결과 + 명세서 편집 영역 인라인

**설명:** 법적 기재요건(R-01·R-02)과 별도로, 사무소 스타일·표현 패턴 준수 여부를 검토한다. R-03은 템플릿 규칙 정의이며, 사무소별 커스터마이즈가 가능하다.

#### 공통 템플릿 규칙

| 규칙 ID | 규칙 | 예시 | 레벨 |
|---|---|---|---|
| TQ-01 | 청구항 어미 통일 | 독립항 어미는 "것을 특징으로 하는 [발명명칭]." 형식 | 주의 |
| TQ-02 | 청구항 구성요소 열거 | 각 구성요소는 세미콜론(;) 또는 줄바꿈으로 구분 | 권고 |
| TQ-03 | 부정 표현 어미 | "않다" → "아니하다" (사무소 설정에 따라 선호 표현 규칙 우선 적용) | 권고 |
| TQ-04 | 효과 기재 방식 | "본 발명은 ~를 통해 ~의 효과가 있다" 구조 | 권고 |
| TQ-05 | 도면 참조 표현 | "도 N을 참조하면" 형식 (도면 없는 경우 생략) | 주의 |
| TQ-06 | 실시예 도입 표현 | "이하, 도면을 참조하여 본 발명의 실시예를 상세히 설명한다." | 권고 |

#### 사무소별 커스터마이즈

- R-03 규칙은 사무소 스타일 설정에서 오버라이드 가능 (→ P-6, `frd-spec-ai-assistant.md`)
- 선호 표현 규칙(T-09)과 충돌 시 선호 표현 규칙 우선 적용

### 5-8. R-04 — 검토 결과 피드백 UI 행동

**위치:** /document > 명세서 편집 영역 (인라인) + 챗 패널 (이슈 목록)

**설명:** R-01·R-02·R-03 검토 결과를 변리사에게 전달하는 UI 행동을 정의한다. 챗 패널의 기재요건 검토 결과 출력과 연동된다 (→ `frd-spec-ai-assistant.md` 7-7).

#### 심각도 레벨 정의 (→ P-507)

| 레벨 | 색상 | 정의 | 처리 |
|---|---|---|---|
| 오류 | 빨강 | 제출 전 반드시 해결해야 하는 기재불비 | 명세서 상단에 오류 건수 배너 표시 |
| 주의 | 노랑 | 심사 거절 가능성이 높아 강하게 권고하는 항목 | 해당 섹션 헤더에 ⚠ 배지 |
| 권고 | 파랑 | 선택적 개선 사항 | 챗 패널 이슈 목록에만 표시 (인라인 없음) |

#### 피드백 표시 방식

| 위치 | 내용 |
|---|---|
| 명세서 편집 영역 상단 배너 | 오류 N건 / 주의 M건 요약. 클릭 시 챗 패널 이슈 목록으로 이동 |
| 섹션 헤더 인라인 배지 | 오류 (빨강 ●) / 주의 (노랑 ▲). 클릭 시 챗 패널 해당 이슈로 스크롤 |
| 챗 패널 이슈 목록 | 이슈 항목 + 설명 + [섹션으로 이동] + [수정 제안 보기] |
| 이슈 항목 | BQ-ID / TQ-ID + 이슈 설명 + 레벨 배지 + 액션 버튼 |

#### 동작 흐름 — 오류 항목 처리

```
챗 패널 이슈 목록에서 오류 항목 확인
    ↓
[섹션으로 이동] 클릭
    ↓
명세서 편집 영역의 해당 섹션으로 스크롤 + 문제 부분 하이라이트
    ↓
[수정 제안 보기] 클릭
    ↓
T-06 실행 → 수정 전후 비교 표시 (→ frd-spec-ai-assistant 7-4)
    ↓
변리사: [반영] 클릭 → 적용 → 해당 이슈 "해결됨"으로 상태 전환
```
