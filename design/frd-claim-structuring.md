## 1. 개요

### 1.1 목적

변리사가 PatSol의 명세서 작성 페이지(document)에서 청구항의 계층 구조(독립항 → 종속항)를 시각적으로 파악하고, 추가·삭제·AI 보정을 수행할 수 있는 청구항 설계 UX를 제공한다.

### 1.2 배경

현재 PatSol의 명세서 작성 페이지에서는 청구항이 단순 텍스트 목록으로만 표시되어:
- 독립항과 종속항의 인용 관계를 시각적으로 파악하기 어렵고
- 신규 청구항 추가 시 어두(「제N항에 있어서」)를 수동으로 입력해야 하며
- 삭제 시 종속 관계 처리를 변리사가 직접 판단해야 한다

> **포지셔닝:** "검색은 어디서든 해와도 되고, 그 결과를 가지고 청구항 설계와 명세서 작성을 PatSol이 연결해준다"
> 청구항 구조화는 이 연결의 핵심 접점이다.

## 2. 용어 정의

### 2.1 포함 관계

```
document 페이지 > 청구항 패널
  └── [발명 카테고리]
        └── 독립항 (1..N)
              └── 종속항 (0..N)

독립항 / 종속항 ←── 말미어(종결부) 발명 카테고리 기준으로 판별
```

### 2.2 용어 정의

| 용어          | 정의                                                                                                                                          |
| ----------- | ------------------------------------------------------------------------------------------------------------------------------------------- |
| **독립항**     | 다른 청구항을 인용하지 않고, 발명의 필수 구성요소 전체를 독립적으로 기재한 청구항. 특허권의 권리범위를 가장 넓게 정의하는 최광의 청구항이다                                                             |
| **종속항**     | 다른 청구항(독립항 또는 종속항)을 인용하여 그 구성요소를 한정하거나 부가하는 청구항. 인용 청구항보다 좁은 권리범위를 가지며, 어두에 인용 항 번호를 명시한다                                                   |
| **발명 카테고리** | 특허법 제2조제3호에 따른 발명의 법적 분류. 청구항의 말미어(종결부)로 판별하며, 물건발명(장치·물질·조성물·기록매체·키트), 방법발명(방법·용도), 제법발명(제조방법)으로 구분된다. PatSol에서는 독립항 단위로 발명 카테고리 레이블을 표시한다 |
| **말미어**     | 청구항의 종결부. 발명 카테고리(물건발명·방법발명·제법발명)를 결정하는 핵심 요소                                                                                               |
| **어두**      | 종속항 문장의 첫머리. 예: 「제1항에 있어서,」 「제1항 또는 제2항에 있어서,」                                                                                              |
| **다중인용청구항** | 둘 이상의 청구항을 동시에 인용하는 청구항. 예: 「제N항 또는 제M항에 있어서,」. 한국·미국 등에서는 다중인용청구항을 다시 인용(재인용)하는 것을 기재불비로 규정한다 (한국 특허법 시행령 제5조)                             |
| **기재불비**    | 청구항이 특허법 요건(명확성·지지요건 등)을 충족하지 못하는 상태                                                                                                        |
| **AI 보정**   | LLM이 개별 또는 전체 청구항의 기재를 검토하여 기재요건에 맞게 개선안을 제시하는 기능                                                                                           |
| **계층 트리**   | 독립항을 루트 노드로, 종속항을 자식 노드로 표현하는 트리 구조                                                                                                         |

## 3. 정책 및 규칙

| #    | 정책            | 내용                                                                                                                                                     |
| ---- | ------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------ |
| P-1  | 카테고리 자동 판별    | LLM이 청구항 말미어를 분석해 세분류 카테고리를 자동 추론하여 초기값 지정. 사용자가 드롭다운으로 수동 변경 가능 (OQ-5 확정)                                                                             |
| P-2  | 종속항 추가 트리거    | 수동(변리사 주도) 단독. 컨텍스트 메뉴 또는 hover `+` 버튼으로 변리사가 직접 추가. AI 제안 트리거는 챗봇 MVP 이관 (OQ-1 확정)                                                                    |
| P-3  | 어두 자동 삽입      | 종속항 추가 시 부모 청구항 번호 기반으로 자동 삽입. 단일: `제N항에 있어서,` / 다중: `제N항 또는 제M항에 있어서,`                                                                                |
| P-4  | 청구항 삭제        | 종속항 있을 경우 경고 모달 표시. 강제 차단 없음. 번호 재정렬은 이번 스프린트 제외 (삭제 후 번호 gap 허용)                                                                                      |
| P-5  | 기재불비 검증 단계    | 기계적(규칙 기반) 검증을 현재 스프린트에 우선 구현. 검증 항목 10개 내외, 구조적 오류(다중인용 재인용 등) 포함. AI 프롬프트 기반 검증(명확성·지지요건 등)은 이후 단계 구현. 경고 및 수정안 제시, 강제 차단 없음. 최종 결정은 변리사             |
| P-6  | DB/서비스 레이어 분리 | DB는 모든 나라 구조 허용(다중인용 재인용 포함, `child_id` 다중 값 저장). 서비스는 한국 특허법 시행령 제5조 기준으로 기재불비 검증 및 경고. `parent_id` 필드는 `child_id`로 변경하여 2개 이상 값 저장 가능 구조 적용          |
| P-7  | 실질 독립항 판별     | 말미어(종결부)의 발명 카테고리가 부모와 다르면 실질 독립항. 어두 인용 여부만으로 독립/종속 판별 불가                                                                                             |
| P-8  | 카테고리 표시 규칙    | 실질 독립항에만 카테고리 레이블 표시. 실질 종속항은 레이블 없음 (OQ-6 확정, 이금희 1안)                                                                                                 |
| P-9  | AI 보정 비활성화    | 보정 실행 후 즉시 비활성화(중복 실행 방지). 개별/일괄 공통 적용                                                                                                                 |
| P-10 | 카테고리 구조       | 발명 카테고리는 실질 독립항 단위로 필드 부여. 종속항은 부모 독립항의 카테고리를 상속. 카테고리 : 실질 독립항 = 1 : N (여러 독립항이 동일 카테고리 가능, 독립항끼리는 서로 다른 카테고리 가능)                                     |
| P-11 | 트리 구조 시각화 분리  | 청구항 리스트 화면에는 계층 트리 시각화 기능 제공. 편집(/document) 화면에는 트리 구조 조회 버튼으로 분리(복잡한 UI 방지). 편집 화면에서는 발명 카테고리 / 독립항 / 종속항 구분만 제공하고, 종속 관계는 앵커로 표시 (Google Patents 참고) |
| P-12 | 명세서 생성 참조 이력  | 명세서 생성 시 청구항·기재불비·도면 중 어디를 참조했는지 이력 저장. 구체적인 저장 구조 및 연동 방식은 별도 논의 필요                                                                                   |
| P-13 | 기존 데이터 마이그레이션 | 기존 청구항 데이터는 독립항 1개·종속항 N개·발명 카테고리 1개 구조로 마이그레이션. 마이그레이션 시 LLM이 말미어를 분석하여 발명 카테고리 초기값을 자동 분류하여 설정                                                       |
| P-14 | 기재불비 저장 방식    | 기재불비는 청구항 단위 및 명세서 항목 단위로 개별 저장. 청구항과 명세서 항목 간 연동 구조를 지원하도록 모델링                                                                                        |

## 4. 기능 상세

https://www.figma.com/design/XD5TxmzMB68uXpKRmrgHpt/Patsol_SB?node-id=200-450

### 4-1. 청구항 목록 조회

**위치:** document 페이지 > 청구항 패널

**설명:** 문서에 포함된 청구항 전체를 독립항·종속항 계층 구조로 표시하는 기본 보기 상태.

### 구성 요소

| 요소 | 설명 |
| --- | --- |
| 청구항 번호 | `청구항 N` 형태로 표시 |
| 유형 레이블 | `독립항` / `종속항` |
| 발명 카테고리 레이블 | 실질 독립항 행에만 표시 (P-8 참조) |
| 내용 미리보기 | 청구항 텍스트 첫 줄 |
| AI 보정 버튼 | `[AI보정]` |
| 컨텍스트 메뉴 | `⋯` — 종속항 추가 / 편집 / 삭제 |
| 펼치기/접기 토글 | `▼` / `▶` — 독립항 좌측에 표시 |

### 동작 규칙

- 독립항은 트리의 루트 노드로 표시
- 종속항은 부모 청구항 하위에 들여쓰기로 표시
- 다중인용청구항의 경우 주요 부모 하위에 위치하며, 보조 인용 관계는 뱃지로 표시 (`[N│M]` 또는 `[N–M]`)
- 접힌 상태: `청구항 1 [장치] ▶ (하위 4항)` 형태로 종속항 수 표시
- 기본값: 전체 펼침

### 카테고리 체계

법리적 대분류(특허법 제2조제3호)와 실무적 세분류를 조합. UI에는 세분류 레이블만 표시하며 대분류는 메타데이터로 보존.

| 대분류 | 세분류 | 핵심 판별 기준 |
| --- | --- | --- |
| 물건발명 | `장치` | 구조·구성을 갖는 기계·기구 |
| 물건발명 | `물질` | 화학물질·단일 화합물 |
| 물건발명 | `조성물` | 복수 성분의 혼합·배합 |
| 물건발명 | `기록매체` | 프로그램이 기록된 컴퓨터 판독 가능 매체 |
| 물건발명 | `키트` | 복수 물건의 결합·세트 |
| 방법발명 | `방법` | 시계열적 단계·행위 |
| 방법발명 | `용도` | 물질·장치의 특정 목적 적용 |
| 제법발명 | `제조방법` | 물건을 생산·합성하는 방법 |

> **판별 기준 보충:** 물건 vs 방법 구별의 핵심 — "시계열적 순서(시간의 경과)를 필수요건으로 하는가". 시간 순서를 요건으로 하면 방법발명, 아니면 물건발명.

### 4-2. 청구항 계층 펼치기 / 접기

**위치:** document 페이지 > 청구항 패널 > 독립항 행

**설명:** 독립항 하위의 종속항 그룹을 접거나 펼쳐서 청구항 목록의 밀도를 조절한다.

### 동작 규칙

- 독립항 좌측에 토글 아이콘 `▼` / `▶` 표시
- 접힌 상태: 종속항 개수 표시. 예: `청구항 1 [물건] ▶ (하위 4항)`
- 기본값: 전체 펼침

### 4-3. 독립항 추가

**위치:** document 페이지 > 청구항 패널

**설명:** 새 독립항을 청구항 목록 하단에 추가한다.

### 구성 요소

| 요소 | 설명 |
| --- | --- |
| `[+ 독립항]` 버튼 | 패널 상단 또는 목록 하단 |

### 동작 흐름

```
[+ 독립항] 클릭
    ↓
목록 하단에 빈 청구항 행 삽입
  - 번호: 현재 최대 번호 + 1 자동 부여
  - 어두 없이 빈 텍스트 에디터 상태
    ↓
편집 포커스 자동 진입
    ↓
저장 → AI 보정 버튼 활성화 (P-9)
```

### 4-4. 종속항 추가

**위치:** document 페이지 > 청구항 패널 > 컨텍스트 메뉴

**설명:** 특정 청구항의 종속항을 추가한다. 어두를 자동으로 삽입한다.

### 구성 요소

| 요소 | 설명 |
| --- | --- |
| 컨텍스트 메뉴 `⋯` → 종속항 추가 | 트리거 |

### 동작 흐름

```
컨텍스트 메뉴 `⋯` → [종속항 추가] 클릭
    ↓
부모 청구항 번호 기반으로 어두 자동 삽입 (P-3)
  - 단일 인용: `제N항에 있어서,`
  - 다중 인용: `제N항 또는 제M항에 있어서,`
    ↓
번호: 부모 항의 마지막 형제 다음 번호 자동 부여
삽입 위치: 부모 청구항의 마지막 직계 자식 다음
    ↓
어두 이후 커서 포커스 자동 진입
    ↓
저장 → 기재불비 검증 실행 (P-6) → AI 보정 버튼 활성화 (P-9)
```

### 4-6. 청구항 개별 AI 보정

**위치:** document 페이지 > 청구항 패널 > 청구항 행

**설명:** 개별 청구항에 대해 LLM이 기재요건 검토 및 개선안을 제시한다. 변리사는 수용 또는 거절한다.

### 구성 요소

| 요소 | 설명 |
| --- | --- |
| `[AI보정]` 버튼 | 청구항 행에 위치. 활성화 조건은 하단 참조 |
| 원문/보정안 비교 뷰 | 보정 완료 시 원문과 보정안을 나란히 표시 |
| `[수용]` / `[거절]` | 수용: 보정안으로 교체 / 거절: 원문 유지, 보정 이력 초기화 |

### 동작 흐름

```
[AI보정] 클릭
    ↓
해당 행 로딩 스피너 표시, 버튼 disabled
    ↓
보정 완료 → 원문 / 보정안 비교 뷰 노출
  - 기재불비 감지 시 경고 + 보정안 제시 (P-5)
    ↓
[수용] 또는 [거절] 선택
    ↓
버튼 disabled (P-9)
```

| 조건 | 버튼 상태 |
| --- | --- |

| 보정 실행 후 | disabled |

| 유형 | 설명 |

| --- | --- | --- |

### 4-8. 다중 종속 표현

**위치:** document 페이지 > 청구항 패널 > 청구항 행

**설명:** 다중인용청구항의 복수 부모 인용 관계를 패널 UI에서 명확하게 표현한다.

### 표현 방식

| 인용 형태 | 뱃지 표기 |
| --- | --- |
| OR 표현 (`제N항 또는 제M항`) | `[N│M]` |
| 범위 표현 (`제N항 내지 제M항`) | `[N–M]` (OQ-3 확정) |

- 트리 상에서는 첫 번째 부모 하위에 위치하고, 두 번째 이후 부모 관계는 뱃지로만 표현
- 뱃지 hover 시 툴팁: 인용 문구 원문 표시

### 4-9. 기재불비 검증 정책

**설명:** PatSol이 적용하는 청구항 기재불비 검증 규칙의 전체 정의. 4-4(종속항 추가), 4-6(개별 보정), 4-7(일괄 보정) 모두 이 정책을 공유한다.

### 다중인용 재인용 판단 매트릭스 (한국 특허법 시행령 제5조)

| 인용 형태 | 조건 | 판단 |
| --- | --- | --- |
| 단일 인용 | 부모가 1개 (독립항·종속항·다중인용청구항 무관) | ✅ 항상 유효 |
| 다중 인용 | 인용 대상이 모두 단일인용항 또는 독립항 | ✅ 유효 |
| 다중 인용 | 인용 대상 중 1개 이상이 다중인용청구항 | ❌ 기재불비 |

### 검증 로직

```
function validateKoreanMultiDep(newClaim, allClaims):
  if newClaim.parents.length <= 1:
    return VALID  // 단일 인용은 항상 OK

  for parent in newClaim.parents:
    if allClaims[parent].parents.length > 1:
      return INVALID  // 부모 중 다중인용청구항 존재

  return VALID
```

### 실질 독립항 판별 로직

**핵심 원칙 (확인: 한상호 2026-04-03)**
> 어두에서 다른 청구항을 인용한다고 해서 무조건 종속항이 아니다. 말미어의 발명 카테고리가 부모와 다르면 실질 독립항으로 판별한다.

| 말미어 패턴 | 발명 카테고리 |
| --- | --- |
| `~장치`, `~시스템`, `~조성물`, `~화합물`, `~키트`, `~기록매체` | 물건발명 |
| `~하는 방법`, `~단계를 포함하는` | 방법발명 |
| `~의 용도`, `~에 사용하기 위한` | 방법발명(용도) |
| `~을 제조하는 방법` | 제법발명 |

```
function classifyClaim(claim, allClaims):
  if claim.parents is empty:
    return INDEPENDENT

  parentLegalObject = getLegalObject(allClaims[claim.parents[0]].ending)
  myLegalObject = getLegalObject(claim.ending)

  if myLegalObject != parentLegalObject:
    return INDEPENDENT  // 말미어 발명 카테고리가 다름 → 실질 독립항
  else:
    return DEPENDENT    // 부모와 동일 → 실질 종속항
```

### 문서 출처별 검증 적용 범위

| 적용 시점 | 대상 | 동작 |
| --- | --- | --- |
| 종속항 추가 (4-4) | PatSol 내 작성 | 실시간 경고, 저장 허용 (강제 차단 없음) |
| 개별 AI 보정 (4-6) | 작성·import 모두 | 기재불비 감지 후 보정안 제시 |
| 일괄 AI 보정 (4-7) | 작성·import 모두 | 사전 규칙 검출 → LLM 보정안 제시 |
| 외부 import 파싱 | PCT/EP 등 외국 문서 | 경고 표시만, 원문 변경 없음 |
