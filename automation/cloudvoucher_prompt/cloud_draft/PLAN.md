# 클라우드바우처 신청서 자동 초안 생성 시스템

## 목적

1. 수요기업이 문서 작업 부담 없이 클라우드바우처를 신청할 수 있도록 예시 초안 제공
2. 심사관이 공급기업(완드/PatSol) 관련 신청서를 획일적으로 판단하지 않도록 10종의 다양한 초안 출력

---

## 전체 구조

```
설문 입력 (Survey JSON)
        +
기반 콘텐츠 (base_content.md)
        +
템플릿 변형 10종 (templates/template_01~10.json)
        ↓
  chatgpt API 병렬 호출 (섹션별 비동기)
        ↓
  10가지 다양한 초안 출력
  outputs/{company_name}/draft_T01~T10.md
```

---

## Phase 1 — 설문 폼 설계

수요기업 입력 허들을 낮추기 위해 **필수 항목 A·B + 구조화 입력 C**로 구성. 선택 항목 D는 결과물 다양성을 높이는 변수로 활용.

---

### A. 기업 기본 정보 (필수)

| 필드                     | 유형    | 예시                                       |
| ---------------------- | ----- | ---------------------------------------- |
| `company_name`         | 텍스트   | 특허법인 공간                                  |
| `key_customers`        | 단일 선택 | 중소기업 중심 / 대기업 중심 / 비슷함                   |
| `strengths`            | 텍스트   | 주요 기술분야, 소규모 전문 인력, 중소기업 특화 이력, 신속한 의사결정 |
| `employees_patent`     | 숫자    | 8 (변리사·변리사 보조)                           |
| `employees_non_patent` | 숫자    | 4 (행정·영업 등 비변리 직원)                       |

---

### B. 도입 목적 및 문제점 (필수)

우선순위 1·2순위로 입력하여 프롬프트에서 **강조점 순서**를 결정함.

| 필드                 | 유형                  | 선택지                                      |
| ------------------ | ------------------- | ---------------------------------------- |
| `pain_points_rank` | 순위 선택 (1순위·2순위·3순위) | 수작업 많음 / 시스템 분산 / 보안 우려 / 인력 부족 / 비용 부담  |
| `goal_rank`        | 순위 선택 (1순위·2순위·3순위) | 업무 효율화 / 보안 강화 / 비용 절감 / 경쟁력 강화 / 성장 잠재력 |

---

### C. 클라우드 현황 구조화 입력 (필수)

기업별 현황 차이를 프롬프트에 직접 주입하여 획일적 서술을 방지함.

| 필드                   | 유형              | 선택지 / 예시                                          |
| -------------------- | --------------- | -------------------------------------------------- |
| `cloud_current_state`  | 단일 선택         | 현재 미사용 / 일부 사용 중 / 주요 업무에 사용 중                  |
| `cloud_purpose`      | 단일 선택         | 파일 스토리지 / 업무 협업 도구 / 전문 서비스 SaaS / 자체 서버 대체     |
| `cloud_readiness`    | 단일 선택         | 도입 검토 전 / 도입 검토 중 / 도입 결정 완료                     |
| `cloud_method`       | 단일 선택         | 퍼블릭 클라우드 / 프라이빗 클라우드 / 하이브리드                     |
| `cloud_timeline`     | 단일 선택         | 즉시 (~3개월) / 단기 (3~6개월) / 중기 (6개월~1년)              |
| `it_staff`           | 단일 선택         | 없음 / 1명 / 2~3명 / 4명 이상                            |
| `data_management`    | 단일 선택         | 개인 PC / 사내 서버 / 외부 클라우드 / 혼용                      |
| `it_asset_scale`     | 단일 선택         | 소형 (PC·복합기 중심) / 중형 (사내 서버 보유) / 대형 (전용 인프라 운영)  |

---

### D. 선택 항목

| 필드                  | 유형                    | 선택지 / 예시                         |
| ------------------- | --------------------- | -------------------------------- |
| `cloud_budget`      | 단일 선택                 | 500만 미만 / 500~1,000만 / 1,000만 이상 |
| `revenue_scale`     | 단일 선택                 | 5억 미만 / 5~50억 / 50억 이상           |
| `current_tools`     | 텍스트 (현재 사용 AI·SaaS 툴) | ChatGPT, 윈도우 공유폴더 등              |
| `specific_concerns` | 텍스트 (기타 우려사항)         | 보안 감사 대응 어려움 등                   |

| `pain_points_rank` 2순위 | 가. 현 업무 방식의 한계점 | 두 번째 ㅇ 항목으로 배치 |

| `cloud_readiness` | 가. 공공지원 필요 배경 | 자체 전환 한계 논거 강도 조정 |
| `it_staff` | 가. 공공지원 필요 배경 | IT 전담 인력 부재 논거 반영 여부 |
| `data_management` | 다. AS-IS 현황 | 현재 데이터 관리 방식 서술 |
| `it_asset_scale` | 다. AS-IS 현황 / 가. 공공지원 배경 | 인프라 자체 구축 부담 논거 |
| `cloud_method` | 나. 세부 이용 계획 | 도입 방식 명시 |
| `cloud_timeline` | 나. 세부 이용 계획 | 단계별 이용 계획 일정 |
| `key_customers` | 가. 기업 현황 | 고객 유형별 업무 복잡도 서술 방향 |
| `strengths` | 가. 기업 현황 / 가. 이용목적 | 기업 강점 기반 도입 기대 논거 |
| `employees_patent` / `employees_non_patent` | 가. 기업 현황 | 전문 인력 구성비 서술 |

## Phase 2 — 10가지 템플릿 변형 전략

**4개 축**의 조합으로 10종을 설계. 서술 방식·논거 프레임뿐 아니라 **항목 구성 자체**도 변주하여 구조적 다양성을 확보함.

### 변형 축

| 축 | 변형 요소 |
|---|---|
| **A. 강조점** | 비용절감 / 보안 / 경쟁력 / 업무효율 / 성장잠재력 |
| **B. 서술 방식** | 서술형 (단락) / 구조형 (항목 나열) |
| **C. 논거 프레임** | 문제→해결 / 목적→수단 / AS-IS/TO-BE 비교 |
| **D. 항목 구성** | 아래 별도 정의 |

---

### 축 D — 항목 구성 변주

> **제목 변주 원칙**: 항목 순서뿐 아니라 **소항목 제목 표현**도 동일한 의미 안에서 변주한다.
> 예시: "이용목적" → "이용 목적 / 도입 목적 / 서비스 활용 목적 / 도입 배경 및 목적"
> 구성(α~δ)이 같더라도 제목 표현이 달라지면 문서의 표면적 다양성이 높아진다.
> 프롬프트에서 LLM이 의미를 유지하면서 자연스러운 동의어·유사 표현으로 제목을 선택하도록 지시한다.

#### 소항목 제목 변주 풀

| 원래 제목 | 변주 예시 |
|---|---|
| 이용목적 | 이용 목적 / 도입 목적 / 서비스 활용 목적 / 도입 배경 및 목적 |
| 현 업무 방식의 한계점 | 현행 운영의 문제점 / 기존 업무 방식의 한계 / 현황 및 개선 필요성 |
| 클라우드 서비스 도입 필요성 | 서비스 도입의 필요성 / 클라우드 전환 필요성 / 디지털 전환 필요 배경 |
| 공공 지원 필요 배경 | 정부 지원의 필요성 / 공공 지원이 필요한 이유 / 자체 도입의 한계 및 지원 필요성 |
| 기업 현황 | 기업 비즈니스 현황 / 기업 운영 현황 / 기업 현황 및 특성 |
| 클라우드서비스 세부 이용 계획 | 서비스 세부 활용 계획 / 클라우드 서비스 활용 방안 / 세부 이용 및 운영 계획 |

#### 가. 이용목적 하위 항목 구성 (4종)

| 구성 | 항목 순서 | 특징 |
|---|---|---|
| 가-α | 이용목적 → 현 업무 한계점 → 도입 필요성 → 공공지원 배경 | 기본형. 소항목 제목은 변주 풀에서 선택 |
| 가-β | 기업 현황 및 이용목적 → 도입 필요성 → 공공지원 배경 (한계점 통합) | 간결형. 3항목 압축. 소항목 제목 변주 적용 |
| 가-γ | 이용목적 → 공공지원 배경 → 현 업무 한계점 → 도입 필요성 | 공공지원 논거 선행. 소항목 제목 변주 적용 |
| 가-δ | 기업 현황 → 이용목적 → 현 업무 한계점 및 도입 필요성 → 공공지원 배경 | 기업 소개 선행. 소항목 제목 변주 적용 |

#### 나. 2) 클라우드서비스 세부 이용 계획 하위 항목 구성 (4종)

| 구성 | 항목 순서 | 특징 |
|---|---|---|
| 나-α | DB통합검색 → 비교발명분석 → 청구항설계 연계 → 명세서초안 → 데이터관리 | 기본형 (기능 나열) |
| 나-β | [통합 워크플로우 개요] → 선행기술조사 → 청구항 설계 → 명세서 작성 → 데이터 보안 | 업무 흐름(workflow) 순서 중심 |
| 나-γ | AI 기반 검색·분석 기능 → 변리사 설계 지원 기능 → 데이터 관리·보안 기능 | 기능 카테고리 묶음형. AI 특화 부각 |
| 나-δ | 문제별 해결 기능 대응형 (각 AS-IS 한계점 → 대응 기능 1:1 매핑) | 한계점-기능 대응 구조 |

## Phase 3 — 프롬프팅 아키텍처

### 3-layer 프롬프트 구조

```
┌─────────────────────────────────────────┐
│  SYSTEM PROMPT (고정)                   │
│  - 역할: 지원사업 신청서 전문 작성가     │
│  - 특허 도메인 컨텍스트                  │
│  - PatSol 서비스 핵심 기능               │
│  - 금지 표현 (AI 명세서 자동작성 등)    │
└─────────────────────────────────────────┘
            +
┌─────────────────────────────────────────┐
│  BASE CONTENT (고정 논거 풀)             │
│  - base_content.md에서 섹션별           │
│    핵심 논거 문장 추출                   │
│  - "이 논거들을 참고하되 그대로 쓰지    │
│     말고 변형하여 활용"                  │
└─────────────────────────────────────────┘
            +
┌─────────────────────────────────────────┐
│  TEMPLATE INSTRUCTION (변형별 10종)     │
│  - 강조점, 서술 방식, 섹션 구조 지시    │
│  - 어조 및 어휘 수준 지정               │
│  - 소항목 제목 변주 지시:               │
│    "각 소항목의 제목은 의미를 유지하면서 │
│     자연스러운 동의어·유사 표현으로      │
│     바꿔 작성하라. 예: '이용목적' →     │
│     '도입 목적' / '서비스 활용 목적'"   │
└─────────────────────────────────────────┘
            +
┌─────────────────────────────────────────┐
│  SURVEY VARIABLES (기업별 주입)         │
│  - {company_name}, {main_work}          │
│  - {strengths}                          │
│  - {employees_patent}, {employees_non_patent} │
│  - {pain_points_rank}, {goal_rank}      │
│  - {cloud_current_state}, {cloud_readiness} │
│  - {it_staff}, {data_management}        │
│  - {it_asset_scale}, {cloud_method}     │
│  - {cloud_timeline}                     │
│  - {key_customers}, {cloud_budget} (선택)│
└─────────────────────────────────────────┘
```

### 섹션별 생성 전략

- 생성 대상 섹션: `현황`, `이용목적`, `이용계획`, `필요성`, `사업계획`, `기대효과`
- 6개 섹션을 **비동기 병렬 호출**로 생성 (기존 구조 유지)
- 각 섹션 프롬프트에 해당 섹션의 base content 논거만 선택적으로 주입
- 동일 회사 설문으로 10개 템플릿 배치 실행 → 10개 초안 생성

