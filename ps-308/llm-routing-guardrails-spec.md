# LLM 라우팅 · 가드레일 · 제안 설계 노트
> 특허 검색 입력 처리 — 구조적 빈틈 보완 방안
> 작성일: 2026-02-27

---

## 목적

현재 특허 검색 플로우(ia_spec-v0.1)는 사용자 입력을 **검색식 / 자연어(충분) / 자연어(불충분)** 세 갈래로만 분기한다. 이 설계에는 다음 구조적 빈틈이 존재한다.

- 출원번호·CPC 코드 등 자연어도 검색식도 아닌 입력이 잘못된 경로로 진입함
- 불완전한 검색식 시도를 탐지·보정하는 레이어가 없음
- 검색 결과 이후 LLM이 완전히 관여하지 않음

**목표:** 모든 입력에 대해 의도를 먼저 분류한 뒤, 각 경로에 적합한 라우팅·가드레일·제안을 적용하여 검색 성공률과 사용자 경험을 높인다.

---

## 현재 플로우 (As-Is)

```
입력
├── 상세 검색식 → Elastic Search → 결과
└── 자연어
    └── AI 충분도 판단 (충분/불충분 이진)
        ├── 충분  → Vector DB Top50 → 결과         [경로 A]
        └── 불충분 → 클러스터링 → Q&A 폼 → 결과    [경로 B]
```

### As-Is 빈틈 사례

| 입력 예시 | 현재 라우팅 | 실제 의도 |
|---|---|---|
| `KR10-2023-0147821` | 자연어 불충분 → Q&A 폼 | 특정 문서 직접 조회 |
| `삼성SDI 배터리` | 자연어 불충분 → Q&A 폼 | 출원인 + 키워드 혼합 검색 |
| `(배터리 AND 열관리` | Elastic Search 오류 | 검색식 작성 시도, 문법 오류 |
| `H01M 10/6563` | 경로 불명확 | CPC 코드 단독 분류 조회 |

---

## 개선 설계 (To-Be)

### How 1 — 입력 의도 분류기 (Intent Classifier)

충분도 판단 **앞**에 별도 LLM 레이어를 두어 입력 의도를 먼저 분류한다.

```
입력
└── Intent Classifier (LLM + 정규식 보조)
    ├── [출원번호/등록번호 패턴] → 단건 직접 조회 경로
    ├── [CPC / IPC 코드 단독]  → 코드 기반 분류 검색 경로
    ├── [검색식 작성 의도 감지] → Query Builder Assistant 경로
    └── [그 외 자연어]         → 기존 충분도 판단 경로
```

- 출원번호 패턴은 정규식(`KR10-\d{4}-\d{7}`, `US\d{7}` 등)과 LLM 병행
- 감지 시 Vector DB / Elastic Search를 거치지 않고 특허 원문 API 직접 조회
- **가드레일 역할:** 특정 입력 유형이 잘못된 파이프라인으로 진입하는 것을 차단

- [ ] 의도 분류값 구분 세부 정의

---

### How 2 — 충분도 판단 세분화 (3-Way Sufficiency)

이진 판단(충분/불충분)을 3단계로 세분하여 불필요한 Q&A 사이클을 줄인다.

| 판정 | 정의 | 처리 |
|---|---|---|
| **충분** | 의도 명확, 범위 적절 | Vector DB Top50 즉시 실행 (경로 A) |
| **보완 가능** | 의도 명확, 범위 조정 필요 | 필터 1~2개 제안 → 사용자 확인 후 즉시 검색 (경로 A') |
| **불충분** | 의도 불명확 | 클러스터링 → Q&A 폼 → 검색 (경로 B) |

- "보완 가능" 예시: "리튬이온 배터리 열관리" → 결과 수천 건 예상 → "출원 기간을 좁히거나 기술 세부 분야를 지정하면 더 정확합니다. 바로 검색할까요?"
- **가드레일 역할:** 충분한 입력을 불필요한 Q&A 루프로 보내지 않음

---

### How 3 — 검색식 작성 보조 (Query Builder Assistant)

검색식 작성 의도가 감지되었으나 문법 오류가 있는 경우 LLM이 보정 및 변환을 수행한다.

| 기능 | 입력 예시 | LLM 처리 |
|---|---|---|
| 문법 오류 감지·수정 | `(배터리 AND 열관리` | "닫힘 괄호 누락. `(배터리 AND 열관리)`로 수정할까요?" |
| 자연어 → 검색식 변환 | "삼성이 출원한 배터리 특허 최근 3년" | `출원인:"삼성" AND 배터리 AND 출원일:[2023 TO 2025]` 초안 생성 후 확인 |
| 검색식 자연어 해설 | 복잡한 Boolean 식 입력 | "이 검색식은 ~를 찾습니다" 형태로 해설 제공 |

---

### How 4 — 결과 후 대화형 리파인먼트 (Post-Search Conversational Refinement)

어느 경로로 도달했든 결과 섹션 이후에 지속 대화 인터페이스를 제공한다.

```
[검색 결과 N건 표시 중]

 ┌──────────────────────────────────────────┐
 │  결과에 대해 질문하거나 검색을 조정하세요 │
 │  예) "출원인을 삼성으로 좁혀줘"           │
 │      "미국 특허도 찾아줘"                 │
 │      "이 결과들의 공통 기술 요소가 뭐야"  │
 └──────────────────────────────────────────┘
```

| 사용자 발화 유형 | LLM 처리 |
|---|---|
| 필터 조정 ("출원인 LG만") | 현재 결과에 필터 적용, 재검색 불필요 |
| 범위 축소 ("핵심 기술만") | 유사도 상위 N건 재정렬 |
| 범위 확장 ("비슷한 특허도") | 동의어·상위 개념어 포함 쿼리 확장 후 재실행 |
| 분석 요청 ("공통 기술 요소") | 결과 문서 분석 후 요약 리포트 생성 |

- **명세서 작성으로의 연결:** 분석·합성 결과를 AI 명세서 작업으로 내보내기 가능

---

### How 5 — 검색 세션 컨텍스트 유지

여러 검색과 대화를 하나의 세션으로 묶어 히스토리에 저장한다.

- 세션 내 "이전 검색으로 돌아가기" 지원
- 세션 종료 시 "검색 요약 + 관심 특허 목록"을 명세서 작업으로 내보내기
- 검색 히스토리 메뉴에서 특정 세션 재개 가능

---

## 개선된 전체 플로우 (To-Be)

```
입력
└── Intent Classifier
    ├── 출원번호/코드 패턴   → 직접 조회 → 단건 상세
    ├── 검색식 작성 의도     → Query Builder Assistant → Elastic Search → 결과
    └── 자연어
        └── 3-Way Sufficiency
            ├── 충분     → Vector DB Top50 → 결과
            ├── 보완 가능 → 필터 제안 1회 → Vector DB Top50 → 결과
            └── 불충분   → 클러스터링 → Q&A 폼 → Vector DB Top50 → 결과

결과 섹션 (모든 경로 합류)
└── Post-Search Conversational Refinement (LLM)
    ├── 필터·정렬 조정
    ├── 쿼리 확장·축소
    └── 결과 분석·합성 → 명세서 작업으로 연결
```

## 예상 결과

| 개선 항목 | 기대 효과 |
|---|---|
| Intent Classifier | 출원번호 등 특수 입력의 잘못된 경로 진입 제거 |
| 3-Way Sufficiency | 불필요한 Q&A 루프 감소, 검색까지의 단계 수 축소 |
| Query Builder Assistant | 검색식 오류로 인한 0건 결과 및 사용자 이탈 감소 |
| Post-Search LLM | 재검색 없이 결과를 정제·분석, 명세서 작업 연결률 향상 |
| 세션 컨텍스트 | 검색 히스토리 재활용성 향상, 반복 작업 감소 |

## 미결정 항목 (Open Questions)

- Intent Classifier의 오분류 처리: 잘못 분류된 경우 사용자에게 경로 수정 UI 제공 여부
- Post-Search LLM의 분석 기능 범위: 결과 내 분석에만 한정할지, 외부 데이터(논문 등) 포함 여부
- 세션 컨텍스트 보존 기간 및 저장 방식
- 경로 B Q&A 반복 횟수 제한 (기존 OQ-3 미결정 항목과 연계)
