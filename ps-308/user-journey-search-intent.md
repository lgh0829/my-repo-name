> **1차 타깃 사용자 범위**: 명세서 작성 업무를 수행하는 **특허법인 소속 개인 변리사**로 한정한다. 이 문서의 모든 분석은 이 단일 페르소나를 기준으로 작성된다.

---

## 1. 누구인가

| 속성        | 내용                                                                           |
| --------- | ---------------------------------------------------------------------------- |
| 직군        | 특허법인 소속 변리사 (어소시에이트 ~ 파트너)                                                   |
| 핵심 업무     | 의뢰인의 발명을 기반으로 특허 명세서(청구항·발명의 설명) 작성                                          |
| 검색을 하는 이유 | 명세서 작성 전·중에 선행기술 조사를 수행하여 청구항 범위를 설계하기 위해                                    |
| 검색 빈도     | 건당 최소 1회 이상, 주요 사건은 수 차례 반복 조사                                               |
| 도구 숙련도    | 특허 검색 전문가 수준 — 검색식(Boolean·CPC) 작성 가능, 윈텔윕스 검색식 사용에 매우 익숙, 키프리스/키워트 DB 경험 있음 |

---

## 2. 명세서 작성 흐름 속 검색의 위치

변리사의 명세서 작성은 아래 흐름으로 진행된다. 검색은 단일 이벤트가 아니라 **반복적으로 끼어드는 업무**다.

```
발명 상담·OA 접수
       │
       ▼
[선행기술 조사 ①] ← 청구항 초안 설계 전 — 넓은 범위 탐색
       │
       ▼
청구항 초안 작성
       │
       ▼
[선행기술 조사 ②] ← 특정 구성요소 신규성·진보성 확인 — 좁은 범위 정밀 검색
       │
       ▼
명세서 본문 작성
       │
       ▼
[선행기술 조사 ③] ← OA 대응 또는 보정 시 추가 확인 (필요 시)
       │
       ▼
출원·제출
```

각 단계별로 검색 목적과 요구 정밀도가 다르기 때문에, 검색 도구는 **탐색형(①)과 정밀형(②③)을 모두 지원**해야 한다.

## 3. 입력 방식별 핵심 목적

### 3-A. 검색식 입력 (주요 모드)

변리사는 체계적 조사가 필요한 경우 직접 Boolean 검색식 또는 CPC 코드를 입력한다.

**핵심 목적: "내가 설계한 조건 그대로 처리하고, 빠짐없이 가져와라."**

- **정밀도(precision)** 와 **재현 가능성(reproducibility)** 이 최우선 — 동일 식으로 나중에 재실행해도 같은 결과셋이 나와야 한다
- LLM이 검색식을 임의로 변형하는 것을 **극도로 불신**한다
- 오류가 있을 경우, 조용히 수정하는 것보다 **명시적으로 보여주고 승인을 받는 것**을 선호한다

**결과에서 기대하는 것**

1. 검색식이 **그대로 실행되었다는 확인** — 수정이 있었다면 before/after를 나란히 표시
2. **히트 건수와 조건별 분포** — 어떤 항(AND/OR/NOT, CPC)이 결과를 어떻게 좁혔는지
3. **검색식 저장·재실행** 기능 — 조사 이력 관리와 보고서 첨부용
4. 결과 목록에서 **청구항 상태(등록·소멸·심사중)** 즉시 확인

**spec과의 연결**

spec 빈틈 사례 `(배터리 AND 열관리` → Elastic Search 오류가 정확히 이 맥락의 맹점이다. Query Builder Assistant(How 3)는:
- 오류를 **조용히 수정하지 않고** 원문과 수정안을 명시적으로 병렬 표시
- 수정안을 **제안으로** 제시하고 변리사가 최종 승인하도록 설계

---

### 3-B. 발명 자유 서술 입력 (탐색 모드)

의뢰인의 발명 설명을 받은 직후, 기술 영역의 전체 지형을 빠르게 파악할 때 사용한다.

**핵심 목적: "이 기술 영역에 어떤 선행기술이 깔려 있는지 빠르게 조감하고 싶다."**

- 초기 탐색이므로 **재현율(recall)** 우선 — 좁게 걸러서 놓치는 것이 더 위험하다
- *결과를 보고 청구항의 차별화 포인트를 **스스로 도출**하는 것이 목적*

**결과에서 기대하는 것**

1. 기술 개념 레벨의 유사 특허 목록 — 단순 키워드 매칭이 아닌 의미 기반 유사도
2. **어떤 구성요소가 겹치는지** 근거 제시 — 명세서 차별화 포인트 설계에 바로 활용
3. **선행기술 출원일 분포** — 어느 시점부터 기술이 집중 출원됐는지 타임라인
4. **화이트스페이스 힌트** — 아직 청구되지 않은 구성요소 조합 제안
5. 탐색 결과를 기반으로 **정밀 검색식으로 전환** 제안 (Post-Search Refinement)

**spec과의 연결**

3-Way Sufficiency(How 2)의 핵심 수혜자다. 발명 서술이 충분히 구체적이면 바로 Vector DB Top50으로 가고, 범위가 너무 넓으면 "보완 가능" 판정 후 CPC 코드 또는 키워드 필터 1~2개 제안으로 불필요한 Q&A 루프를 제거한다.

---

### 3-C. 문서(발명신고서·기존 특허) 업로드

의뢰인이 제출한 발명신고서, 또는 회피 설계 기준이 되는 경쟁사 특허를 그대로 올리는 경우다.

**핵심 목적: "이 문서와 기술적으로 얼마나 가까운 특허들이 존재하는가?"**

문서 자체가 이미 구체적인 기술 컨텍스트를 담고 있으므로, 변리사는 쿼리를 별도로 작성하지 않고 **문서 전체를 검색 입력으로** 쓰고 싶어 한다.

**결과에서 기대하는 것**

1. **유사도 기반 정렬된 특허 목록** — 청구항 단위로 어느 항이 가장 근접한지 대응 표시
2. **특허 패밀리·국가 확장** — 동일 발명의 PCT/해외 패밀리 포함 여부 (회피 설계 범위 확인)
3. **출원일 선후관계** — 업로드 문서 대비 선행기술인지 후행 특허인지
4. **청구항 생존 상태** — 등록·소멸·심사중 여부 (회피 필요성 판단)

**spec과의 연결**

현재 As-Is 플로우에는 **문서 업로드 경로 자체가 없다.** To-Be에서 Intent Classifier(How 1)가 *"문서 임베딩 직접 벡터 검색"* 경로를 별도로 열어줘야 하며, 텍스트 추출 후 자연어 쿼리로 flatten하면 문서 구조 정보가 소실된다.

## 4. 입력 방식 비교 요약

| 구분 | 검색식 | 자유 서술 | 문서 업로드 |
|---|---|---|---|
| 주요 발생 시점 | 조사 ②③ (정밀) | 조사 ① (탐색) | 조사 ① 또는 회피 설계 시 |
| 결과 우선 지표 | 정밀도 + 재현성 | 재현율 + 커버리지 | 유사도 + 청구항 대응 근거 |
| LLM 개입 허용도 | 낮음 — 승인 후 수정만 허용 | 높음 — 범위·필터 제안 환영 | 중간 — 임베딩은 위임, 결과 해석은 직접 |
| spec 연결 경로 | Query Builder Assistant (How 3) | 3-Way Sufficiency (How 2) | Intent Classifier 신규 경로 필요 |

---

## 5. 핵심 관찰

**변리사의 모든 검색은 명세서 작성이라는 단일 목적에 귀결된다.** 검색 결과를 보는 행위 자체가 목적이 아니라, 결과를 해석하여 청구항 범위를 확정하고 명세서 본문을 작성하는 것이 최종 아웃풋이다. 이 때문에 세 가지 입력 방식 모두 **"결과 → 다음 명세서 작업"으로 이어지는 연결 동선**이 필수적이다.

- 탐색 결과는 → 청구항 차별화 포인트 메모 또는 정밀 검색식 전환으로
- 정밀 검색 결과는 → 선행기술 리스트 저장 및 명세서 선행기술란 초안으로
- 문서 비교 결과는 → 회피 설계 방향 메모 또는 청구항 보정 근거로

spec의 How 4(Post-Search Refinement)와 How 5(세션 컨텍스트)가 이 연결 동선을 담당해야 하며, 특히 **검색 세션과 명세서 작업 세션 간 컨텍스트 전달**이 변리사 워크플로우의 핵심 마찰 지점이다.
