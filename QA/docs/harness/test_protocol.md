# 테스트 프로토콜

## 결과 기재 방법

| 기호 | 의미 | 조건 |
|---|---|---|
| ✅ PASS | P | 예상 결과와 완전히 일치 |
| ❌ FAIL | F | 예상 결과와 다름. Jira 버그 이슈 신규 등록 필수 |
| ⚠️ 부분 | W | 일부 동작하나 이슈 있음. 내용 상세 기재 |
| ⏭️ SKIP | S | 전제 조건 미충족으로 테스트 불가. 이유 기재 |

## Chrome 브라우저 조작 (mcp__Claude_in_Chrome)

Chrome MCP 도구를 사용해 실제 UI를 조작한다.

### 기본 흐름

1. `list_connected_browsers` — 연결된 브라우저 확인
2. `navigate` — 테스트 대상 URL로 이동
3. `get_page_text` / `read_page` — 현재 화면 내용 확인
4. `find` — 특정 요소 위치 파악
5. `form_input` / `javascript_tool` — 입력·조작
6. `computer` — 버튼 클릭 등 시각 조작
7. `gif_creator` / `upload_image` — 증거 스크린샷 캡처

### 주의사항

- 실제 데이터 변형이 일어나는 작업(저장·삭제)은 사전에 확인 후 진행한다.
- 테스트 중 발생하는 콘솔 에러는 `read_console_messages`로 캡처해 결과 문서에 포함한다.
- 네트워크 오류(4xx/5xx)는 `read_network_requests`로 확인한다.

## 테스트 시나리오 작성 방법

**시나리오는 QA 시점에 실제 화면을 보고 만든다.** 사전 정의된 TC가 없는 기능은 아래 방법으로 도출한다.

### 방법 1 — 화면 캡처 / Figma 기반

Figma URL 또는 스크린샷을 받으면:
1. 화면의 모든 인터랙션 요소(버튼·입력·드롭다운·링크) 파악
2. 정상 경로(happy path) → 예외 경로(empty·error·boundary) 순으로 TC 도출
3. `docs/harness/<이슈키>_<기능명>.md`에 TC 목록 작성 후 테스트 진행

### 방법 2 — 사용자 질문으로 보충

시나리오 정보가 불충분하면 다음 항목을 질문한다:

1. **진입 경로**: 어느 메뉴에서 시작하는가?
2. **전제 조건**: 데이터가 있어야 하는가? 없어야 하는가?
3. **핵심 동작**: 무엇을 검증하고 싶은가?
4. **예외 케이스**: 빈 상태, 권한 없음, 네트워크 오류 등을 포함하는가?
5. **통과 기준**: 어떤 상태여야 PASS인가?

## 진입 경로 목록

| 경로 ID | 진입 경로 |
|---|---|

## 버그 이슈 등록 절차

FAIL 또는 재현 가능한 W 항목 발견 시:

1. Jira에 신규 작업 이슈 생성
2. Summary: `[버그] <기능명> - <현상 요약>`
3. Description에 포함:
   - 재현 경로
   - 예상 결과
   - 실제 결과
   - 스크린샷/동영상 첨부
   - 관련 TC 번호
4. 원래 QA 이슈(예: PS-869)에 링크 연결
