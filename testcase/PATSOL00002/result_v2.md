# 도면 품질 평가 결과

**대상 명세서**: PATSOL00002 / spec.md
**평가 기준**: 5개 항목 (충족 / 부분충족 / 미충족)

---

## 도 3 평가

**도면 설명 (spec.md line 72)**
> 운영자 단말로부터 채팅방 생성 요청(구독료 정보 포함)을 수신하여 채팅방을 생성하고 이용 권한을 부여하는 전체 절차를 나타낸 흐름도

### 평가 결과

| # | 항목 | 결과 |
|---|---|---|
| 1 | 명세서의 핵심 구성요소가 도면에 반영되었는가 | 부분충족 |
| 2 | 도면부호와 구성요소가 올바르게 대응되는가 | 미충족 |
| 3 | 구성요소 간 연결·배치관계가 명세서 설명과 일치하는가 | 부분충족 |
| 4 | 명세서에 없는 요소가 임의로 추가되지 않았는가 | 미충족 |
| 5 | 여러 도면에서 동일 발명이 일관된 구조로 표현되는가 | 미충족 |

### 세부 근거

**1. 핵심 구성요소 반영 — 부분충족**
- 운영자 단말(20), 사용자 단말(30), 채팅방 관리 모듈(13), 구독/결제 관리 모듈(14), 권한 부여 모듈(15), 정산 모듈(16) 등 주요 구성요소 반영됨
- 정기 결제 루프 전체(도 5 해당 내용)와 정산 모듈(16)이 도 3에 혼입되어 범위 초과

**2. 도면부호 대응 — 미충족**

| 도면 표기 | 사용 부호 | 명세서 정의 | 오류 |
|---|---|---|---|
| `PAYMENT SUCCESSFUL?` 다이아몬드 | 12 | 12 = MEMORY | ❌ |
| `UPANTS TECSURBITATION DECISSOR` | 12 | 12 = MEMORY | ❌ |
| 운영자 단말 블록 우측 | 11 | 11 = PROCESSOR | ❌ 맥락 부적합 |

**3. 연결·배치관계 — 부분충족**
- 구독 여부 확인(14) → 미구독 시 구독 요청 전송 흐름: 명세서와 일치
- 구독 중(YES) → PROCESSOR(11) 직접 연결: 명세서 미기술 경로, 권한 부여 모듈(15)로 가야 논리적
- 양 컬럼 연결 화살표(상단) 진입점 불명확하여 흐름 추적 어려움
- 정기결제 실패 → 정산 모듈(16) 연결: spec section 5-4에서 결제 실패 시 이용 권한 회수만 기술

**4. 명세서에 없는 요소 — 미충족**
- `UPANTS TECSURBITATION DECISSOR (12)`: 명세서에 존재하지 않는 의미불명 텍스트 — AI 생성 오류(hallucination) 추정, 치명적 결함
- `EXTENDS USAGE PERIOD` 박스 중복 표기
- 정기결제 루프 전체가 도 3 기술 범위 초과 (도 5에 해당)

**5. 도면 간 일관성 — 미충족**

| 항목 | 도 2 | 도 3 | 불일치 여부 |
|---|---|---|---|
| 부호 12 | MEMORY | PAYMENT SUCCESSFUL? / UPANTS... | ❌ |
| 부호 11 | PROCESSOR | 운영자 단말 블록 옆 혼용 | ❌ |

### 종합 의견
도 3은 전반적인 흐름 골격은 명세서와 유사하나, 부호 12의 중복·오용, `UPANTS TECSURBITATION DECISSOR`라는 의미불명 요소 삽입, 도 5 내용의 무단 혼입 등 치명적 오류가 복수 존재함. 실제 특허 도면으로 사용하려면 전면 재작성 필요.

## 도 4 평가

**도면 설명 (spec.md line 74)**
> 사용자 계정의 채팅방 이용 요청에 대해 구독 여부를 판단하고, 미구독 시 구독 요청 전송 및 결제 성공 시 이용권을 제공하는 절차를 나타낸 흐름도

### 평가 결과

| # | 항목 | 결과 |
|---|---|---|
| 1 | 명세서의 핵심 구성요소가 도면에 반영되었는가 | 부분충족 |
| 2 | 도면부호와 구성요소가 올바르게 대응되는가 | 부분충족 |
| 3 | 구성요소 간 연결·배치관계가 명세서 설명과 일치하는가 | 부분충족 |
| 4 | 명세서에 없는 요소가 임의로 추가되지 않았는가 | 부분충족 |
| 5 | 여러 도면에서 동일 발명이 일관된 구조로 표현되는가 | 부분충족 |

### 세부 근거

**1. 핵심 구성요소 반영 — 부분충족**
- 사용자 단말(30), 서버(10), 프로세서(11), 메모리(12), 채팅방 관리 모듈(13), 구독/결제 관리 모듈(14), 권한 부여 모듈(15) 모두 등장
- **결제 실패(NO) 경로 누락**: `WAS PAYMENT SUCCESSFUL? (240)`의 NO 분기에 후속 처리 없음. spec section 5-4에서 결제 실패 시 이용 권한 회수·만료 처리 기술

**2. 도면부호 대응 — 부분충족**
- 구성요소 부호(10, 11, 12, 13, 14, 15, 30)는 도 2 정의와 일치
- 단계 번호(200, 210, 220, 230, 240, 250, 260, 270)가 구성요소 부호와 동일한 `-숫자` 형식으로 혼용되어 부호 체계 혼란 유발
- 명세서에서 해당 단계 번호를 어디에도 정의하지 않음

**3. 연결·배치관계 — 부분충족**

| 흐름 | 명세서 일치 여부 |
|---|---|
| 이용 요청 → 구독 여부 확인(14) | ✅ |
| 구독 중 → 권한 부여 모듈(15) 즉시 승인 | ✅ |
| 미구독 → 구독 요청 전송(220) → 결제(230) | ✅ |
| 결제 성공 → 이용 기간 설정·정기결제일 결정(250) | ✅ |
| 결제 실패(NO) → 후속 처리 없음 | ❌ |
| 구독 승인(210) 이후 → END 연결 경로 불명확 | △ |

두 번째 블록의 "RUNS PROGRAM IN MEMORY 12" 표현: 메모리(12)가 처리 주체인 것처럼 오해 유발, 명세서에 없는 기술 표현

**4. 명세서에 없는 요소 — 부분충족**
- 단계 번호(200~270): 명세서에 정의되지 않은 임의 번호
- "PROVIDES ACCESS TICKET": spec의 "이용권 제공"에 대응하나 "ticket" 용어는 명세서 미사용
- "CHATTING ROOM": 도 2·3의 "CHAT ROOM"과 용어 불일치

**5. 도면 간 일관성 — 부분충족**

| 항목 | 도 2 | 도 3 | 도 4 | 비고 |
|---|---|---|---|---|
| 구성요소 부호 | 정의 기준 | 오류 다수 | 올바르게 사용 | 도 4는 개선됨 |
| CHAT ROOM 표기 | CHAT ROOM | CHAT ROOM | CHATTING ROOM | ❌ 불일치 |
| 단계 번호 체계 | 없음 | 없음 | 200~270 | ❌ 불일치 |

### 종합 의견
도 4는 도 3 대비 크게 개선되었으며 핵심 흐름과 부호 사용이 전반적으로 양호함. 그러나 결제 실패 경로 누락이 가장 큰 결함이고, 단계 번호 미정의·"CHATTING ROOM" 용어 불일치 등 보완 필요 항목이 남아 있음. 수정 범위는 도 3보다 훨씬 작으며 부분적 수정으로 완성 가능한 수준.
